Credit Memo Pipeline — End-to-End System Documentation

Complete system-level documentation: Stripe (Credit Notes) → Fivetran → Snowflake (dbt) → Workato → NetSuite (Credit Memos)

Type: End-to-End Author: venkatesh.manohar Generated: April 9, 2026 Systems: 5 (Stripe, Fivetran, Snowflake, Workato, NetSuite) Workato Recipes: 5 active, 2 stopped Schedule: 4x daily (5:30, 10:30, 15:30, 20:30 PT)

1. Executive Summary

Business View

When a customer receives a credit (due to a Stripe credit note from a refund, proration, or billing adjustment), a corresponding Credit Memo must be created in NetSuite so Finance can properly account for the revenue adjustment, reduce accounts receivable, and maintain an accurate audit trail. This pipeline automates the entire process — from the moment a Stripe Credit Note is issued to a fully-formed Credit Memo (with line items, rev-rec dates, and correct subsidiary assignment) appearing in NetSuite.

It also handles negative invoices (downgrade prorations generating negative amounts), refunds (Stripe refunds via credit notes), and customer credit adjustments (balance adjustments applied to customer accounts) — all without manual Finance intervention.

Technical View

A 5-system pipeline: Stripe (source of truth for credit notes, refunds, and charges) → Fivetran (Stripe Data Share to Snowflake) → Snowflake (dbt pipeline + Workato export tables in SHARED_EXTERNAL_WORKATO) → Workato (orchestration, business logic, NetSuite API calls) → NetSuite (destination ERP, CREDMEM records).

Workato runs on a cron schedule (4x/day), queries Snowflake for unsynced credit notes and refunds, and creates Credit Memos and Customer Refunds in NetSuite via SuiteTalk API. A log table (WORKATO_LOGS) tracks sync state to ensure idempotency.

Reliability: The core credit memo recipe (RF 5.0) has processed 2,866 jobs with 0 failures (23,887 lifetime API tasks). The refund recipe (RF 7.0) has processed 2,065 jobs with 0 failures. Combined, these recipes maintain a 100% success rate.

2. Business Overview

What Problem Does This Solve?

Intercom's revenue operations require that every Stripe credit note and refund results in a corresponding NetSuite Credit Memo or Customer Refund. Without automation, Finance teams would need to manually reconcile thousands of credit transactions per month, match them to original invoices, calculate partial credits and prorations, and handle multi-currency exchange rate adjustments.

Business Flow

  1. Credit event occurs in Stripe — A credit note is issued against a Stripe invoice (due to refund, proration, or billing error), or a refund is processed against a charge
  2. Credit note line items are defined — Each Stripe credit note has line items with amounts, quantities, and references to the original invoice line items
  3. Data flows to Snowflake — Stripe Data Share (via Fivetran) lands raw Stripe data in Snowflake; dbt transforms it and populates the export tables (NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES, NETSUITE_EXPORT_REFUNDS)
  4. Workato picks up unsynced credits — 4x daily, Workato queries Snowflake for credit notes and refunds not yet in NetSuite
  5. Credit Memo created in NetSuite — Workato creates the CREDMEM via SuiteTalk API with all line items, rev-rec dates, and cross-references
  6. Refunds processed separately — Stripe refunds that involve actual money movement go through the Refunds pipeline (Rec 7.0 / RF 7.0) to create Customer Refund records

4. Architecture Overview

Source Layer

Stripe Billing holds the canonical credit note and refund data. When a customer receives a credit (partial refund, billing adjustment, or downgrade proration), Stripe issues a Credit Note linked to the original invoice. When money is returned, a Refund is created against the original charge.

Source Layer — Technical

Stripe Data Share provides raw data to STAGING_STRIPE_DATA_SHARE in Snowflake (invoices, customers, subscriptions). Stripe credit note and refund events are processed through dbt models into the SHARED_EXTERNAL_WORKATO export tables. The BASE.STRIPE_REFUND table (236K rows) also provides raw refund data.

Data Layer

Snowflake serves as the central data platform. Multiple export tables feed different credit scenarios: credit notes from Stripe, negative invoices from downgrades, refunds from Stripe charges, and customer credit adjustments. Each table has a specific grain and purpose.

Data Layer — Technical

Key export tables in SHARED_EXTERNAL_WORKATO:
NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES (3,849 rows, 34 cols) — primary credit memo source
NETSUITE_EXPORT_CREDIT_MEMO_NEGATIVE_INVOICE (2,727 rows, 16 cols) — downgrade prorations
NETSUITE_EXPORT_REFUNDS (1,926 rows, 18 cols) — Stripe refunds
NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS (20,924 rows, 14 cols) — balance adjustments
WORKATO_LOGS (67M rows) — idempotency tracking

Orchestration Layer

Workato is the automation engine. It runs 4x daily, finds credit notes and refunds that need syncing, applies business rules (amount filtering, customer lookup, subsidiary assignment), and creates the NetSuite records.

Orchestration Layer — Technical

Two parallel pipelines:
Credit Memos: 2.0 Master → Rec 5.0 (Snowflake SQL, group by credit note) → RF 5.0 (NetSuite API per credit note)
Refunds: 2.0 Master → Rec 7.0 (Snowflake SQL, group by refund) → RF 7.0 (NetSuite API per refund)
Both use WORKATO_LOGS CTE for idempotency checking.

Target Layer

NetSuite receives Credit Memos and Customer Refunds. Finance uses these for revenue adjustments, accounts receivable reduction, and reconciliation with Stripe. Custom fields link back to Stripe IDs for traceability.

Target Layer — Technical

NetSuite SuiteTalk API. Credit Memos use the CREDMEM record type. External ID = Stripe Credit Note ID for credit memos, Stripe Refund ID for refunds. Customer lookup uses Stripe cus_xxx ID. Subsidiary, currency, and exchange rate are passed through from Stripe.

6. End-to-End Flow

The complete data journey for a Credit Memo follows this sequential path across five systems:

  1. Stripe Billing — A credit note is issued against a Stripe invoice (e.g., partial refund, billing adjustment, downgrade proration), or a refund is processed against a charge. Each credit note has line items with amounts, quantities, and references to original invoice line items.
  2. Fivetran / Stripe Data Share — Stripe data lands in Snowflake via the Stripe Data Share connector to STAGING_STRIPE_DATA_SHARE. This includes invoices, customers, and subscription data. Credit note and refund data is processed through additional dbt models.
  3. Snowflake dbt Pipeline — A sequential dbt pipeline transforms the raw data:
    • STAGING_STRIPE_DATA_SHARE — raw Stripe tables (invoices, customers, subscriptions)
    • BASE.STRIPE_REFUND — refund dimension (236K rows)
    • INT_STRIPE_DEBIT_CREDIT_STATEMENT_BY_LINE_ITEM — intermediate debit/credit analysis (52,940 rows)
    • NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES — primary export table (3,849 rows, 34 cols) for Workato
    • NETSUITE_EXPORT_CREDIT_MEMO_NEGATIVE_INVOICE — downgrade prorations (2,727 rows, 16 cols)
    • NETSUITE_EXPORT_REFUNDS — Stripe refunds (1,926 rows, 18 cols)
  4. Workato Orchestration — Runs 4x daily via the 2.0 Master Recipe. Rec 5.0 queries the credit note export table, groups by credit note, and for each calls RF 5.0 which: looks up the customer in NetSuite, builds credit memo line items, and calls the NetSuite API. Separately, Rec 7.0 queries refunds and calls RF 7.0 for each.
  5. NetSuite ERP — Receives Credit Memos (CREDMEM), Customer Refunds, and Credit Balance Adjustments via SuiteTalk API. The latest credit memo is CM20621 (April 5, 2026). Total of 20,969 credit memos exist in NetSuite.
Feedback loops: Workato writes sync results (ns_internal_id, status) back to WORKATO_LOGS in Snowflake for dedup tracking. The WORKATO_LOGS CTE in each master recipe ensures that already-synced records are not re-processed.

Key SQL: Rec 5.0 Master Query (Snowflake) 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE
        external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES'
        AND environment_type = 'prod'
        AND is_successful_write_to_netsuite = true
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id ORDER BY job_run_completed_at DESC
    ) = 1
)
SELECT CUSTOMER, SUBSIDIARY, DATE, STRIPE_CREDIT_NOTE_ID,
  STRIPE_INVOICE_ID, STRIPE_INVOICE_LINE_ITEM, CREDIT_NOTES_NUMBER,
  STRIPE_CREDIT_NOTE_LINE_ITEMS_ID, SALESFORCE_CONTRACT_ID,
  STRIPE_INVOICE_STATUS, CURRENCY, EXCHANGE_RATE,
  ITEM, ITEM_DESCRIPTION, QTY, ACTUAL_QTY, AMOUNT,
  DISCOUNT_ITEM, DISCOUNT_AMOUNT, STRIPE_CREDIT_NOTE_TOTAL_CENTS,
  STRIPE_CREDIT_NOTE_TOTAL, REV_REC_START_DATE, REV_REC_END_DATE,
  DISCOUNT_PERCENTAGE, CHARGE_ID, COLLECTION_METHOD
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES ce
LEFT JOIN stg_workato_log wl ON wl.external_id = ce.stripe_credit_note_id
WHERE date >= '{Start_Date}'
  AND date <= DATEADD(day, -1, CURRENT_DATE)
  AND amount != 0
  AND wl.netsuite_internal_id IS NULL
ORDER BY created_date ASC
Key SQL: Rec 7.0 Refunds Master Query (Snowflake) 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE
        external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS'

SELECT r.customer, stripe_refund_id, r.subsidiary,
  credit_note_id, c.category, r.stripe_invoice_id,
  stripe_charge_id, stripe_automatic_payout_id,
  refund_amount, r.date, r.created_date, r.netsuite_internal_id
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS r
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER c
  ON r.customer = c.external_id
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT p
  ON r.stripe_charge_id = p.STRIPE_PAYMENT_ID

  AND r.date <= DATEADD(day, -1, CURRENT_DATE)
  AND wl.netsuite_internal_id IS NULL
ORDER BY r.created_date

9. Key Business Logic

Credit Note vs. Refund Routing

Business View

Stripe credit notes and refunds serve different purposes. A credit note reduces the amount owed on an invoice (e.g., partial credit, billing adjustment) and maps to a NetSuite Credit Memo. A refund represents actual money returned to the customer and maps to a NetSuite Customer Refund. The 2.0 Master Recipe orchestrates both pipelines sequentially: credit memos first (Rec 5.0), then refunds (Rec 7.0).

Technical View

The 2.0 Master Recipe calls four child recipes in sequence: (1) Rec 5.0 Credit Memo Master, (2) Rec 8.0 Create CBA, (3) Rec 6.0 Create Customer Payment, (4) async call to 3.0 Master for Refunds/Uncollectible/Cancellations. Each uses a different source_table_name in the WORKATO_LOGS CTE for idempotency. RF 5.0 uses netsuite and workato_custom_code connectors; RF 7.0 uses only snowflake (reads) and the Rec 7.0 foreach pattern.

Key Business Rules

Credit Memo Line Item Construction

RF 5.0 builds NetSuite credit memo line items from the Snowflake export data. For each credit note, it:

  1. Looks up the customer in NetSuite using the Stripe cus_xxx external ID
  2. Maps the ITEM field to a NetSuite item (via billing plan ID mapping)
  3. Sets quantities (QTY, ACTUAL_QTY), amounts, and discount information
  4. Applies revenue recognition dates (REV_REC_START_DATE, REV_REC_END_DATE)
  5. Sets the exchange rate for multi-currency transactions
  6. References the Stripe credit note ID, invoice ID, and Salesforce contract ID via custom fields
  7. Writes the result (ns_internal_id) to WORKATO_LOGS for dedup

Refund Processing Logic

Rec 7.0 finds refunds where a Stripe refund exists but has not been logged to NetSuite. RF 7.0 then:

  1. Passes the customer, subsidiary, refund amount, and all Stripe IDs (refund_id, charge_id, invoice_id, credit_note_id, payout_id)
  2. Looks up the original payment in NetSuite via stripe_charge_id
  3. Creates a Customer Refund record linked to the original payment
  4. Sets the stripe_automatic_payout_id for bank reconciliation
  5. Logs the result to WORKATO_LOGS with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS'

11. Edge Cases & Special Scenarios

ScenarioHandlingImpact
$0 credit note lines Excluded by amount != 0 filter in Rec 5.0 SQL Zero-amount lines are metadata-only; no financial impact in NetSuite
Multi-currency credit notes Exchange rate passed from Stripe (EXCHANGE_RATE column) to NetSuite; subsidiary determines base currency Credit memos posted in correct subsidiary currency with proper FX rate
Same-day credit notes Filtered out by date <= DATEADD(day, -1, CURRENT_DATE) — only processes credits from prior days Prevents picking up credits that may still be updating in Stripe (1-day buffer)
CBAs not in NetSuite when originating invoice was manual When the original invoice was a manual proration/downgrade, the routing debug string in the NS ID field prevents RF 8.0 from parsing correctly Known gap: manual proration invoices generate CBAs that fail to sync; requires manual investigation
Refund without a linked payment Rec 7.0 LEFT JOINs to NETSUITE_EXPORT_PAYMENT; if no match, refund may fail in NetSuite Requires manual payment lookup or creation before the refund can be applied
Negative invoice (downgrade proration) Separate table NETSUITE_EXPORT_CREDIT_MEMO_NEGATIVE_INVOICE (2,727 rows) handles these via Rec 5.0 Downgrade-generated negative amounts routed to credit memo rather than invoice pipeline
Voided Stripe invoices creating NS CREDMEMs in different months NetSuite voiding CREDMEMs get a new trandate (potentially different month than original Stripe event) Creates false "missing_in_stripe" in monthly recon; trend report excludes voided invoices from missing counts
Partial credit notes Only the specific line items credited appear in the export table; full credit notes have all lines NetSuite CREDMEM accurately reflects only the credited portions
Duplicate credit note IDs in logs WORKATO_LOGS CTE uses QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1 Ensures only the most recent log entry is checked for idempotency
3-day buffer for Workato/Fivetran sync lag Monthly recon windows extended by 3 days to account for sync delays Avoids false mismatches when data takes time to flow through the pipeline

Steps execute sequentially: Rec 5.0 (Credit Memos) → Rec 8.0 (CBAs) → Rec 6.0 (Payments) → then asynchronously calls 3.0 Master for Refunds, Uncollectible, and Cancellations.
