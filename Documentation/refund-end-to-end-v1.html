Refund Pipeline — End-to-End System Documentation

Complete system-level documentation: Stripe → Fivetran → Snowflake (dbt) → Workato → NetSuite

Type: End-to-End Author: venkatesh.manohar Generated: April 9, 2026 Systems: 5 (Stripe, Fivetran, Snowflake, Workato, NetSuite) Workato Recipes: 7 active, 1 stopped Schedule: 4x daily (5:30, 10:30, 15:30, 20:30 PT)

1. Executive Summary

Business View

When a customer receives a refund in Stripe (due to a credit note, billing error, or cancellation), a Customer Refund must be created in NetSuite (our ERP) so that Finance can accurately reconcile cash, update accounts receivable, and close out the billing cycle. This pipeline automates the entire process — from the moment a refund is issued in Stripe through to a fully-formed Customer Refund record in NetSuite.

The pipeline also handles two adjacent flows: voided invoices (Stripe invoices voided before collection, creating credit memos in NetSuite) and uncollectible invoices (invoices marked as bad debt, creating journal entries in NetSuite).

Technical View

A 5-system pipeline: Stripe (source of truth for refunds and credit notes) → Fivetran (CDC replication to Snowflake) → Snowflake (dbt staging + Workato export tables in SHARED_EXTERNAL_WORKATO) → Workato (orchestration, business logic, API calls) → NetSuite (destination ERP).

Workato runs on a cron schedule (4x/day), queries Snowflake for unsynced refunds (using WORKATO_LOGS for dedup), and creates Customer Refund records in NetSuite via SuiteTalk API. Three parallel sub-pipelines handle refunds, voided invoices, and uncollectible invoices through a single master orchestrator.

Reliability: The core refund child recipe (RF 7.0) has processed 2,065 jobs with 0 failures. The refund master recipe (Rec 7.0) has 858 jobs, 0 failures. The voided invoice pipeline (RF 10.1) has 903 jobs, 0 failures. Combined lifetime API tasks exceed 540,000.

2. Business Overview

What Problem Does This Solve?

Intercom's billing operations require that every Stripe refund results in a corresponding NetSuite Customer Refund record. Without automation, Finance teams would need to manually create thousands of refund records, match them against original payments, handle multi-subsidiary routing, and manage voided and uncollectible invoice write-offs — all while maintaining accurate cash reconciliation.

Business Flow

  1. Customer requests a refund — A Stripe refund is issued against a charge, often linked to a credit note on the original invoice
  2. Stripe processes the refund — The refund object is created with the refund amount, linked charge, credit note, and payout information
  3. Data flows to Snowflake — Fivetran replicates Stripe refund data to BASE.STRIPE_REFUND; a dbt model transforms it into the export table NETSUITE_EXPORT_REFUNDS
  4. Workato picks up unsynced refunds — 4x daily, Workato queries Snowflake for refunds not yet in NetSuite (checked via WORKATO_LOGS)
  5. Customer Refund created in NetSuite — Workato creates the refund via SuiteTalk API, linking it to the correct customer, subsidiary, and original payment
  6. Voided invoices handled — Stripe voided invoices create Credit Memos in NetSuite with full line-item detail and rev-rec dates
  7. Uncollectible invoices handled — Stripe uncollectible invoices create Journal Entries in NetSuite for bad debt write-off

4. Architecture Overview

Source Layer

Stripe Billing holds the canonical refund and payment data. When a refund is processed, Stripe creates a Refund object linked to the original Charge, with optional Credit Note and Invoice references. Stripe also manages payout reconciliation via the automatic payout ID.

Source Layer — Technical

Fivetran CDC connector continuously replicates Stripe Refund, Charge, Invoice, and Credit Note objects to INTERCOM_PROD.BASE. The raw STRIPE_REFUND table has 236,683 rows with columns: STRIPE_REFUND_ID, STRIPE_BALANCE_TRANSACTION_ID, STRIPE_CHARGE_ID, STRIPE_REFUND_AMOUNT, STRIPE_REFUND_STATUS, STRIPE_REFUND_REASON, CREATED_AT.

Data Layer

Snowflake serves as the central data platform. The dbt pipeline transforms Stripe refund data and enriches it with customer, subsidiary, and payment information to produce export tables that Workato reads. Three export tables serve three different flows: refunds, voided invoices, and uncollectible invoices.

Data Layer — Technical

Pipeline: BASE.STRIPE_REFUND (raw, 236K rows) → SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS (1,926 rows, 18 cols). Adjacent tables: NETSUITE_EXPORT_VOIDED_INVOICE (6,749 rows, 24 cols), EXT_NETSUITE_EXPORT_UNCOLLECTIBLE (11,268 rows, 12 cols). The export table filters to only refunds requiring NetSuite sync. Workato reads the export table, joined with NETSUITE_EXPORT_CUSTOMER (for category) and NETSUITE_EXPORT_PAYMENT (for charge mapping), cross-referenced against WORKATO_LOGS for dedup.

Orchestration Layer

Workato is the automation engine. It runs 4x daily via a top-level orchestrator (Recipe 3.0) that calls three sub-pipelines in sequence: Refunds (Rec 7.0 → RF 7.0), Voided Invoices (Rec 10.1 → RF 10.1), and Uncollectible Invoices (Rec 10.0 → RF 10.0). Each sub-pipeline queries its own Snowflake export table and creates the appropriate NetSuite record.

Orchestration Layer — Technical

Orchestrator (3.0 Master, callable function trigger) → Rec 7.0 (Snowflake SQL, dedup via WORKATO_LOGS) → RF 7.0 (foreach row, NetSuite API). Parallel sub-pipelines: Rec 10.1 (voided invoices, groups by invoice, builds line items) → RF 10.1. Rec 10.0 (uncollectible, currently stopped) → RF 10.0. Error handling via try/catch with email alerts to danny.mcgean, nan.lu, chase.clark.

Target Layer

NetSuite receives Customer Refunds, Credit Memos, and Journal Entries. Finance uses these for cash reconciliation, accounts receivable management, and financial reporting. The Stripe Refund ID serves as the external ID for idempotent upserts.

Target Layer — Technical

NetSuite SuiteTalk API. Customer Refunds use CUSTREFUND record type; voided invoice Credit Memos use CREDMEMO; uncollectible Journal Entries use JOURNALENTRY. External ID = STRIPE_REFUND_ID for refunds, STRIPE_INVOICE_ID for voided/uncollectible. Custom fields include Stripe Charge ID, Credit Note ID, Automatic Payout ID, and Refund Method.

6. End-to-End Flow

The complete data journey for a Stripe refund follows this sequential path across five systems:

  1. Stripe Billing — A refund is issued against a charge (either manually by support, automatically via credit note, or through the Stripe dashboard). Stripe creates a Refund object linked to the original Charge, with optional Credit Note and Invoice references. The refund has a unique ID (re_xxx), amount, status, and reason.
  2. Fivetran CDC — Continuously replicates Stripe objects (Refund, Charge, Invoice, Credit Note) to INTERCOM_PROD.BASE in Snowflake. The raw STRIPE_REFUND table currently holds 236,683 rows.
  3. Snowflake dbt Pipeline — Transforms raw Stripe data and produces export tables in SHARED_EXTERNAL_WORKATO:
    • NETSUITE_EXPORT_REFUNDS (1,926 rows, 18 cols) — enriched refund data with customer, subsidiary, charge, and payout information
    • NETSUITE_EXPORT_VOIDED_INVOICE (6,749 rows, 24 cols) — voided invoices with line items, amounts, rev-rec dates, and item mapping
    • EXT_NETSUITE_EXPORT_UNCOLLECTIBLE (11,268 rows, 12 cols) — invoices marked uncollectible for bad debt write-off
  4. Workato Orchestration — The 3.0 Master Orchestrator runs 4x daily and calls three sub-pipelines in sequence:
    • Refunds: Rec 7.0 queries NETSUITE_EXPORT_REFUNDS (left-joining NETSUITE_EXPORT_CUSTOMER and NETSUITE_EXPORT_PAYMENT), filters out already-synced records via WORKATO_LOGS, then for each refund calls RF 7.0 which creates a Customer Refund in NetSuite.
    • Voided Invoices: Rec 10.1 queries NETSUITE_EXPORT_VOIDED_INVOICE, groups lines by invoice, builds a line-item array, then calls RF 10.1 which creates a Credit Memo in NetSuite.
    • Uncollectible Invoices: Rec 10.0 (currently stopped, replaced by 10.1 flow) queries the uncollectible export table and creates Journal Entries in NetSuite.
  5. NetSuite ERP — Receives Customer Refund records (CUSTREFUND), Credit Memos (CREDMEMO for voided invoices), and Journal Entries (JOURNALENTRY for uncollectible invoices). External IDs ensure idempotent upserts.
Feedback loop: Workato writes sync results (ns_internal_id, status, error info) back to WORKATO_LOGS in Snowflake. The master recipes use a CTE against WORKATO_LOGS to check is_successful_write_to_netsuite = true for dedup, ensuring no refund is processed twice.

Recipe Detail: Rec 7.0 — Refunds Master (Core SQL)

Snowflake SQL Query (executed by Rec 7.0) 
with stg_workato_log as (
    select
        external_id,
        job_run_completed_at as last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    from intercom_prod.shared_external_workato.WORKATO_LOGS
    where
        external_id is not null
        and source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS'
        and environment_type = 'prod'
        and is_successful_write_to_netsuite = true
    qualify row_number() over (
        partition by external_id
        order by job_run_completed_at desc
    ) = 1
)
select
    r.customer,
    stripe_refund_id,
    r.subsidiary,
    credit_note_id,
    c.category,
    r.stripe_invoice_id,
    stripe_charge_id,
    stripe_automatic_payout_id,
    refund_amount,
    r.date,
    r.created_date,
    r.netsuite_internal_id
FROM
    SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS r
LEFT JOIN
    SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER c
    ON r.customer = c.external_id
LEFT JOIN
    SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT p
    ON r.stripe_charge_id = p.STRIPE_PAYMENT_ID
LEFT JOIN
    stg_workato_log wl on wl.external_id = r.stripe_refund_id
WHERE
    r.date >= :Start_date
    and r.date <= DATEADD(day, -1, CURRENT_DATE)
    and wl.netsuite_internal_id is null
ORDER BY r.created_date

9. Key Business Logic

Deduplication Strategy

Business View

Every refund must be processed exactly once. Before creating a NetSuite record, Workato checks whether the STRIPE_REFUND_ID has already been successfully synced. This prevents duplicate refunds that would cause cash reconciliation errors.

Technical View

A CTE against WORKATO_LOGS filters for source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS' and is_successful_write_to_netsuite = true. It uses ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) to get the latest log entry per refund. The main query LEFT JOINs against this CTE and filters WHERE wl.netsuite_internal_id IS NULL to find unsynced refunds.

Subsidiary Routing

The SUBSIDIARY column in NETSUITE_EXPORT_REFUNDS determines which NetSuite subsidiary the refund is applied to. This is critical for multi-entity financial reporting across Intercom's legal entities (US, Ireland, Australia, etc.).

Customer Lookup

RF 7.0 maps the Stripe customer ID (cus_xxx) to a NetSuite customer record. The NETSUITE_EXPORT_CUSTOMER table provides the mapping between Stripe external IDs and NetSuite internal IDs, along with category classification.

Date Windowing

Rec 7.0 uses a date window: r.date >= :Start_date AND r.date <= DATEADD(day, -1, CURRENT_DATE). The Start_date parameter is passed from the 3.0 orchestrator. The upper bound excludes today's date, giving time for Fivetran to fully sync before processing.

Voided Invoice Line-Item Grouping

Business View

A voided invoice may have multiple line items (subscriptions, add-ons, taxes). All lines for a single invoice must be grouped together to create a single Credit Memo in NetSuite with the correct line-item detail.

Technical View

Rec 10.1 queries all voided invoice rows, extracts pluck('stripe_invoice_id').uniq for unique invoice IDs, then iterates over each unique invoice. For each invoice, it filters the full result set to get all lines, builds a Workato list variable (Invoices) with 16 fields per line item (amount, item, item_description, actual_qty, rev_rec_start_date, rev_rec_end_date, discount_percentage, tax_amount, etc.), and passes the grouped array to RF 10.1 as the invoice_lines parameter.

Uncollectible Invoice Write-Off

Uncollectible invoices (bad debt) create Journal Entries in NetSuite rather than Credit Memos. Rec 10.0 queries NETSUITE_EXPORT_INVOICE (not NETSUITE_EXPORT_REFUNDS) for invoices where stripe_invoice_status = 'uncollectible' and amount != 0, then passes customer, subsidiary, date, invoice total, amount due, and the Stripe Invoice ID as the external ID to RF 10.0.

Note: The Rec 10.0 uncollectible master is currently stopped (as of 2026-02-27). The skip flag on step 5 of the 3.0 orchestrator is set to true, meaning uncollectible invoices are no longer being auto-processed. The voided invoice pipeline (Rec 10.1) may have absorbed some of this functionality.

Error Handling

All recipes use try/catch blocks. The 3.0 orchestrator catches timeout errors and sends email notifications to: danny.mcgean@intercom.io, nan.lu@intercom.io, chase.clark@connorgp.com. Sub-pipeline errors are caught silently (empty catch blocks) to prevent one pipeline failure from blocking the others.

11. Edge Cases

Known Edge Cases

Edge CaseWhat HappensResolution
Refund with no credit note CREDIT_NOTE_ID is NULL in export table RF 7.0 still creates CUSTREFUND; credit note field left blank in NetSuite
Refund for unknown customer NETSUITE_EXPORT_CUSTOMER join returns no match Customer category defaults to NULL; RF 7.0 may need to create/upsert customer first
Duplicate refund ID in WORKATO_LOGS Multiple log entries for the same STRIPE_REFUND_ID CTE uses ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1 to take latest entry only
Same-day refund Refund created today will NOT be picked up (date filter excludes today) By design — r.date <= DATEADD(day, -1, CURRENT_DATE) gives 1-day buffer for Fivetran sync
Voided invoice with $0 lines Rec 10.1 query filters amount != 0 $0 lines are excluded from Credit Memo creation to avoid empty line items
Multi-line voided invoice One invoice has multiple line items with different items, amounts, dates Rec 10.1 groups all lines by stripe_invoice_id using pluck().uniq and builds a line-item array for RF 10.1
Uncollectible invoice pipeline stopped Rec 10.0 is stopped; step 5 of 3.0 orchestrator is skipped New uncollectible invoices are not auto-processed; may need manual intervention or migration to 10.1 voided flow
Sandbox vs. Production SANDBOX_NETSUITE_EXPORT_REFUNDS (6,439 rows) has more rows than production (1,926) Sandbox includes test data; WORKATO_LOGS environment_type = 'prod' filter ensures only production syncs are deduped
Manual reprocessing needed A specific refund needs to be re-synced Use recipe 5.02.7 (Manual) with the specific Stripe_Refund_Id as input — bypasses date window and WORKATO_LOGS dedup
Timeout on large batch Too many refunds in a single run causes Workato timeout 3.0 orchestrator catch block sends email notification; run can be manually resumed from the batch start date
