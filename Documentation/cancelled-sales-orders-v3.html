Rec 9.0 — Cancelled Sales Orders Master Recipe

Recipe ID: 54665267 Status: Active Author: venkatesh.manohar Generated: 2026-04-01 Last Run: 2026-04-01 Version: 18 (by diana.sy)

1. Executive Summary

What it does: This integration automatically cancels (closes) Sales Orders in NetSuite when the corresponding Salesforce contract is marked as "Cancelled." It ensures that the financial system of record (NetSuite) stays synchronized with contract lifecycle events originating from Salesforce.

Why it exists: When a customer cancels their Intercom subscription, the Salesforce contract status changes to "Cancelled." However, the associated Sales Order in NetSuite remains open unless explicitly closed. Open Sales Orders can distort revenue recognition, create billing errors, and cause audit discrepancies. This recipe automates the closure process to eliminate manual intervention and ensure financial accuracy.

High-level flow: Snowflake identifies cancelled contracts that have a matching Sales Order in NetSuite but have not yet been processed. For each one, Workato searches NetSuite for the Sales Order by Salesforce Contract ID, marks all line items as "Closed," sets the cancellation date, and logs the result back to Snowflake for tracking and deduplication.

2. Business Overview

The Problem

When a customer cancels their Intercom subscription, the contract in Salesforce is updated to "Cancelled" status. The data pipeline (dbt) writes this information to the NETSUITE_EXPORT_CONTRACTS table in Snowflake. However, the corresponding Sales Order in NetSuite -- the financial document that drives billing and revenue recognition -- remains open. Without automation, finance teams would need to manually find and close each cancelled Sales Order, which is time-consuming and error-prone.

What This Integration Solves

  • Automated Sales Order Closure: Cancelled contracts in Salesforce automatically trigger Sales Order closure in NetSuite
  • Revenue Accuracy: Prevents revenue from being recognized on cancelled contracts
  • Billing Hygiene: Prevents future billing runs from picking up cancelled orders
  • Audit Trail: Every closure is logged to the WORKATO_LOGS table with success/failure status and the NetSuite internal ID

4. Architecture Overview

Layered Architecture

Source Layer (Salesforce)

Business: Contract cancellation events originate in Salesforce when a customer churns or downgrades.

Technical: Salesforce contract data is replicated to Snowflake via Fivetran and transformed by dbt into the NETSUITE_EXPORT_CONTRACTS staging table.

Data Layer (Snowflake)

Business: Provides a reliable staging area where the integration can identify which cancellations need processing.

Technical: Three key tables are involved: NETSUITE_EXPORT_CONTRACTS (41K rows, cancelled contract data), NETSUITE_EXPORT_SALES_ORDER (180K rows, confirms NS sales order existence), and WORKATO_LOGS (66M rows, deduplication/audit trail).

Orchestration Layer (Workato)

Business: Coordinates the entire cancellation workflow automatically on a schedule or on-demand trigger.

Technical: Master recipe (Rec 9.0, ID: 54665267) runs a SQL query and iterates. For each row, it calls child recipe (RF 9.0, ID: 54665277) which performs the NetSuite search + update. The child recipe uses try/catch to handle errors gracefully.

Target Layer (NetSuite)

Business: Sales Orders are closed to reflect the cancellation, ensuring accurate financials.

Technical: RF 9.0 performs a search_object_v2 on SalesOrder records using the Salesforce contract ID, then update_object to close all line items and set the cancellation date custom field (custbody_cg_cancellation_close).

6. End-to-End Flow

Level 0 -- Simple View

A customer cancels their Intercom subscription. Salesforce records the cancellation. Snowflake identifies it as unprocessed. Workato closes the corresponding Sales Order in NetSuite. The result is logged.

Level 1 -- Detailed View

1
Extraction: Rec 9.0 runs a SQL query against Snowflake that joins NETSUITE_EXPORT_CONTRACTS with NETSUITE_EXPORT_SALES_ORDER and WORKATO_LOGS. It returns contracts where contract_status = 'Cancelled', a matching Sales Order exists in NetSuite (so.netsuite_internal_id IS NOT NULL), but the contract itself has not been logged to WORKATO_LOGS yet (wl.netsuite_internal_id IS NULL).
2
Iteration: For each row returned (each cancelled contract), the master recipe calls RF 9.0 with the salesforce_contract_id and cancellation_date as parameters.
3
NetSuite Search: RF 9.0 searches NetSuite for Sales Orders matching the Salesforce contract ID via the custom field custbodycustentity_sf_contract_id.
4
Concurrency Wait: A 1-second sleep prevents concurrent API rate-limit issues with NetSuite.
5
Sales Order Update: Each matching Sales Order is updated: all line items are set to isClosed = true, and the cancellation date custom field is populated. The subsidiary determines whether additional custom fields are set.
6
Logging: On success, a record is written to WORKATO_LOGS with is_successful_write_to_netsuite = true and the NetSuite internal ID. On failure, the error is caught and logged with is_successful_write_to_netsuite = false.

Level 2 -- Subsidiary Branching

The cancellation date field behavior depends on the subsidiary. If the subsidiary internal ID is 4 (indicating a specific legal entity), the cancellation date formatting may differ. This is handled via a ternary condition in the custom field mapping of the update step.

7. Processing Stages

Stage 1: Ingestion (Snowflake SQL Query)

What happens: The master recipe executes a complex SQL query that identifies cancelled contracts needing Sales Order closure in NetSuite.

How it works: The query uses a CTE (stg_workato_log) to get the latest log entry per contract from WORKATO_LOGS, then joins NETSUITE_EXPORT_CONTRACTS with NETSUITE_EXPORT_SALES_ORDER on salesforce_contract_id. Filters ensure: (a) contract is cancelled, (b) no existing log entry with a NetSuite ID (i.e., not already processed), and (c) a Sales Order exists in NetSuite.

Why it matters: This deduplication-at-source approach prevents reprocessing, saving API calls and avoiding duplicate closures.

View SQL Query 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CONTRACTS'
        AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id
        ORDER BY job_run_completed_at DESC
    ) = 1
)
SELECT DISTINCT
    c.salesforce_contract_id,
    c.customer,
    c.cancellation_date
FROM shared_external_workato.netsuite_export_contracts c
LEFT JOIN shared_external_workato.netsuite_export_sales_order so
    ON c.salesforce_contract_id = so.salesforce_contract_id
LEFT JOIN stg_workato_log wl
    ON wl.external_id = c.salesforce_contract_id
WHERE c.contract_status = 'Cancelled'
    AND wl.netsuite_internal_id IS NULL
    AND so.netsuite_internal_id IS NOT NULL

Stage 2: Iteration (For Each Contract)

What happens: The master recipe loops through each row returned by the query.

How it works: Workato's built-in "For Each" loop processes rows sequentially, passing salesforce_contract_id and cancellation_date to the child recipe function.

Why it matters: Sequential processing ensures orderly execution and prevents NetSuite API throttling.

Stage 3: NetSuite Search + Update (RF 9.0)

What happens: For each contract, NetSuite is searched for matching Sales Orders, then each is updated to close all line items.

How it works: search_object_v2 finds SalesOrder records by the custom field. After a 1-second concurrency wait, update_object sets isClosed = true on all line items and populates the cancellation date custom field. The subsidiary ID determines date formatting logic.

Why it matters: Closing line items prevents future billing and stops revenue recognition on cancelled contracts.

Stage 4: Logging (WORKATO_LOGS)

What happens: The outcome (success or failure) is recorded for every contract processed.

How it works: On success, the mapper writes source_table_name, external_id, is_successful_write_to_netsuite = true, and the netsuite_internal_id. On failure (caught by try/catch), it writes is_successful_write_to_netsuite = false with the error details. Both paths also log a message to the Workato job report.

Why it matters: Logging enables deduplication (preventing reprocessing), auditing, and error investigation.

9. Key Business Logic

Deduplication Logic

The SQL query ensures each cancelled contract is processed exactly once. It checks the WORKATO_LOGS table for an existing entry with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CONTRACTS'. Only contracts with no matching log entry (or no successful NetSuite internal ID) are returned for processing.

Sales Order Closure Logic

When a Sales Order is found in NetSuite, all line items are closed by setting isClosed = true. This is a bulk operation -- the recipe does not selectively close individual lines. The entire order is effectively cancelled.

Cancellation Date Handling

The cancellation date from Salesforce (stored in NETSUITE_EXPORT_CONTRACTS.cancellation_date) is passed to the NetSuite custom field custbody_cg_cancellation_close. The formatting logic includes a subsidiary check:

  • If subsidiary internal ID = 4: may apply specific date formatting
  • Otherwise: standard date format is used

Sales Order Existence Check

The SQL query includes so.netsuite_internal_id IS NOT NULL which ensures only contracts with an existing Sales Order in NetSuite are processed. Contracts where the Sales Order was never created (e.g., cancelled before order creation) are excluded.

11. Edge Cases & Special Scenarios

ScenarioWhat HappensImpact
No Sales Order in NetSuite The SQL query filters these out (so.netsuite_internal_id IS NOT NULL). The contract is never sent to RF 9.0. None -- safely excluded. These are likely contracts that were cancelled before a Sales Order was created.
Multiple Sales Orders per Contract The NetSuite search returns all matching Sales Orders. The DISTINCT in the SQL and the fact that RF 9.0 updates the first found order means only one is typically processed per invocation. Low risk -- most contracts have a single Sales Order. If multiple exist, the search result handles it.
Already-closed Sales Order NetSuite will accept the update (setting isClosed on already-closed lines is idempotent). No error occurs. None -- safe to reprocess.
NetSuite API Failure The try/catch block catches the error. RF 9.0 logs the failure to WORKATO_LOGS with the error details. The recipe continues processing the next contract. The failed contract will be retried on the next run (since it won't have a successful log entry).
NetSuite Rate Limiting The 1-second sleep(1) between search and update mitigates this. If a 429 error occurs, it is caught and logged. Temporary -- will succeed on retry.
Subsidiary-specific Date Formatting Subsidiary ID 4 triggers different cancellation date handling via a ternary condition in the custom field mapping. Ensures correct date formatting per legal entity requirements.
Contract re-cancellation (duplicate events) The WORKATO_LOGS dedup check prevents reprocessing. If a contract was already successfully processed, it is skipped. None -- deduplication is robust.

12. Operational Notes

Deduplication

Deduplication is achieved at the SQL query level using a LEFT JOIN to WORKATO_LOGS. The CTE uses QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1 to get only the most recent log entry per contract. Only contracts where wl.netsuite_internal_id IS NULL (meaning no successful prior run) are returned.

Note: If a contract was previously attempted but failed (logged with is_successful_write_to_netsuite = false), it will be retried because the log entry will have netsuite_internal_id = NULL.

Error Handling

RF 9.0 wraps the entire NetSuite interaction in a try/catch block. On success, a success log is written. On error, the catch block writes an error log with the error type and message. The master recipe's For Each loop continues to the next contract regardless of individual failures.

Known limitation: The catch block has max_retry_count = 0 and retry_interval = 2 seconds. This means there are no automatic retries within a single job run. Retries only occur on the next scheduled run.
