Rec 3.0 — Sales Order Master Recipe

Automated Sales Order creation and synchronization from Snowflake to NetSuite

Recipe ID: 54665261 Status: Active Author: venkatesh.manohar Created: 2024-10-10 Last Updated: 2026-01-20 Jobs Succeeded: 942 Jobs Failed: 0

1. Executive Summary

The Rec 3.0 Sales Order Master Recipe is a Workato integration that automatically creates and updates Sales Orders in NetSuite based on Salesforce contract data staged in Snowflake. It is a core component of Intercom's Order-to-Cash (OTC) pipeline, ensuring that every activated Salesforce contract becomes a properly structured Sales Order in the company's ERP system (NetSuite).

Business View

When a sales deal closes in Salesforce, this system automatically creates the corresponding Sales Order in NetSuite — the document that authorizes billing, revenue recognition, and fulfillment. Without it, finance would need to manually key every order, which is error-prone and unscalable.

Technical View

A scheduled Workato recipe function queries NETSUITE_EXPORT_SALES_ORDER in Snowflake, groups line items by contract, and calls child recipe functions to upsert Customers and create/update Sales Orders in NetSuite via SuiteTalk API. Deduplication is handled through a log table.

Reliability: This recipe has processed 942 runs with a 100% success rate (0 failures). It runs on a scheduled cadence (approximately every 5 hours).

2. Business Overview

What Problem Does This Solve?

Intercom's sales process starts in Salesforce (CRM) where deals are negotiated. Once a contract is activated, the finance team needs a Sales Order in NetSuite to trigger invoicing, revenue recognition, and financial reporting. This integration automates that handoff.

The Business Flow

  1. Sales closes a deal in Salesforce, and a contract is activated
  2. Data pipelines (dbt/Fivetran) transform contract data into a staging table in Snowflake
  3. This recipe picks up new/unprocessed orders from Snowflake on a regular schedule
  4. For each contract:
    • The customer record is created or updated in NetSuite (if not already present)
    • A Sales Order is created with all line items (products, quantities, pricing, billing schedules)
  5. NetSuite now has the Sales Order, which drives invoicing and revenue recognition

4. Architecture Overview

Source Layer (Salesforce → Snowflake)

Contract and customer data originates in Salesforce and is replicated into Snowflake via Fivetran. dbt models then transform this raw data into export-ready staging tables with all the fields NetSuite requires — including billing schedules, item codes, subsidiaries, and pricing.

Data Layer (Snowflake)

Three key tables in SHARED_EXTERNAL_WORKATO: the Sales Order table (180K+ rows), Customer lookup table, and Item mapping table. A WORKATO_LOGS table tracks which records have already been processed to prevent duplicates.

Orchestration Layer (Workato)

Workato reads unprocessed records, groups them by contract, and orchestrates the creation of customers and sales orders in the correct sequence. It handles edge cases like amendments, co-terms, and 100% discounts.

Target Layer (NetSuite)

NetSuite receives Customer upserts and Sales Order creates/updates via SuiteTalk API. Custom fields link back to Salesforce IDs for traceability. Billing schedules and revenue recognition dates are set per line item.

6. End-to-End Flow

Level 0: Simple View

Salesforce contract data → Staged in Snowflake → Workato reads new orders → Creates Customer + Sales Order in NetSuite → Logs completion.

Level 1: Detailed View

  1. Extraction: Rec 3.0 queries NETSUITE_EXPORT_SALES_ORDER with a LEFT JOIN to WORKATO_LOGS to find unprocessed records (where netsuite_internal_id IS NULL in the log). Filters for activated contracts with non-zero amounts (or 100% discounts).
  2. Grouping: Results are grouped into a unique contract list. Each unique salesforce_contract_id becomes one batch, and all line items for that contract are collected into a SalesOrderLines payload.
  3. Routing: For each contract, the master recipe calls RF 3.0 (54665270) with the full contract payload.
  4. Customer Processing: RF 3.0 first calls Rec 1.01 (54665259) which looks up the customer in Snowflake and then calls RF 1.0 (54251831) to upsert the customer in NetSuite.
  5. Sales Order Processing: RF 3.0 checks if a Sales Order already exists in NetSuite (by searching for the SFDC Contract ID). If it exists, it updates; if not, it creates a new one. Line items are built with proper item codes, quantities, pricing, rev rec dates, and billing schedules.
  6. Logging: Each processed record is logged to WORKATO_LOGS with the NetSuite internal ID, preventing reprocessing on the next run.

Level 2: Branching Logic

New Sales Order Path

When no existing SO is found in NetSuite for the given contract, the system creates a brand new Sales Order using netsuite.add_object. All line items are assembled from the Snowflake data and attached.

Update Sales Order Path

When an existing SO is found (by SFDC Contract ID search), the system updates it using netsuite.update_object. This handles amendments where new line items are added to an existing order. If the update fails, the error is caught and logged.

7. Processing Stages

Stage 1: Ingestion (Rec 3.0)

What Happens

The recipe queries Snowflake to find all sales order records that have not yet been processed into NetSuite. It joins three tables — sales orders, customers, and items — to build a complete picture of each order.

How It Works

A CTE first builds a stg_workato_log subquery that finds the latest log entry per external_id. The main query LEFT JOINs to this and filters for rows where netsuite_internal_id IS NULL — meaning the record has never been successfully written to NetSuite. Additional filters: contract_status = 'Activated' and (amount != 0 OR total_contract_value = 0) OR (amount = 0 AND discount = 100).

View SQL Query 
WITH stg_workato_log AS (
    SELECT external_id,
           job_run_completed_at AS last_successful_job_run_completed_at,
           netsuite_internal_id, source_table_name, error_type, error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_SALES_ORDER'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1
)
SELECT so.external_id, customer, salesforce_contract_id, date, original_contract_start_date,
       salesforce_opportunity_id, so.status, po_number, memo, customer_payment_type,
       so.terms, item, item_description, actual_qty, amount, list_price, discount,
       rev_rec_start_date, rev_rec_end_date, sfdc_order_product_id, billing_schedule,
       total_contract_value, c.subsidiary, i.item_code, prorate_multiplier,
       so.netsuite_internal_id, default_subscription_term, so.created_date,
       order_type, proserve_type, so.reseller, is_co_term, contract_term, pricebook_name
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_SALES_ORDER so
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER c ON so.customer = c.external_id
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_ITEM i ON so.item = i.billing_plan_id
LEFT JOIN stg_workato_log wl ON so.external_id = wl.external_id
WHERE so.date >= :Start_Date
  AND wl.netsuite_internal_id IS NULL
  AND ((so.amount != 0 OR so.total_contract_value = 0) OR (so.amount = 0 AND discount = 100))
  AND contract_status = 'Activated'
ORDER BY so.date

Stage 2: Grouping (Rec 3.0)

What Happens

Multiple line items belonging to the same contract are grouped together. A single Salesforce contract might have 5-10 product lines (e.g., seats, resolutions, add-ons), and they all need to be on one Sales Order.

How It Works

The recipe iterates through query results building a Unique_Contract_List variable and a Contracts list. For each row, it checks if the salesforce_contract_id has been seen; if not, it adds the contract-level data. Line items are batched into the SalesOrderLines parameter sent to the child recipe.

Stage 3: Customer Processing (Rec 1.01 → RF 1.0)

What Happens

Before creating a Sales Order, the system ensures the customer exists in NetSuite. It looks up the customer by ID, and either creates a new customer record or updates the existing one with the latest information from Salesforce.

How It Works

Rec 1.01 (54665259) queries NETSUITE_EXPORT_CUSTOMER by external_id. For each matching row, it calls RF 1.0 (54251831) which performs a NetSuite upsert with fields: ExternalID, CompanyName, Category, Subsidiary, SFDC ID, Status, Email, Billing/Shipping address, Terms, and Reseller. The upsert is idempotent (safe to re-run). RF 1.0 has processed 230,903 records with zero failures.

Stage 4: Sales Order Creation (RF 3.0)

What Happens

The core Sales Order is created or updated in NetSuite with all line items, billing schedules, and financial details. The system handles both new orders and amendments to existing ones.

How It Works

RF 3.0 (54665270) searches NetSuite for an existing SO by SFDC Contract ID. If found, it builds update line items; if not, it creates a new SO. Line items are classified into different lists based on item type and billing schedule. For amendments, HR 1.0 (54665268) calculates the billing period end date using contract dates and billing frequency. Custom Ruby code blocks handle complex line item assembly logic.

Stage 5: Logging & Deduplication

What Happens

After successfully processing each order, the system records it so the same order is not processed again. If an error occurs, it is logged for investigation.

How It Works

RF 3.0 writes to WORKATO_LOGS via a mapper action. The log entry includes the external_id, netsuite_internal_id, source_table_name, and timestamp. On the next run, the LEFT JOIN in Stage 1 filters out any external_id that already has a log entry with a netsuite_internal_id.

9. Key Business Logic

Order Type Classification

The order_type field from the sales order determines how the order is processed. This maps to how the deal was structured in Salesforce:

Order TypeDescriptionImpact on Processing
New BusinessA brand-new contract with a customerCreates new SO with standard billing
AmendmentChanges to an existing contract (upgrade, add-on)May update existing SO; calls HR 1.0 for billing period calculation
RenewalContract renewal at end of termCreates new SO linked to same customer
Co-TermAdding products aligned to existing contract end dateProrated amounts; is_co_term = true

Billing Period Calculation (HR 1.0)

For amendments, the billing period end date must be calculated to align with the existing billing cycle. HR 1.0 takes four inputs:

  • Contract Start Date — when the original contract began
  • Contract End Date — when the contract expires
  • Amendment Date — when the change takes effect
  • Monthly Billing Frequency — how often billing occurs (e.g., 1 = monthly, 3 = quarterly, 12 = annual)

The recipe iterates through billing periods from the contract start date, adding the billing frequency each time, until it passes the amendment date. The output includes the billing period end date, current cycle number, total cycles, and remaining cycles.

Line Item Assembly

RF 3.0 classifies each line item into separate lists based on whether it needs special handling. Key factors include:

  • Item code mapping: The billing_plan_id from Salesforce is mapped to a NetSuite item_code via the NETSUITE_EXPORT_ITEM table
  • Revenue recognition dates: rev_rec_start_date and rev_rec_end_date are set per line item for proper revenue recognition in NetSuite
  • Discount handling: The discount percentage is applied at the line level; 100% discounts are treated specially (allowed through with $0 amount)
  • Prorate multiplier: For mid-cycle amendments, the prorate_multiplier adjusts amounts proportionally

Customer Creation Logic

RF 1.0 uses NetSuite's upsert operation (identified by external_id) which is idempotent:

  • If the customer does not exist, it is created with all fields
  • If the customer already exists, its information is updated to the latest from Salesforce
  • After upsert, the recipe searches for the customer's subsidiary internal ID and returns it for use in the Sales Order
  • Address sub-records (billing and shipping) are created as needed

11. Edge Cases & Special Scenarios

Missing Customer

If the customer external_id in the sales order does not match any record in NETSUITE_EXPORT_CUSTOMER, the LEFT JOIN in the main query still returns the sales order row — but with NULL subsidiary and address fields. Rec 1.01 will then look up the customer separately by ID and attempt to create it. If the customer truly does not exist in the source, the process will log an error.

Missing Sales Order in NetSuite

When RF 3.0 searches NetSuite for an existing Sales Order (by SFDC Contract ID) and finds none, it takes the "create new" path using netsuite.add_object. This is the normal path for new contracts.

Existing Sales Order (Amendment)

When an SO is found in NetSuite, the system updates it with new/changed line items. This uses a try/catch block — if the update fails (e.g., the SO is already approved or in a locked period), the error is caught, logged, and processing continues for other contracts.

Zero Amount with 100% Discount

The filter (amount != 0 OR total_contract_value = 0) OR (amount = 0 AND discount = 100) ensures that legitimate $0 orders (promotional, 100% discount) are still processed, while accidental $0 records are excluded.

Negative Amounts

Negative amounts can appear in amendment scenarios (downgrade/reduction). These flow through the same Sales Order creation path and appear as negative line items in NetSuite.

Reseller Orders

When reseller is populated on the sales order, this value is passed through to both the Customer record and the Sales Order in NetSuite, enabling proper channel partner attribution.

Professional Services Items

Items with a proserve_type value follow slightly different billing rules. The line item classification logic in RF 3.0 routes these to the appropriate item list.

12. Operational Notes

Deduplication Mechanism

Deduplication relies on the WORKATO_LOGS table in Snowflake. After each successful Sales Order creation, a log entry is written with the external_id and netsuite_internal_id. On subsequent runs, the main SQL query LEFT JOINs to this table and only picks up rows where no successful log entry exists. This means:

  • Successful records are never reprocessed
  • Failed records (where netsuite_internal_id was not written) will be retried on the next run
  • The dedup is based on external_id (line-item level), not contract level

Error Handling

The recipe uses a multi-level try/catch pattern:

  • Rec 3.0 (Master): Wraps the entire batch in a try/catch. If the batch fails, it updates an Integration Tracker lookup table.
  • RF 3.0 (SO Processing): Has its own try/catch around the NetSuite operations. Individual order failures are logged but do not stop processing of other orders.
  • RF 1.0 (Customer): Try/catch distinguishes between "expected" errors (handled gracefully) and unexpected ones (logged for investigation).

Schedule & Frequency

The recipe runs approximately every 5 hours (based on job history). It accepts a Start_Date parameter that controls how far back it looks for unprocessed records. Recent jobs show consistent completion times of 4-15 seconds for most runs, with larger batches (when new contracts accumulate) taking up to 10 minutes.

Key Limitations

  • Only processes records with contract_status = 'Activated'. Draft or pending contracts are not picked up.
  • The Start_Date parameter must be set correctly; records before this date will not be queried.
  • If the Snowflake-to-Salesforce data pipeline (Fivetran/dbt) is delayed, new contracts will not appear in the export table until the pipeline catches up.
  • NetSuite API rate limits may affect processing speed during large batch runs.
Important: If a record fails processing, it will be retried on every subsequent run until it succeeds or the root cause is resolved. Monitor WORKATO_LOGS for records with error_type populated but no netsuite_internal_id.
