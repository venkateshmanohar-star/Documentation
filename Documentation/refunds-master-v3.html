Rec 7.0 — Refunds Master Recipe

Automated Stripe-to-NetSuite Refund Processing Pipeline
Recipe ID: 54665265 Status: Active Author: venkatesh.manohar Last Run: 2026-04-01 Success Rate: 824/824 (100%) Generated: 2026-04-01

1. Executive Summary

The Refunds Master Recipe (Rec 7.0) automates the creation of refund records in NetSuite whenever a customer is refunded through Stripe. It reads pending refunds from Snowflake, checks whether each has already been processed, and then creates the necessary Credit Memo and Customer Refund transactions in NetSuite.

In plain terms: When Intercom refunds money to a customer via Stripe, this system ensures the company's accounting books in NetSuite are updated automatically — creating both the credit memo (the "why" of the refund) and the customer refund record (the "how" the money was returned).

High-Level Flow

  1. Stripe processes a refund → data lands in Snowflake via Fivetran/dbt
  2. Rec 7.0 queries Snowflake for unprocessed refunds (filtered against the Workato deduplication log)
  3. For each refund row, RF 7.0 is called to create a Credit Memo and Customer Refund in NetSuite
  4. Outcomes are logged back to Snowflake for audit and deduplication

2. Business Overview

What problem does this solve?

When a customer receives a refund via Stripe, the Finance team needs that refund reflected accurately in NetSuite (the company's ERP / accounting system). Without automation, an accountant would need to manually:

  • Look up the customer in NetSuite
  • Create a Credit Memo (a document that says "we owe the customer X dollars")
  • Create a Customer Refund (a record that says "we sent X dollars back to the customer's payment method")

This recipe eliminates that manual work entirely, processing hundreds of refunds per month with zero failures.

What happens from a business perspective?

Business View

A Stripe refund triggers an automated accounting entry. The system creates two linked records in NetSuite: a Credit Memo (adjusts the invoice) and a Customer Refund (records the actual money returned). The finance team sees accurate, real-time refund data without lifting a finger.

Technical View

A Workato Recipe Function is called with a start date parameter. It executes a SQL query against Snowflake joining NETSUITE_EXPORT_REFUNDS with NETSUITE_EXPORT_CUSTOMER, NETSUITE_EXPORT_PAYMENT, and WORKATO_LOGS to find unprocessed rows. Each row is passed to the child recipe RF 7.0 (ID: 54665275) for NetSuite record creation.

4. Architecture Overview

The system follows a four-layer architecture pattern:

Source Layer (Stripe)

Business: Refunds are initiated here — either by support teams, automated systems, or customer self-service. Each refund has a unique ID, amount, charge reference, and associated invoice.

Technical: Stripe refund events are captured by Fivetran and loaded into Snowflake's FIVETRAN_STRIPE.REFUND table. dbt models then transform this into the NETSUITE_EXPORT_REFUNDS staging table.

Data Layer (Snowflake)

Business: A staging area that prepares refund data for accounting. It enriches refund records with customer classification (subsidiary, category) and payment details (charge ID, payout ID).

Technical: The master recipe runs a complex SQL query that joins 4 tables and uses a CTE to filter out already-processed refunds via the WORKATO_LOGS deduplication table.

Orchestration Layer (Workato)

Business: The "traffic controller" that reads pending refunds and ensures each one gets processed exactly once, with proper error handling and logging.

Technical: Rec 7.0 iterates over SQL results and calls RF 7.0 synchronously for each row. RF 7.0 handles the NetSuite API calls with try/catch blocks and retry logic (3 retries on Customer Refund creation).

Target Layer (NetSuite)

Business: The final destination. Two accounting records are created per refund: a Credit Memo (reduces what the customer owes) and a Customer Refund (records the money returned).

Technical: NetSuite SOAP API calls via Workato connector. Uses search_object_v2 for deduplication check, add_object for Credit Memo and Customer Refund creation. Custom forms 114 and 113 respectively.

6. End-to-End Flow

Level 0 — Simple View

A Stripe refund happens. The system automatically checks Snowflake for any refunds that haven't been recorded in NetSuite yet. For each one, it creates two accounting records in NetSuite: a Credit Memo and a Customer Refund. Everything is logged for audit.

Level 1 — Detailed View

  1. Extraction: Rec 7.0 runs a SQL query against Snowflake that joins the refunds table with customer data and payment data. It filters out any refund that already has a successful entry in the WORKATO_LOGS table.
  2. Iteration: For each unprocessed refund row, the master recipe calls the child recipe RF 7.0 synchronously, passing 11 parameters (customer, subsidiary, date, Stripe IDs, amount, etc.).
  3. Deduplication: RF 7.0 first searches NetSuite for an existing Customer Refund with the same external ID. If found, it skips creation.
  4. Credit Memo Creation: If no existing refund is found, RF 7.0 creates a Credit Memo in NetSuite (custom form 114) with the refund details.
  5. Customer Refund Creation: Using the Credit Memo, RF 7.0 creates a Customer Refund record (custom form 113) with 3 retries on failure.
  6. Logging: Success or failure is logged via Workato's logger, which writes back to WORKATO_LOGS in Snowflake.

Level 2 — Branching Logic

The recipe handles two main branches:

  • New Refund (no existing Customer Refund in NS): Create Credit Memo → Create Customer Refund → Log success
  • Already Processed (existing Customer Refund found): Skip creation → Log as already exists
  • Error Path: If Credit Memo or Customer Refund creation fails, the error is caught, mapped, and logged. Customer Refund has 3 automatic retries before failing.

7. Processing Stages

Stage 1: Ingestion (Snowflake Query)

What Happens

The system pulls all Stripe refunds from the staging table that haven't been successfully processed yet. It also enriches each refund with customer classification (subsidiary, category) and payment details.

How It Works

A SQL query with a CTE (stg_workato_log) joins NETSUITE_EXPORT_REFUNDS with NETSUITE_EXPORT_CUSTOMER on customer = external_id, NETSUITE_EXPORT_PAYMENT on stripe_charge_id = STRIPE_PAYMENT_ID, and filters out rows present in WORKATO_LOGS where is_successful_write_to_netsuite = true. Date range: from input Start_date to yesterday.

Why it matters: This is the "smart filter" that prevents duplicate processing and ensures only new refunds are sent to NetSuite.

Stage 2: Iteration & Dispatch

What Happens

Each refund record is processed one at a time. The master recipe hands off each record to a specialized "worker" recipe that knows how to create the correct NetSuite records.

How It Works

Rec 7.0 uses a foreach loop over the SQL result rows. For each row, it calls RF 7.0 (Recipe Function, ID 54665275) synchronously, passing 11 parameters including Customer, Subsidiary, Date, Stripe Refund ID, Stripe Charge ID, Stripe Invoice ID, Credit Note ID, Payout ID, Refund Amount, external_id, and base_external_id.

Stage 3: Deduplication Check

What Happens

Before creating anything, the system checks NetSuite to see if this refund was already recorded. This prevents duplicate accounting entries.

How It Works

RF 7.0 searches NetSuite for a Customer Refund record with a matching external ID. If CustomerRefunds.size == 0 (no match found), it proceeds to create records. Otherwise, it skips.

Stage 4: NetSuite Record Creation

What Happens

Two linked accounting records are created: (1) a Credit Memo that adjusts the customer's balance, and (2) a Customer Refund that records the actual money returned via their payment method.

How It Works

Step 1: netsuite.add_object creates a Credit Memo (custom form 114). A Ruby code block processes the response. Step 2: netsuite.add_object creates a Customer Refund (custom form 113), wrapped in a try/catch with 3 retries. The Credit Memo internal ID is linked to the Customer Refund.

Stage 5: Logging & Audit

What Happens

Every outcome is recorded: success, failure, or skip. This creates a complete audit trail and feeds back into the deduplication filter for future runs.

How It Works

Three separate mapper+logger pairs handle different outcomes: (1) successful creation, (2) Customer Refund retry failure, and (3) top-level error catch. Logs are written to WORKATO_LOGS in Snowflake with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_REFUNDS'.

9. Key Business Logic

Refund Processing Logic

Every Stripe refund follows this decision path:

  1. Date filter: Only refunds between the Start_date parameter and yesterday are considered. This prevents processing refunds that are still "in flight."
  2. Deduplication (Snowflake-level): The SQL query LEFT JOINs to WORKATO_LOGS and filters out any refund where wl.netsuite_internal_id IS NOT NULL — meaning it was already successfully written to NetSuite.
  3. Deduplication (NetSuite-level): RF 7.0 searches for an existing Customer Refund by external ID. This double-check ensures no duplicates even if the log write failed.
  4. Credit Memo first: The Credit Memo is always created before the Customer Refund, because the Customer Refund references the Credit Memo (it "applies" the credit).
  5. Ruby transformation: Custom Ruby code runs after both the NetSuite search and the Credit Memo creation. This likely handles data formatting and response parsing.

Customer Lookup

The refund is matched to a NetSuite customer using the customer field from NETSUITE_EXPORT_REFUNDS, which maps to external_id in NETSUITE_EXPORT_CUSTOMER. The customer's category and subsidiary (US or IE) determine which NetSuite subsidiary the records are created under.

Subsidiary Handling

Intercom operates through multiple legal entities. The subsidiary field (US or IE) from the refund table determines which NetSuite subsidiary the Credit Memo and Customer Refund are created under. This is critical for correct multi-entity accounting.

Amount Handling

The refund_amount from Snowflake is passed directly to the Credit Memo amount field. The Customer Refund then references the Credit Memo, inheriting the amount.

11. Edge Cases & Special Scenarios

Already-Processed Refund

Scenario: A refund was already processed in a previous run but the log write failed.
Handling: RF 7.0 performs a secondary check by searching NetSuite for an existing Customer Refund with the same external ID. If found (size ≠ 0), creation is skipped and the outcome is logged.

NetSuite Customer Refund Creation Failure

Scenario: The Customer Refund creation in NetSuite fails (e.g., network timeout, validation error).
Handling: The inner try/catch block retries up to 3 times with a 2-second interval. If all retries fail, the error is captured by a mapper (Step 10) and logged (Step 11). The Credit Memo that was already created will remain in NetSuite — a future run or manual intervention can create the missing Customer Refund.

Top-Level Error

Scenario: An unexpected error occurs in RF 7.0 outside the Customer Refund creation (e.g., Credit Memo creation fails, Ruby code error).
Handling: The outer try/catch (no retries) captures the error. A mapper (Step 15) and logger (Step 16) record the failure. The master recipe's foreach loop continues processing the next row — one failure does not stop the entire batch.

Missing Customer in NetSuite

Scenario: The Stripe customer doesn't have a matching NetSuite customer record.
Handling: The Credit Memo creation would fail because NetSuite requires a valid customer reference. This would be caught by the outer try/catch and logged. The refund would need to be reprocessed after the customer is created (typically handled by Recipe 1.0 in the broader OTC pipeline).

Date Boundary

Scenario: Refunds created today are not yet processed.
Handling: The SQL query filters r.date <= DATEADD(day, -1, CURRENT_DATE), meaning only refunds through yesterday are included. This prevents processing refunds that may still be in an intermediate state in Stripe.

12. Operational Notes

Deduplication Strategy

This recipe uses a two-tier deduplication approach:

  1. Snowflake-level (pre-filter): The SQL CTE joins to WORKATO_LOGS and excludes any stripe_refund_id that has a successful log entry. This prevents re-processing at the query level.
  2. NetSuite-level (safety check): RF 7.0 searches NetSuite for an existing Customer Refund by external ID. This catches edge cases where the refund was created but the log write failed.
