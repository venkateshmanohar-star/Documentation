1. Executive Summary

What it does: This integration automatically synchronizes Customer Balance Adjustments (CBAs) from Stripe into NetSuite as Credit Memos and Invoice adjustments. When a customer receives a credit, refund adjustment, or balance change in Stripe, this recipe ensures the corresponding financial record is created in NetSuite for accurate accounting.

Why it exists: Stripe tracks customer balance changes (credits, adjustments, invoice applications) as Customer Balance Transactions. NetSuite, as Intercom's ERP and system of record for finance, needs these reflected as Credit Memos or Invoice updates. Without this automation, the finance team would need to manually create thousands of journal entries each month.

High-level flow: Snowflake staging table → Workato reads un-synced CBA records → For each record, determines the CBA type → Creates or updates the appropriate NetSuite record (Credit Memo or Invoice) → Logs the result back to Snowflake for deduplication tracking.

2. Business Overview

The Problem

When Intercom issues credits to customers — whether due to service issues, contract amendments, invoice disputes, or credit note refunds — these changes happen in Stripe as "Customer Balance Adjustments." However, the company's accounting books live in NetSuite. Every credit that affects a customer's balance must be mirrored in NetSuite to maintain accurate financial reporting, revenue recognition, and audit compliance.

What Happens from a Business Perspective

A dbt model in Snowflake prepares CBA records from Stripe data and enriches them with customer metadata. This recipe picks up those records and, depending on the type of adjustment, creates the right NetSuite transaction:

  • Adjustment (negative amount): Creates a Credit Memo against the customer's most recent open invoice
  • Adjustment (positive amount / no invoice): Creates a standalone Invoice plus links existing Credit Memos to it
  • Invoice Too Small: Creates a Credit Memo against a specific Stripe invoice
  • Applied to Invoice: Updates an existing NetSuite Invoice or applies credit memos to matching invoices
  • Credit Note / Unapplied from Invoice: Handled by dedicated branches in the processing logic

4. Architecture Overview

Source Layer: Stripe

Stripe captures every credit, refund, and balance change as a Customer Balance Transaction. Both US and EU Stripe accounts feed data via Fivetran into Snowflake.
Source tables: fivetran.fivetran_stripe.customer_balance_transaction (US) and fivetran.fivetran_stripe_eu.customer_balance_transaction (EU). Synced by Fivetran on a regular cadence.

Data Layer: Snowflake

A dbt model transforms raw Stripe data into a clean staging table with customer metadata, subsidiary information, and deduplication status. Only records that have not been previously synced are included.
  • NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS — 20,715 rows, staging table for CBAs
  • NETSUITE_EXPORT_CUSTOMER — 320,879 rows, customer dimension with subsidiary and NS internal ID
  • WORKATO_LOGS — 66M rows, deduplication/audit log

Orchestration Layer: Workato

Two recipes work together: a parent recipe queries Snowflake for un-synced records and loops through them, calling a child recipe for each record. The child recipe contains all the business logic for determining what type of NetSuite record to create.
Parent: Rec 8.0 Create CBA (ID: 54665266) — Snowflake SQL + ForEach loop. Child: RF 8.0 Create CBA (ID: 54665276) — Recipe Function with NetSuite SOAP actions, conditional branching by CBA type, error handling with try/catch.

Target Layer: NetSuite

The final destination where Credit Memos and Invoice adjustments are created. These records flow into Intercom's financial reporting, revenue recognition, and audit trail.
NetSuite SOAP API actions: search_object_v2 (Customer, Invoice, Credit Memo), add_object (Create Credit Memo/Invoice), update_object (Update Credit Memo/Invoice). Saved searches used for transaction lookup.

7. Processing Stages

Stage 1: Ingestion (Parent Recipe)

The system queries Snowflake to find all customer balance adjustments that have not yet been synchronized to NetSuite. It joins the CBA table with customer data to get subsidiary and NetSuite internal IDs, and checks the log table to exclude already-processed records.
SQL query with CTE stg_workato_log performs a ROW_NUMBER() window to get the latest log per external_id. Main query LEFT JOINs NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS with NETSUITE_EXPORT_CUSTOMER (on entity = external_id) and the log CTE. Filter: created ≥ date_param AND wl.netsuite_internal_id IS NULL. Limit: 50,000 rows.

Stage 2: Customer Validation (Child Recipe Steps 2-5)

Before creating any financial records, the system ensures the customer exists in NetSuite. If the customer is missing (perhaps newly created in Stripe), it automatically creates them by calling the Customer Master recipe.
Step [2]: netsuite.search_object_v2 searches Customers by External ID. Step [3]: IF customer has no sales team (size = 0), call Rec 1.01 Customer Master (flow_id: 54665259). Step [5]: Ruby code for concurrency wait (rate limiting).

Stage 3: Type-Based Routing (Child Recipe Step 6)

Each CBA record is classified by its type, and the system takes a different path for each. This is similar to how a mail sorting machine routes letters to different bins based on the destination.
Compound IF/ELSIF chain on parameters.Type: Step [6] checks == "adjustment", Step [44] checks == "invoice_too_small", Step [57] checks == "applied_to_invoice". Each branch has sub-conditions on Amount sign and invoice existence.

Stage 4: Record Creation (Child Recipe Steps 8-89)

This is where the actual NetSuite records are created: Credit Memos for credits, Invoices for debits, or updates to existing records. For complex cases (like applying credits to multiple invoices), the system uses a loop to allocate amounts correctly.
NetSuite SOAP actions: add_object for Credit Memo/Invoice creation, update_object for CM application. Repeat loops (Steps 32-38, 72-77) iterate through credit memos/invoices tracking RemainingInvoiceAmount, RemainingCMBalance, and RemainingAmountToApply variables. Saved search (Step 81) used as fallback for transaction lookup.

Stage 5: Logging & Deduplication (Every Branch)

Every outcome — success or failure — is recorded in a central log. This ensures that the same CBA is never processed twice, and provides an audit trail for troubleshooting.
After each NetSuite action, workato_mapper.map creates a NetSuite_Integration_Log payload, followed by logger.log_message. The log data feeds into WORKATO_LOGS table in Snowflake (via a separate logging recipe). The parent recipe's dedup check uses this log on subsequent runs.

9. Key Business Logic

CBA Type Classification

The CBA type is determined upstream in the Snowflake staging model based on Stripe's customer_balance_transaction.type field. The recipe handles five distinct types:

TypeMeaningTypical Scenario
adjustmentDirect balance changeSupport credits, billing corrections, contract amendments
invoice_too_smallInvoice amount < applied balanceWhen a customer's credit exceeds the invoice amount
applied_to_invoiceBalance applied against an invoiceAuto-application of credits to new invoices
credit_noteFormal Stripe credit noteRefund-equivalent credit issued to customer
unapplied_from_invoiceReversal of a previous applicationCredit un-applied after invoice voided

Credit Memo Creation Logic

For negative adjustments: The recipe searches for the customer's most recent open invoice in NetSuite and creates a Credit Memo applied against it. If the creation fails (e.g., duplicate), it searches for an existing Credit Memo with the same external ID and logs it as already synced.

For invoice_too_small: Similar to negative adjustments, but the recipe searches for the specific Stripe invoice (by external ID) rather than the most recent one.

Invoice + Credit Memo Application Logic

For positive adjustments (debit): When Amount ≥ 0 or no matching invoice is found, the recipe creates a new standalone Invoice in NetSuite, then searches for any existing unapplied Credit Memos for that customer. It loops through each CM, applying them to the new Invoice until the invoice is fully covered or all CMs are exhausted. Variables RemainingInvoiceAmount and RemainingCMBalance track allocation.

Applied-to-Invoice Logic

When the type is applied_to_invoice, two sub-paths exist:

  • Amount < 0 and item 496 found: Directly update the existing Invoice (simple case)
  • Amount < 0 and no item 496: Search for matching invoices, then search for Credit Memos, and apply CMs to invoices in a loop with balance tracking across five variables: RemainingAmountToApply, RemainingInvoiceBalance, RemainingCMBalance, CBACreditMemos, Invoices
  • Fallback: If no matching invoices found via search, execute a NetSuite Saved Search for the transaction and log the result

Customer Creation Fallback

If the NetSuite customer search returns a customer with no sales team (size = 0), the recipe calls Rec 1.01 Customer Master (ID: 54665259) to ensure the customer record is fully set up before proceeding. This handles cases where Stripe customers were created but not yet synced to NetSuite.

11. Edge Cases & Special Scenarios

Missing Customer in NetSuite

What happens: If a Stripe customer has never been synced to NetSuite, the customer search returns no sales team. The recipe automatically calls Rec 1.01 Customer Master to create the customer before proceeding with the CBA.

Impact: Adds a small delay to processing but ensures no CBAs are skipped due to missing customers.

Duplicate Credit Memo (Already Exists)

What happens: If creating a Credit Memo fails (Step [13] or [48] catch block), the recipe searches for an existing Credit Memo with the same external ID. If found, it logs the existing NS internal ID and stops — treating it as already synced.

Impact: Prevents duplicate financial records. The stop ensures no further processing occurs for this record.

No Matching Invoice Found

What happens: For adjustment type with Amount < 0, if no open invoice is found (Step [25] check), the recipe takes an alternate path: creates a new Invoice first, then applies existing Credit Memos to it.

For applied_to_invoice: If no invoices found via standard search, the recipe falls back to a NetSuite Saved Search (Step [81]) to find the transaction by other criteria.

Negative / Zero Amounts

Amount < 0: Treated as a credit. Creates a Credit Memo or applies credits to invoices.

Amount > 0: Treated as a debit. Creates an Invoice and links available Credit Memos.

Amount = 0: Filtered out at the Snowflake level (not included in the staging table query).

Credit Memo / Invoice Allocation Loop

What happens: When applying Credit Memos to an Invoice (or vice versa), the recipe uses a REPEAT loop with WHILE condition. It tracks remaining balances and iterates until either the full amount is allocated or all available records are exhausted.

Variables tracked: RemainingInvoiceAmount, RemainingCMBalance, RemainingAmountToApply, CBACreditMemos, Invoices.

Concurrency / Rate Limiting

What happens: Multiple steps include "Concurrency Wait" Ruby code blocks (Steps [5], [9], [24], [27], [36], [46], [61], [67]). These introduce small delays to avoid overwhelming the NetSuite SOAP API with concurrent requests.

12. Operational Notes

Deduplication

Deduplication is achieved through a two-layer mechanism:

  1. Query-level: The parent recipe's SQL query LEFT JOINs with WORKATO_LOGS and filters for wl.netsuite_internal_id IS NULL, ensuring only un-synced records are processed.
  2. Action-level: If a Credit Memo creation fails with a duplicate error, the catch block searches for the existing record and logs it, preventing re-processing on subsequent runs.

Error Handling

The recipe uses try/catch blocks at two levels:

  • Outer try/catch (Steps 1/90): Catches any unhandled error in the entire child recipe flow. Maps the error to the integration log and writes it with the Logger.
  • Inner try/catch (Steps 12/14, 28/39, 47/49): Catches specific NetSuite action failures (e.g., duplicate CM). Handles gracefully by searching for existing records or logging the error.

All errors are written to WORKATO_LOGS with error_type and error_message fields, making them queryable in Snowflake for monitoring.

Key Limitations

  • The recipe processes up to 50,000 rows per run. If the backlog exceeds this, multiple runs are needed.
  • NetSuite SOAP API rate limits are managed via Ruby concurrency waits, but high-volume days may still encounter throttling.
  • The date parameter must be provided by the caller (scheduler recipe). If not set correctly, records may be missed or re-processed.
  • Item 496 check in the applied_to_invoice branch is hardcoded — changes to this NetSuite item would require recipe updates.
