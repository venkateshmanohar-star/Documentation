1. EXECUTIVE SUMMARY

1. Executive Summary

Payroll-WD-Netsuite-v2 is a Workato integration recipe that automates the transfer of US payroll general ledger data from Workday to NetSuite. When Workday triggers a payroll run, it sends an XML payload via webhook containing GL line items (debits, credits, accounts, departments, locations, and pay components). The recipe parses this data, applies account-mapping and department-reclassification business rules, aggregates the results using Workato Smart Lists, and creates a Journal Entry in NetSuite. Slack notifications are sent on both success and failure.

One-line summary: Receives Workday payroll GL XML via webhook, applies account/department reclassification rules, aggregates line items, and posts a consolidated Journal Entry to NetSuite with Slack alerts.

2. BUSINESS OVERVIEW

2. Business Overview

Purpose

Intercom processes US payroll through Workday. Each payroll cycle generates a General Ledger report containing hundreds of line items across accounts, departments, and pay components. This integration eliminates manual journal entry creation by automatically transforming Workday's payroll GL data into a NetSuite Journal Entry, ensuring timely and accurate financial close.

Business Context

  • Stakeholders: Finance/Accounting team, Payroll team, FP&A
  • Business Process: Semi-monthly or biweekly payroll processing for US employees
  • Value: Eliminates manual data entry of payroll journal entries, reduces risk of human error, and accelerates month-end close
  • Subsidiary: 10 Intercom US (hardcoded subsidiary = 1 in NetSuite)

Key Data Elements

  • Payment Date: The payroll payment date drives the JE transaction date
  • GL Accounts: Account codes (e.g., 2245, 2246, 6150, 9999) map to NetSuite chart of accounts
  • Departments: Cost center codes (e.g., 9101 IT, 61xx-64xx R&D)
  • Locations: Employee work locations (e.g., US CA Home Office)
  • Pay Components: Payroll elements (wages, taxes, benefits, deductions)

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

The integration follows a linear pipeline architecture: Receive → Parse → Transform → Aggregate → Load → Notify. The recipe is event-driven, triggered by Workday's payroll delivery webhook. All processing occurs within a single Workato recipe with no child recipes or callable flows.

Design Principles

  • Single recipe: No child recipes or sub-flows; all logic contained in one recipe
  • Idempotent aggregation: Smart List SQL groups and sums line items to produce clean JE lines
  • Lookup-driven mapping: Account, Location, and Department IDs resolved via Workato lookup tables at JE creation time
  • Try/Catch pattern: Entire processing wrapped in error handling with Slack alerts
  • Conditional routing: 6-branch if/elsif/else logic for pay component and account reclassification

6. END-TO-END FLOW

6. End-to-End Flow

  1. Webhook Trigger: Workday sends an HTTP POST with a SOAP XML payload containing the payroll GL report for a specific payment date.
  2. Log Raw Payload: The raw webhook body is logged for debugging.
  3. Parse XML: The SOAP envelope is parsed using the XML Parser action, extracting structured report entries.
  4. Extract Payment Date: The Payment_Date_Prompt field is extracted and stored in a payment_date variable.
  5. Create Workday-Report-Input List: A structured list schema is created for staging parsed data (Account_Name, Debit, Credit, memo, Class, Currency, Department, Location, pay_component).
  6. Populate Initial List: All report entries are batch-inserted into the Workday-Report-Input list with their raw field values.
  7. Try Block Begins: Core processing wrapped in error handling.
  8. Declare Lists: Netsuite-Input (Account, Debit, Credit, Department, Class, Location) and a fresh Workday-Report-Input list are declared.
  9. Foreach Loop: Each report entry is iterated through a 6-branch conditional router:
    • Branch 1 (WA Industrial Insurance): Reclassifies Washington Industrial Insurance entries on account 2245 to account 2242 with memo "Washington Industrial"
    • Branch 2 (Benefits/Clearing): Matches specific pay components (Parking, Transit, FSA) or clearing accounts (2245, 2246, 2252, 2253, 2242); uses lookup table for account mapping; flagged for removal (aggregation only)
    • Branch 3 (R&D Departments): Departments starting with 61, 62, 63, 64 get special account extraction
    • Branch 4 (Account 9999): Reclassifies entries with account starting with 9999 via lookup table
    • Branch 5 (Employee Taxes + 6150): Employee Taxes with account 6150 get department hardcoded to 1101
    • Branch 6 (Default): All other entries pass through with standard field extraction
  10. Log Transformed Data: The processed list items are logged.
  11. Smart List Insert: Transformed data is inserted into the "Workday-Report" Smart List table.
  12. Smart List SQL Aggregation: A UNION ALL query runs three aggregation passes:
    • Pass 1: Flagged-for-removal rows grouped by memo only (clearing accounts)
    • Pass 2: Non-removal debit=0 rows grouped by Department, Location, Account, memo
    • Pass 3: Non-removal credit=0 rows grouped by Department, Location, Account, memo
  13. Log Aggregated Results: The Smart List query results are logged.
  14. Create NetSuite Journal Entry: A JournalEntry is created with:
    • Subsidiary: 1 (Intercom US)
    • Transaction Date: payment_date from Workday
    • Memo: "{payment_date}-US Workday Payroll"
    • Line items from Smart List results, with account/location/department/class resolved via Payroll-Netsuite-Lookup
  15. Success Check: If the JE internal_id is present, a success Slack message is posted.
  16. Catch Block: On any error, an error notification is posted to Slack.

9. KEY BUSINESS LOGIC

9. Key Business Logic

6-Branch Account Reclassification Router

The core business logic routes each Workday GL line through a 6-branch conditional to apply account reclassification and department overrides before aggregation.

BranchConditionActionRemove Flag
1. WA Industrial Insurance pay_component starts with "WA: Washington Industrial Insurance" AND Account_Name starts with "2245" Reclassify account to 2242, set memo to "Washington Industrial" true
2. Benefits / Clearing pay_component in [Parking, Transit, FSA-Dependent, FSA-Health] OR Account_Name starts with [2245, 2246, 2252, 2253, 2242] Account resolved via Workday_Payroll_Netsuite_Lookup project property true
3. R&D Departments Department starts with 61, 62, 63, or 64 Account extracted from Account_Name (split). Standard field mapping. false
4. Account 9999 Account_Name starts with "9999" Account resolved via Workday_Payroll_Netsuite_Lookup false
5. Employee Tax 6150 memo = "Employee Taxes" AND Account_Name starts with "6150" Department hardcoded to 1101; account extracted from Account_Name false
6. Default All other entries Account extracted from Account_Name. Standard field mapping with department lookup. false

Remove Flag and Aggregation Logic

The Remove flag (set to "true" in branches 1 and 2) controls how lines are aggregated in the Smart List SQL query:

  • Remove = true (Clearing accounts): Grouped by memo only. Department, Class, and Location are blanked out. These represent liability/clearing entries that get consolidated.
  • Remove = false, Debit = 0: Grouped by Department, Location, Account, and memo. These are credit entries with full dimensional breakdowns.
  • Remove = false, Credit = 0: Grouped by Department, Location, Account, and memo. These are debit entries with full dimensional breakdowns.

Lookup Table Resolution

At JE creation time, the following fields are resolved via the Payroll-Netsuite-Lookup lookup table:

  • Account: lookup('Payroll-Netsuite-Lookup', Object: "Account", Key: {account_code})
  • Location: lookup('Payroll-Netsuite-Lookup', Object: "Location", Key: {Updated_Location})
  • Department: lookup('Payroll-Netsuite-Lookup', Object: "Department", Key: {department_code})
  • Class: lookup('Payroll-Netsuite-Lookup', Object: "Department", Key: {Class}) (note: uses Department object type)

Department Extraction

Department codes in Workday come as compound strings (e.g., "9101 IT"). The recipe splits on space and takes the first segment (numeric code) for mapping. For R&D departments (61xx-64xx), a special check determines if the department code includes certain patterns.

Currency Handling

The Base_Currency field from Workday (e.g., "USA") is passed through. The Workday report is filtered to US GAAP (USD) accounting book entries only.

11. EDGE CASES

11. Edge Cases

Washington Industrial Insurance Reclassification: When pay_component starts with "WA: Washington Industrial Insurance" AND account is 2245, the account is reclassified to 2242 with a static "Washington Industrial" memo. This is an AND condition -- both must match. If WA insurance appears on a different account, it falls to a later branch.
Benefits/Clearing Account Detection: Branch 2 uses an OR condition combining specific pay components (Parking, Transit, FSA-Dependent, FSA-Health) with account prefix matching (2245, 2246, 2252, 2253, 2242). This branch sets Remove=true, meaning these lines get aggregated by memo only (no dept/loc breakdown).
Account 9999 Catch-all: Account code 9999 is a Workday clearing/suspense account. These entries are reclassified via the lookup table rather than using the Workday account directly.
Employee Taxes with 6150: When memo = "Employee Taxes" and account starts with 6150, the department is hardcoded to 1101. This overrides whatever department Workday provides. This handles a specific tax expense allocation requirement.
Debit/Credit Separation in Aggregation: The Smart List SQL separates non-removal entries into two queries: one for debit=0 rows (credit entries) and one for credit=0 rows (debit entries). This prevents netting of debits and credits within the same grouping.
Comma-stripped Amounts: All Debit and Credit values are processed with .to_s.gsub(',', '').to_f to handle Workday's comma-formatted numbers (e.g., "1,441,131.19" becomes 1441131.19).
Class Lookup uses Department Object: The Class field's NetSuite internal ID is resolved via lookup('Payroll-Netsuite-Lookup', Object: "Department", Key: {Class}), which uses the "Department" object type. This is likely intentional (shared lookup table structure) but could be confusing during maintenance.
No Retry on JE Creation: The NetSuite JE creation step has max_retry_count: 0. If the JE creation fails, it goes directly to the catch block with a Slack error notification. There is no automatic retry.

  • Pay Component Changes: New benefit types that should be treated as clearing accounts must be added to Branch 2's inclusion list.
