Section 1: Executive Summary

1. Executive Summary

This Workato recipe synchronizes Account Owner and AE Champion contact information from Salesforce (SFDC) to the Stripe Ireland (IE) billing instance in real time. When an account record is created or updated in Salesforce, the recipe is triggered via a Platform Event, searches Stripe IE for matching customers by app_id_code, and updates the Stripe customer's metadata fields with the latest SFDC account owner and AE champion details.

Section 2: Business Overview

2. Business Overview

Business Context

Intercom operates multiple Stripe instances across regions. The Stripe Ireland (IE) instance serves EMEA and international customers. Sales and customer success teams manage account ownership in Salesforce, and this ownership data must be reflected in Stripe to ensure:

  • Billing Accuracy: Invoices and payment communications reach the correct account owner.
  • Revenue Attribution: Revenue is correctly attributed to account owners and AE champions in downstream reporting.
  • Operational Visibility: Finance and RevOps teams can trace Stripe customers back to their SFDC account owners without manual lookups.
  • Compliance: Data consistency across CRM and billing systems supports audit requirements.

Business Rules

  • Only processes events where SourceType__c = 'StripeSync' on the Salesforce Platform Event.
  • If the SFDC account has no assigned owner (AccountOwnerId__c is blank), the job logs the condition, publishes an error notification, and stops gracefully.
  • If no matching Stripe IE customer is found for the given AppIdCode__c, the job logs and stops without error.
  • If the Stripe customer's existing metadata already matches the SFDC values, no update is performed (skip-if-unchanged optimization).
  • Both Account Owner and AE Champion fields are synced in a single update call to Stripe.

Section 4: Architecture Overview

4. Architecture Overview

This integration follows an event-driven architecture pattern. Salesforce publishes a Platform Event (Workato__e) whenever an account is created or updated with ownership changes relevant to Stripe. Workato subscribes to this event, validates the payload, searches Stripe IE for the corresponding customer, and conditionally updates the Stripe customer's metadata. All errors and audit events are broadcast to a shared Pub/Sub topic for centralized monitoring.

Key Architectural Decisions

  • Platform Event over Outbound Message: Uses SFDC Platform Events for reliable, ordered delivery with replay capability.
  • REST over Stripe Connector: Uses the generic REST connector to call Stripe API directly, allowing full control over metadata field mapping.
  • Idempotent Updates: Compares current Stripe metadata with incoming SFDC values and skips if unchanged, reducing unnecessary API calls.
  • Error Isolation: Each customer in the loop is wrapped in try/catch with a forced loop-break on error, preventing one bad record from blocking others.
  • Pub/Sub for Error Broadcasting: All errors publish to Topic 37790, enabling centralized alerting and downstream error-handling recipes.

Section 6: End-to-End Flow

6. End-to-End Flow

1 SFDC Platform Event Fires

When an account is created or updated in Salesforce with ownership changes, SFDC publishes a Workato__e platform event containing account owner, AE champion, and app identifier fields.

2 Event Filter

Workato receives the event and applies the trigger filter: SourceType__c = 'StripeSync'. Only events targeting Stripe synchronization proceed.

3 Account Owner Validation

Checks if AccountOwnerId__c is present. If blank, logs a message ("Account record has not been assigned"), publishes an error event to Topic 37790, and stops the job gracefully.

4 Stripe Customer Search

Calls Stripe IE API: GET v1/customers/search?query=metadata['app_id_code']:'{AppIdCode__c}' to find matching Stripe customer(s) by app ID code.

5 Match Validation

If zero results returned from Stripe search, logs the miss, publishes to Topic 37790, and stops gracefully. If one or more customers found, proceeds to the iteration loop.

6 For Each Stripe Customer

Iterates over each matched Stripe customer. For each record: accumulates the record into an audit list, then checks if the existing Stripe metadata already matches the incoming SFDC values.

7 Conditional Update

If metadata differs, calls POST v1/customers/{id} with updated metadata fields (7 fields). If metadata already matches, logs "No updates necessary" and skips. Each update is wrapped in try/catch for error isolation.

8 Audit & Notification

On success or failure, publishes detailed event to Pub/Sub Topic 37790 with transaction type, identifiers, and outcome details for centralized monitoring.

Section 9: Key Business Logic

9. Key Business Logic

9.1 Trigger Filter Logic

The recipe only processes Salesforce Platform Events where:

SourceType__c == "StripeSync"

This ensures the recipe ignores events intended for other sync targets (e.g., NetSuite, other Stripe instances).

9.2 Owner Assignment Guard

If the SFDC account has no owner assigned (i.e., AccountOwnerId__c is blank), the recipe:

  1. Logs: "Account record has not been assigned. Stopping job."
  2. Publishes error event to Topic 37790 with transaction type "SFDC Account Owner Email"
  3. Stops the job gracefully (no error raised)

9.3 Stripe Customer Lookup

The recipe searches Stripe IE using the customer search API with the app_id_code metadata field:

GET v1/customers/search?query=metadata['app_id_code']:'{AppIdCode__c}'

This relies on the Stripe customer having been previously created with an app_id_code metadata field matching the SFDC account's AppIdCode__c.

9.4 Idempotent Update (Skip-if-Unchanged)

Before updating, the recipe compares the current SFDC account owner email with the existing Stripe metadata:

IF AccountOwnerEmail__c == stripe_customer.metadata.sfdc_account_owner_email
   THEN skip (log "already matches")
   ELSE perform update

This is a key optimization that reduces unnecessary Stripe API calls when the data has not actually changed.

9.5 Metadata Fields Synced

The following 7 metadata fields are written to the Stripe customer record on each update:

Stripe Metadata KeySFDC Source FieldDescription
metadata[sfdc_account_id]AccountId__cSFDC Account record ID
metadata[sfdc_account_name]AccountName__cAccount name from SFDC
metadata[sfdc_account_owner_id]AccountOwnerId__cSFDC user ID of account owner
metadata[sfdc_account_owner_email]AccountOwnerEmail__cEmail of the account owner
metadata[sfdc_account_owner_name]AccountOwnerName__cFull name of the account owner
metadata[sfdc_ae_champion_id]AEChampion__cSFDC user ID of AE champion
metadata[sfdc_ae_champion_email]AEChampionEmail__cEmail of the AE champion

9.6 Error Isolation Pattern

The inner loop uses a try/catch with forced stop pattern: if a Stripe update fails for one customer, the error is caught, published to Pub/Sub, and then a stop with stop_with_error = true forces the loop to break and proceed to the next record. This prevents one bad record from blocking the entire batch.

Known Consideration: The change-detection logic only compares AccountOwnerEmail__c against sfdc_account_owner_email. If only the AE Champion changes (without an owner email change), the recipe will detect it as "unchanged" and skip the update. This is a potential gap if AE Champion changes independently of account owner.

12.4 Retry and Recovery

  • Max Retry Count: 0 (no automatic retries on Stripe API errors within try/catch blocks)
  • Retry Interval: 2 seconds (configured but not actively used since max retries = 0)
  • Recovery Process: Failed jobs should be investigated via the Pub/Sub Topic 37790 error messages. Manual re-trigger may be needed for persistent failures.

12.5 Dependencies

  • Salesforce Workato__e Platform Event must be deployed and active in the SFDC org.
  • SFDC triggers/flows that publish Workato__e events must include all required fields with SourceType__c = 'StripeSync'.
  • Stripe IE customers must have app_id_code in their metadata (set during customer creation).
  • Workato REST connection to Stripe IE must have valid API credentials with customer read/write permissions.
  • Topic 37790 must exist and have downstream subscribers for error monitoring.

12.6 Maintenance Recommendations

Recommendation 1: Consider adding AE Champion email to the change-detection comparison (currently only compares account owner email). This would catch AE-only changes that are currently skipped.
Recommendation 2: Enable at least one retry (max_retry_count = 1) on the Stripe API update try/catch blocks to handle transient network errors without requiring manual intervention.
