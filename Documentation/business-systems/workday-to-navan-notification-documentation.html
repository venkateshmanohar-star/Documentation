Workday to Navan Notification

Automated daily data quality check between Workday and Navan

Recipe ID: 59686983 Status: Active Version: 13 Author: venkatesh.manohar Last Updated: April 15, 2026 Recipe Author: diana.sy

1. Executive Summary

This Workato recipe automates a daily data quality check between Workday (HR system of record) and Navan (travel & expense management). Every day at 1:00 AM PT, it pulls the full employee roster from a custom Workday report, identifies employees with missing or null fields critical to Navan sync (Cost Allocation Segment, Location, Cost Center), and sends an email notification to the Finance / Accounts Payable team listing the affected employees so they can remediate the data in Workday before it causes Navan sync failures.

2. Business Overview

Business Problem

Navan requires accurate employee data -- including cost allocation segment, location, and cost center -- to properly classify travel and expense transactions for financial reporting and GL coding. When these fields are missing or null in Workday, the downstream Navan sync either fails silently or misclassifies expenses, creating reconciliation headaches for the Finance team.

Business Value

  • Proactive data quality -- catches missing Workday fields before they cause Navan sync issues
  • Reduced manual effort -- automates what was previously a periodic manual audit
  • Faster remediation -- daily notification gives the People Ops and Finance teams a clear action list
  • Financial accuracy -- ensures expense classifications align with cost center and subsidiary structures

4. Architecture Overview

The integration follows a simple Extract-Transform-Notify pattern:

  1. Extract -- Pull the full employee roster from Workday via a custom report URL
  2. Parse -- Convert the JSON response into a structured dataset
  3. Store -- Load all employees into a Workato Smart List (in-memory collection)
  4. Query -- Run a SQL-like query to find employees with null Cost Allocation Segment, Location, or Cost Center
  5. Notify -- If any employees are found, format an HTML email from a template and send to the Finance / AP team

6. End-to-End Flow

Flow Summary

1
Daily Trigger (1:00 AM PT)
The Workato clock trigger fires every 24 hours at 1:00 AM Pacific Time, initiating the data quality check process.
2
Fetch Workday Custom Report
Calls the Workday REST API to execute the custom report Workato_Integration_Report_-_Navan authored by ISU_Workato. Returns JSON containing all employees with 27+ fields including email, name, location, cost center, subsidiary, and manager.
3
Parse JSON Response
The JSON parser action converts the raw Workday API response into structured objects, mapping each field (Cost_Allocation_Segment, Email_-_Work, location, Cost_Center, etc.) to typed columns.
4
Log Raw Data (Debug)
Logs the parsed JSON output to the Workato job report for debugging purposes.
5
Load into Smart List
Creates (or overwrites) an in-memory Smart List called Employee_Collection indexed by Employee_ID. This serves as a queryable database for the next step.
6
Query for Missing Data
Runs a SQL-like query against the Smart List to find employees where Cost_Allocation_Segment IS NULL, location IS NULL, or Cost_Center IS NULL. Returns Email, Class, Location, CostCenter, and HireDate for matching rows.
7
Conditional Check
If the query returns zero rows (no missing data), the recipe ends without sending email. If rows exist (data issues found), continue to notification.
8
Format HTML Email
Uses Workato Template ID 96282 ("Workday <> Navan Notification") to generate an HTML email body. The template iterates over the list of employees with missing data, rendering each employee's Email, Class, Location, CostCenter, and HireDate.
9
Send Email Notification
Sends the HTML email to diana.sy@intercom.io, nan.lu@intercom.io, accountspayable@intercom.io, and david.africa@intercom.io with subject: "Navan Employee Sync Notification - [today's date]".

9. Key Business Logic

Data Quality Validation Rules

The core business logic centers on three mandatory fields that must be populated in Workday for proper Navan synchronization:

FieldWorkday ColumnNavan UsageBusiness Impact if Missing
Cost Allocation Segment Cost_Allocation_Segment Maps to "Class" in Navan for expense classification Expense reports cannot be properly classified by business segment (e.g., Core, Growth)
Location location Employee location for travel policy assignment Incorrect travel policies applied; potential tax/compliance issues
Cost Center Cost_Center GL coding for expense allocation Expenses booked to wrong cost center; budget tracking errors

Conditional Logic

Rule: Email notification is ONLY sent when at least one employee has missing data. This prevents alert fatigue from daily emails when all data is clean.

IF query_result.rows.size != 0 THEN
    generate_template(affected_employees)
    send_email(to: finance_team)
ELSE
    // Recipe completes silently -- no notification
END

Smart List Overwrite Strategy

The Smart List is created with force: true, meaning each daily run completely replaces the previous day's data. This is a full-refresh pattern (not incremental), ensuring the query always operates on the latest Workday data without risk of stale records.

Email Subject Line Pattern

Subject follows the format: "Navan Employee Sync Notification - " + today()

Example: Navan Employee Sync Notification - 2026-04-15

This date-stamped subject enables recipients to track which day's data quality issue each email corresponds to and easily search their inbox for specific dates.

11. Edge Cases

#Edge CaseCurrent HandlingRisk Level
1 Workday API downtime -- Report endpoint returns error or timeout Recipe job fails. No notification sent. Workato retries per platform settings. Medium
2 All employees have complete data -- Query returns 0 rows IF condition evaluates false, recipe ends silently. No email sent. This is expected behavior. Low
3 Terminated employees with missing data -- Inactive employees have null fields No active/terminated filter in the query. Terminated employees with null fields will appear in the notification, potentially creating noise. Medium
4 Workday report schema change -- Fields renamed or removed JSON parser would fail or return null for changed fields. Recipe may fail or produce incorrect results. High
5 Email delivery failure -- SMTP issues or invalid recipients Workato platform handles retry. Job would show as failed. Low
6 Empty Workday report -- Report returns 0 employees Smart List created with 0 rows. Query returns 0. No email. Could mask an API issue. Medium
7 Large number of missing records -- Hundreds of employees with null fields Email may be very long but will still be generated. No pagination or truncation in the template. Low
8 Deprecated field names -- Report includes both old and new field naming conventions The report includes deprecated fields (e.g., Email_-_Work) alongside new ones (Email___Work). The query uses the deprecated naming convention through the Smart List column references. Medium
Recommendation: Consider adding an Active_Status = '1' filter to the Smart List query to exclude terminated employees from the notification. This would reduce noise and focus the Finance team on actionable records.

Monitoring & Alerting

  • Job failures are visible in the Workato job history dashboard
  • No external monitoring is configured (no Datadog, PagerDuty, etc.)
  • Failed jobs appear to be intermittent (13 out of 391) and do not require manual intervention
  • The recipe logs raw data to the job report (Step 3) which aids debugging

Maintenance Notes

  • Workday Report: The custom report Workato_Integration_Report_-_Navan is maintained by ISU_Workato in Workday. Any changes to the report schema require corresponding updates to the JSON parser and Smart List configuration.
  • Email Recipients: To add or remove notification recipients, update the to field in Step 8 (Send Email action).
  • Template: The HTML email template (ID: 96282) is managed in Workato's template library. Changes to the notification format should be made there.
  • Deprecated Fields: The Workday report includes both deprecated and current field naming conventions. A future cleanup should standardize to the current convention (triple underscore pattern).
