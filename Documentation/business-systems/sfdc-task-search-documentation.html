[FUNCTION] Most Recent Salesforce Task Search by Lead/Contact (Last N days)

Reusable Workato function recipe that queries Salesforce for the most recent Task activity associated with a Lead or Contact within a configurable time window.
Recipe ID: 39468877 Status: Running Author: monette.sommers Doc by: venkatesh.manohar Generated: 2026-04-15

1. Executive Summary

This Workato function recipe (callable recipe) provides a reusable service that searches Salesforce Task records associated with a given Lead or Contact ID within a configurable time window (last N days). It identifies the most recent Task, resolves the Task assignee's user details (email, full name), and returns a structured result payload to the calling recipe.

The recipe is designed as a shared utility within Intercom's go-to-market automation stack, enabling parent recipes to determine whether recent sales engagement activity exists for a given Lead or Contact before taking downstream actions such as lead routing, enrichment, or handoff decisions.

2. Business Overview

Business Purpose

In Intercom's sales and marketing operations, determining whether a Lead or Contact has been recently engaged by a sales rep is critical for:

  • Lead routing decisions -- Preventing re-assignment of Leads that are already being actively worked by an SDR/AE
  • Duplicate engagement prevention -- Avoiding multiple reps reaching out to the same prospect simultaneously
  • Sequence attribution -- Identifying which Outreach sequence (if any) generated the most recent Task touchpoint
  • SLA compliance -- Verifying that Leads are being followed up within required timeframes

4. Architecture Overview

This recipe follows a callable function pattern -- it is not triggered by external events but is invoked synchronously by parent recipes that need to determine recent Salesforce Task activity for a given Lead or Contact.

Architecture Pattern: Callable Function with Smart List Processing

The recipe employs a four-phase architecture:

  1. Input Reception -- Accepts sfdc_record_id (Lead/Contact ID) and last_n_days (search window) from the calling recipe
  2. Data Retrieval -- Executes a parameterized SOQL query against the Salesforce Task object
  3. In-Memory Processing -- Loads results into a Workato Smart List and runs a SQL query to identify the most recent Task by CreatedDate
  4. Enrichment & Response -- Looks up the Task assignee's User record in Salesforce, then returns a structured result payload
Design Decision: The Smart List pattern is used instead of SOQL ORDER BY / LIMIT because it allows more flexible in-memory querying (e.g., MAX aggregation with JOIN) within the Workato environment, decoupling the retrieval from the analysis step.

7. Processing Stages

Stage 1: Input Reception

ParameterTypeRequiredDescription
sfdc_record_idStringYesSalesforce Lead or Contact ID (18-char) to search for matching Tasks
last_n_daysIntegerYesSearch window in days (e.g., 30 = last 30 days from execution time)

Stage 2: SOQL Task Retrieval (Step 1)

Executes the following SOQL query against Salesforce:

SELECT Id, WhoId, WhatId, OwnerId, CreatedDate, CompletedDateTime,
       Outreach_Attributed_Sequence_Name__c
FROM Task
WHERE WhoId = ':sfdc_record_id'
  AND CreatedDate = LAST_N_DAYS::last_n_days
LIMIT 150

Fields retrieved:

  • Id -- Activity/Task ID
  • WhoId -- Related Lead or Contact ID
  • WhatId -- Related Account or Opportunity ID
  • OwnerId -- Assigned User ID
  • CreatedDate -- Task creation timestamp
  • CompletedDateTime -- Task completion timestamp
  • Outreach_Attributed_Sequence_Name__c -- Outreach sequence attribution (custom field)

Stage 3: Variable Initialization (Step 2)

Creates two runtime variables:

  • found_matching_task (Boolean) -- defaults to nil
  • number_of_matching_tasks (Integer) -- defaults to nil

Stage 4: Conditional Branch (Step 3)

Evaluates whether the SOQL query returned any results:

  • Condition: Task list size = 0 OR Task list size is blank
  • IF TRUE (Step 4): Sets found_matching_task = false and number_of_matching_tasks = 0
  • ELSE (Steps 6-9): Proceeds to Smart List processing

Stage 5: Smart List Processing (Steps 6-7)

Creates an in-memory list indexed by Task Id, then runs the following SQL query:

WITH
    base AS (
        SELECT * FROM Tasks
    ),
    get_max AS (
        SELECT MAX(CreatedDate) AS most_recent_task
        FROM base
    )
SELECT
    b.*,
    CASE WHEN b.CreatedDate = gm.most_recent_task THEN 1
         ELSE 0
    END AS latest_task
FROM base AS b
JOIN get_max AS gm ON b.CreatedDate = gm.most_recent_task;
Note: The JOIN condition b.CreatedDate = gm.most_recent_task means that if multiple Tasks share the same most-recent CreatedDate, all of them will be returned. The recipe uses .first to select the first match for the response payload.

Stage 6: User Enrichment (Step 8)

Queries the Salesforce User object to resolve the Task assignee:

SELECT Id, Email, Name
FROM User
WHERE Id = ':OwnerId'
LIMIT 1

Stage 7: Result Assembly (Steps 9-10)

Updates the variables and returns the complete result payload to the calling recipe.

9. Key Business Logic

Most Recent Task Selection

The core business logic centers on identifying the most recently created Salesforce Task for a given Lead/Contact within the specified time window. The selection algorithm:

  1. Retrieve all Tasks matching the Lead/Contact ID within the LAST_N_DAYS window (up to 150 records)
  2. Load into Smart List and compute MAX(CreatedDate) across all results
  3. JOIN back to the full result set to return only rows matching the max date
  4. Select .first from the result set for the response payload

Outreach Sequence Attribution

The recipe captures the Outreach_Attributed_Sequence_Name__c custom field, which links the Task to an Outreach sales engagement sequence. This enables calling recipes to determine:

  • Whether the most recent activity was part of an automated Outreach sequence
  • Which specific sequence generated the touchpoint
  • Attribution data for sales engagement reporting

Task Assignee Resolution

Rather than returning only the OwnerId (a Salesforce User ID), the recipe performs an additional SOQL lookup against the User object to resolve human-readable assignee details:

  • Email -- Used for routing notifications or matching to downstream systems
  • Full Name -- Used for display in alerts, reports, or Slack notifications

Boolean Found/Not-Found Pattern

The recipe returns a found_matching_task boolean flag, allowing calling recipes to implement simple branching logic:

IF found_matching_task = true
    // Lead has recent activity -- skip routing or defer
ELSE
    // No recent activity -- proceed with routing/assignment

11. Edge Cases & Error Handling

Known Edge Cases

ScenarioBehaviorRisk Level
No Tasks found for the given Lead/Contact Returns found_matching_task = false, number_of_matching_tasks = 0, matching_tasks fields are null Low -- handled by IF branch
Multiple Tasks share the same MAX(CreatedDate) All tied Tasks are returned by the SQL query, but only the .first record is used in the return payload Medium -- arbitrary tie-breaking
More than 150 Tasks in the time window SOQL LIMIT 150 means only the first 150 records are evaluated; the actual most recent Task may be missed if SOQL does not return records in CreatedDate order by default High -- potential data gap
Invalid or non-existent sfdc_record_id SOQL returns 0 results; recipe follows the "not found" path. No error raised. Low -- graceful handling
Task OwnerId references a deactivated User User SOQL query returns the deactivated user record (Salesforce retains deactivated users); Email and Name are still populated Low -- Salesforce behavior
Task has no Outreach sequence attribution Outreach_Attributed_Sequence_Name__c will be null in the response Low -- optional field
Task WhatId is null (no related Account) related_account_id will be null in the response; this is normal for Tasks linked only to a Lead without an Account Low -- expected behavior
Potential Risk: The SOQL query uses LIMIT 150 without an explicit ORDER BY CreatedDate DESC. Salesforce's default sort order for SOQL is not guaranteed. For Leads/Contacts with more than 150 Tasks in the given window, the most recent Task might not be in the returned result set. Consider adding ORDER BY CreatedDate DESC to the SOQL query.
