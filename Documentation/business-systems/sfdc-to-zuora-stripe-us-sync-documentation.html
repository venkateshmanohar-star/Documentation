SECTION 1: EXECUTIVE SUMMARY

1. Executive Summary

SFDC to Zuora and Stripe U.S. Account Owner, Champion Sync is a high-volume, real-time Workato integration recipe that synchronizes Salesforce Account Owner and AE Champion metadata to matching records in both Zuora and Stripe U.S.. Triggered by Salesforce Platform Events, the recipe ensures that billing and payment systems always reflect the current account ownership assigned in the CRM.

With over 2.87 million successful jobs and nearly 14 million lifetime tasks, this is Intercom's highest-volume Workato recipe. It processes every SFDC Account create/update event where SourceType__c = 'StripeSync', looks up matching accounts in Zuora (by CRM ID) and Stripe (by App ID Code), compares owner metadata, and updates records only when a change is detected—avoiding unnecessary writes.

SECTION 2: BUSINESS OVERVIEW

2. Business Overview

Business Purpose

When a Salesforce Account's owner or AE Champion changes, the downstream billing systems (Zuora and Stripe) need to reflect the new ownership. This is critical for:

  • Revenue Operations: Correct account ownership in billing systems ensures accurate sales rep attribution for commission calculations, renewal forecasting, and territory alignment.
  • Customer Success: Stripe metadata containing owner information is used by internal tools to route support queries and surface the right CSM/AE contact for a given customer.
  • Finance/Billing: Zuora's SalesRepEmail__c and SalesRepName fields drive invoice-level attribution and are referenced in revenue recognition workflows.

Business Context

This recipe is part of the broader SFDC-to-Billing Sync family, which keeps CRM account metadata synchronized across Zuora (subscription billing) and Stripe (payment processing). The sync is event-driven—changes in SFDC fire a Platform Event, which Workato consumes in near-real-time.

The recipe operates on both new and existing accounts. When an account is created in SFDC (or its owner changes), the Platform Event payload includes the Account Owner ID, Email, Name, AE Champion, and App ID Code. The recipe then performs a smart comparison: it only writes to Zuora/Stripe when the owner metadata has actually changed, reducing unnecessary API calls by a significant margin.

Stakeholders

  • Revenue Operations — Depends on accurate owner attribution for commission and territory management
  • Sales Leadership — Relies on correct AE Champion metadata for pipeline visibility
  • Finance Operations — Zuora owner fields feed into billing and rev rec processes
  • Systems Integration Team (Enterprise Systems) — Owns and maintains this recipe

SECTION 4: ARCHITECTURE OVERVIEW

4. Architecture Overview

The recipe follows an event-driven, fan-out architecture. A single Salesforce Platform Event triggers parallel lookup and update paths to two independent target systems (Zuora and Stripe). The design uses a compare-before-write pattern to minimize unnecessary API calls—critical given the recipe's 2.8M+ job volume.

Key Architectural Patterns

  • Event-Driven Trigger: Salesforce Platform Events provide near-real-time change detection without polling, ensuring sub-second reaction to account changes.
  • Parallel Target Updates: Zuora and Stripe lookups happen sequentially but are processed independently—a failure in one does not block the other (each is within its own conditional branch).
  • Compare-Before-Write: The recipe checks whether the current owner email in the target system matches SFDC before issuing an update, avoiding redundant writes.
  • Forced-Error Loop Break: For multi-match scenarios (foreach loops), the recipe uses a try/catch with intentional stop with error to break out of loops after processing—a well-known Workato design pattern for early loop termination.
  • Centralized Error Reporting: All error conditions publish structured messages to Pub/Sub Topic 37790, enabling downstream error aggregation and alerting.

SECTION 6: END-TO-END FLOW

6. End-to-End Flow

Flow Summary

  1. Trigger: Salesforce fires a Workato__e Platform Event when an Account record is created or updated. The recipe filters for SourceType__c = 'StripeSync'.
  2. Validation: Check if AccountOwnerId__c is blank. If unassigned, log and stop.
  3. Zuora Lookup: Query Zuora for Account records where CrmId LIKE '%{AccountId}%'.
  4. Stripe Lookup: Search Stripe U.S. customers where metadata['app_id_code'] = '{AppIdCode}'.
  5. Zero Match Guard: If both Zuora and Stripe return 0 results, log and stop.
  6. Zuora Update Branch: For each matching Zuora account, compare owner email. Update only if different or missing.
  7. Stripe Update Branch: For each matching Stripe customer, compare owner email. Update only if different or missing.
  8. Completion: Job stops successfully after processing both branches.

SECTION 9: KEY BUSINESS LOGIC

9. Key Business Logic

9.1 Event Filtering

The trigger filter SourceType__c = 'StripeSync' ensures only account changes originating from the Stripe sync pathway are processed. Other SFDC Platform Events (e.g., those with different SourceType values) are silently ignored at the trigger level, preventing duplicate processing.

9.2 Compare-Before-Write Pattern

The core optimization in this recipe is the compare-before-write pattern. For each target record:

  1. Blank check: If the target field is blank/empty, update immediately (filling missing data).
  2. Equality check: Compare SFDC AccountOwnerEmail with the target's existing owner email. If they match, skip the update entirely.
  3. Mismatch update: Only when the emails differ does the recipe issue an API update call.
Impact: Given 2.8M+ events, most are likely no-ops (owner hasn't changed). This pattern avoids millions of unnecessary API calls to Zuora and Stripe, preserving rate limits and reducing latency.

9.3 Zuora CRM ID Lookup Strategy

The Zuora lookup uses a dual-wildcard LIKE query: 

WHERE CrmId LIKE '%{AccountId}%' OR CrmId LIKE '{AccountId}%'

This accounts for cases where the CRM ID in Zuora may have been stored with a prefix or suffix (e.g., leading characters from migration). The dual pattern ensures broad matching while still being anchored to the SFDC Account ID.

9.4 Stripe Search by App ID Code

Stripe customers are looked up using the metadata['app_id_code'] search parameter rather than SFDC Account ID. This is because Stripe's customer metadata stores the Intercom App ID Code as the primary cross-reference key, not the SFDC Account ID directly.

9.5 Forced-Error Loop Break Pattern

Within the foreach loops, the recipe uses a distinctive Workato pattern:

  • The processing logic is wrapped in a try block.
  • After processing a record, a stop with error action fires intentionally.
  • The outer catch block catches this forced error and does nothing, effectively breaking the loop and advancing to the next record.

This is necessary because Workato's foreach loops do not support break or continue natively. The pattern comment reads: "This block catches the forced break loop error and breaks the loop by having nothing defined in the error block."

SECTION 11: EDGE CASES

11. Edge Cases

11.1 Unassigned Account Owner

Scenario: AccountOwnerId__c is blank in the SFDC Platform Event payload.
Behavior: Recipe logs the issue, publishes to error topic, and stops. No updates are made to Zuora or Stripe. This prevents writing null/empty owner data to downstream systems.

11.2 No Matching Records in Either System

Scenario: Both Zuora ZOQL query and Stripe search return 0 results.
Behavior: Recipe logs "Found 0 matches" and stops. This occurs when a new SFDC Account has not yet been provisioned in Zuora/Stripe, or when the App ID Code hasn't been set.

11.3 Multiple Matching Records

Scenario: The Zuora LIKE query or Stripe metadata search returns multiple records.
Behavior: The foreach loop processes ALL matching records. Each record is independently compared and updated if needed. The forced-error loop break pattern ensures each record in the loop is processed cleanly before advancing.

11.4 Zuora API Errors

Scenario: Zuora update call fails (e.g., invalid data, API timeout).
Behavior: The try/catch block catches the error, logs "Failed to update Zuora due to an error", publishes the error to Pub/Sub Topic 37790 with the Zuora Customer ID as identifier, and continues processing (the catch block allows the loop to proceed to the next record).

11.5 Stripe API Errors

Scenario: Stripe customer update fails (e.g., invalid customer ID, rate limit).
Behavior: Similar to Zuora: caught by try/catch, logged, published to error topic. The Stripe Customer ID is included in the error message for debugging.

11.6 AE Champion Data in Zuora vs. Stripe

Scenario: AE Champion fields are present in the SFDC payload.
Behavior: AE Champion data (AEChampion__c, AEChampionEmail__c) is written only to Stripe, not to Zuora. Zuora only receives the three Sales Rep fields. This is by design, as Zuora's billing model doesn't use champion metadata.

11.7 Wildcard CRM ID Matching

Scenario: The dual-wildcard LIKE query could theoretically match unrelated Zuora accounts if an Account ID is a substring of another CRM ID.
Behavior: The recipe does not validate the match further. A false positive would result in the wrong Zuora account being updated. This is mitigated by the fact that SFDC 18-character IDs are unique and unlikely to be substrings of other IDs.

12.2 Failure Analysis

The 0.40% failure rate (11,398 out of 2,884,188 total jobs) is very low for a recipe of this volume. Analysis of recent failed jobs shows that most failures are intentional "stop with error" events used for loop termination or early exits—these are not actual errors but a Workato design pattern. True API failures from Zuora or Stripe are caught by try/catch blocks and reported via Pub/Sub.

12.3 Monitoring & Alerting

  • Pub/Sub Topic 37790: All errors publish structured messages with transactionType and identifier fields. A downstream recipe or dashboard should consume this topic for alerting.
  • Workato Logger: Each decision point logs a descriptive message (e.g., "Account Owner info already matches, no update needed"), enabling job-level debugging via the Workato console.
  • Job Report: The 8-column job report (SFDC Account ID, Name, Owner Email, App ID, App ID Code, PO Number, Zuora Customer ID, Stripe Customer ID) enables bulk job analysis in Workato's jobs dashboard.
