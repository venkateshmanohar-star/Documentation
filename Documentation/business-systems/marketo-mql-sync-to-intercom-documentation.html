Marketo RECENT_MQL_DATE Sync to Intercom

Real-time lead qualification date synchronization from Marketo to Intercom contacts
Recipe ID: 51338253 Status: Active Author: diana.sy Created: 2024-08-01 Doc Author: venkatesh.manohar Generated: 2026-04-15
1. EXECUTIVE SUMMARY

1. Executive Summary

This integration synchronizes the Recent MQL (Marketing Qualified Lead) Date and Recent MQL Type from Marketo to Intercom contact records in real time. When a lead is updated in Marketo with a new MQL date, the recipe immediately pushes that data to Intercom via a custom webhook endpoint, ensuring that Sales and Support teams in Intercom always have the latest lead qualification information.

The system processes approximately 130,000+ successful jobs since inception, with a 99.7% success rate. It uses Intercom's custom /hooks/marketo/lead_recent_mql_update endpoint and includes error handling that publishes failures to the centralized AOF (Automation Operations Framework) topic for monitoring and alerting.

2. BUSINESS OVERVIEW

2. Business Overview

Problem Solved

Marketing teams qualify leads in Marketo based on engagement scoring, campaign interactions, and demographic fit. When a lead reaches MQL status, this qualification event needs to be visible to customer-facing teams in Intercom so they can take timely action -- engaging hot leads with relevant messaging, routing conversations appropriately, and personalizing outreach based on lead maturity.

What Happens from the Business Perspective

  1. Lead qualifies in Marketo: A lead's Recent_MQL_Date__c field is set when they meet qualification criteria (e.g., scoring threshold, campaign engagement, form fills).
  2. Data syncs to Intercom: Within seconds, the MQL date and MQL type are pushed to the matching Intercom contact (matched by email address).
  3. Sales/CS teams act: Intercom operators can now see MQL status on contact records, use it for routing rules, build segments of recently qualified leads, and trigger automated campaigns.

Key Business Value

  • Speed to lead: Real-time sync ensures no delay between qualification and sales engagement.
  • Unified customer view: Intercom contacts enriched with marketing qualification data, enabling better conversations.
  • MQL type context: The Recent_MQL_Type__c field provides context on how the lead was qualified (e.g., inbound, product-led, event-driven).
  • Automation enablement: Intercom workflows can trigger based on MQL date presence/recency.

7. PROCESSING STAGES

7. Processing Stages

Stage 1: Ingestion

Component: Marketo Trigger -- "Updated Lead"

Mechanism: Polling trigger that monitors for lead updates in Marketo. Captures the email, Recent_MQL_Date__c, and Recent_MQL_Type__c fields. Uses a page token (nextPageToken) to track position in the Marketo activity stream.

Volume: Approximately 160+ leads per day based on recent job history (~1,000 jobs over ~6 days).

Stage 2: Filtering / Validation

Component: Trigger Filter

Logic: Only processes leads where Recent_MQL_Date__c is present (not null/empty). This prevents unnecessary API calls for lead updates that do not involve MQL qualification.

Stage 3: Transformation and Routing

Component: HTTP POST Action ("Update Intercom")

Transformation: Constructs a JSON payload mapping Marketo field names to Intercom-expected field names:

  • email mapped to email (used for contact matching)
  • Recent_MQL_Date__c mapped to recent_mql_date
  • Recent_MQL_Type__c mapped to recent_mql_type
  • Account property intercom_marketo_hooks_secret mapped to secret (authentication)

Stage 4: Output / Delivery

Component: Intercom Custom Webhook Endpoint

Endpoint: POST https://api.intercom.io/hooks/marketo/lead_recent_mql_update

Authentication: Shared secret passed in the request body. The endpoint validates the secret before processing.

Content-Type: application/json

Stage 5: Error Handling and Logging

Component: Try/Catch Block + Workato Pub/Sub

On Error: Publishes a structured error message to the AOF | PS-001 | AOF_Topic topic containing job details, error code, error message, and severity. The job then terminates with an error status.

Retry: No automatic retries configured (max_retry_count = 0, retry_interval = 2s).

9. KEY BUSINESS LOGIC

9. Key Business Logic

Trigger Filter Logic

The recipe only processes Marketo lead updates where the Recent_MQL_Date__c field is present (not null or empty). This is the sole filtering criterion -- there is no additional classification or routing logic. Every lead that has an MQL date set or updated is synced to Intercom.

Contact Matching Strategy

Intercom matches contacts by email address. The Marketo lead's email field is passed in the POST body. If a matching Intercom contact (Lead or User) exists, its recent_mql_date and recent_mql_type custom attributes are updated. The matching and update logic is handled server-side by Intercom's webhook handler.

Authentication

The API call includes a secret field in the JSON body, sourced from the Workato account property intercom_marketo_hooks_secret. This shared secret is validated by the Intercom endpoint to ensure only authorized requests are processed.

Error Handling Strategy

  • No retries: max_retry_count = 0. Failed jobs are not retried automatically.
  • Non-2xx = error: The HTTP action is configured with ignore_http_errors: false, meaning any non-2xx response triggers the catch block.
  • Error publication: All errors are published to the AOF topic with severity LOW and project_notifications_required: TRUE.
  • Job termination: After publishing the error, the job is stopped with stop_with_error: true, ensuring it appears as "failed" in job history.

Fallback Behavior

There is no fallback mechanism. If the Intercom endpoint is down or the email does not match an Intercom contact, the error is logged to AOF but the data is not queued for retry. Manual reprocessing would be needed for failed records.

11. EDGE CASES

11. Edge Cases and Special Scenarios

No Matching Intercom Contact

If the email from Marketo does not match any existing Intercom contact (Lead or User), the behavior depends on Intercom's webhook handler. The recipe itself does not handle this case -- it will either succeed silently (if the endpoint returns 2xx) or fail (if the endpoint returns an error). Based on the low failure rate (~0.3%), most emails likely have matching contacts.

MQL Date Cleared / Removed

The trigger filter requires Recent_MQL_Date__c to be present. If a lead's MQL date is cleared (set to null), the recipe will not fire, meaning the old value will persist in Intercom. There is no mechanism to clear or reset the MQL date in Intercom.

Duplicate Lead Updates

If a lead is updated multiple times in rapid succession, each update with a present MQL date will trigger a separate job. Since the data is idempotent (setting the same date/type), duplicate processing is harmless but adds to task consumption.

Email Changes in Marketo

If a lead's email address changes in Marketo after an MQL sync, the old Intercom contact retains the MQL data while the new email-based contact does not. There is no update propagation for email changes.

Intercom API Downtime

With zero retries configured, any Intercom downtime results in permanent failure for affected leads. These must be reprocessed manually or through a separate reconciliation process.

Secret Rotation

The shared secret (intercom_marketo_hooks_secret) is stored as a Workato account property. If rotated, both the Workato property and the Intercom endpoint validation must be updated simultaneously to avoid auth failures.

12. OPERATIONAL NOTES

12. Operational Notes

Deduplication

There is no deduplication mechanism. Each Marketo lead update with an MQL date triggers a job. The operation is idempotent (setting the same value repeatedly has no adverse effect), so deduplication is not critical for data integrity but impacts Workato task consumption.

Error Handling

  • Try/Catch pattern wraps the HTTP POST to Intercom.
  • No retries (max_retry_count = 0). Failed jobs fail immediately.
  • Error published to AOF topic (PS-001) with full diagnostic context.
  • Job stopped with error after publishing, ensuring visibility in Workato dashboard.
  • Severity: LOW -- errors are classified as low severity in the AOF framework.
  • Historical failure rate: 418 failures out of 130,305 total jobs (0.32%).

Monitoring and Alerting

  • Errors published to AOF | PS-001 | AOF_Topic with project_notifications_required: TRUE.
  • Job report columns configured: Lead ID and Email Address for easy debugging in Workato's job history.

Limitations

  • One-directional sync: Data flows Marketo to Intercom only. Changes in Intercom are not reflected back to Marketo.
  • No retry mechanism: Failed syncs require manual intervention.
  • No MQL date clearing: If MQL date is removed in Marketo, Intercom retains the old value.
  • Email-only matching: No fallback matching strategy (e.g., by Marketo ID or phone).
  • Shared secret auth: Less secure than OAuth; relies on secret remaining confidential.
  • No batching: Each lead update processed individually; high-volume MQL campaigns could create many sequential jobs.
