1. EXECUTIVE SUMMARY

1. Executive Summary

Pull_Netsuite_Account is a scheduled Workato recipe that refreshes a lookup table used by the broader Payroll-Workday-NetSuite integration pipeline. Every Monday at 8:00 AM Pacific, it queries a Snowflake staging table (INTERCOM_PROD.SHARED_EXTERNAL_WORKATO.EXT_NETSUITE_EXPORT_PAYROLL) containing NetSuite reference objects—Departments, Accounts, Locations, and Classes—truncates the existing Payroll-Netsuite-Lookup table in Workato, and reloads the full dataset as a batch insert.

This "full-refresh" pattern ensures that downstream payroll journal-entry recipes always map to the current NetSuite Chart of Accounts, department hierarchy, location list, and class segments without stale or orphaned references.

Key Value: Eliminates manual lookup maintenance and guarantees that payroll postings to NetSuite reference valid, up-to-date master data every week.
2. BUSINESS OVERVIEW

2. Business Overview

Business Context

Intercom's payroll processing pipeline moves compensation, benefit, and tax data from Workday (the HR/Payroll system of record) into NetSuite (the ERP/GL system of record). To create valid journal entries in NetSuite, each line item must be coded with the correct Account, Department, Location, and Class internal IDs.

These reference values change over time—new departments are created, accounts are renamed, locations are added for new offices, and class segments evolve with the business. This recipe keeps the mapping data fresh so that the payroll-to-GL posting recipes do not fail due to stale references.

  • If a new department or account is created in NetSuite mid-week and payroll runs before the next Monday sync, journal entry lines may fail to map correctly, causing downstream recipe errors or mis-coded GL entries.

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

The recipe follows a simple three-step Extract-Truncate-Load (ETL) pattern:

  1. Extract: A SQL query runs against Snowflake to pull all four object types (Department, Account, Location, Class) with a UNION ALL, applying a special key-extraction rule for departments.
  2. Truncate: The Workato lookup table is completely cleared to remove stale entries.
  3. Load: The full result set is batch-inserted into the lookup table with four columns: Object, Values (internal ID), Key, and value_name.
Design Note: Truncate happens before the Snowflake query in the step ordering (Step 1 = truncate, Step 2 = query, Step 3 = load), but logically this means the lookup table is empty during the ~15 seconds the query executes. This is acceptable because the recipe runs at 8:00 AM Monday, well outside payroll processing windows.

6. END-TO-END FLOW

6. End-to-End Flow

Flow Narrative

  1. Trigger (Clock): Every Monday at 8:00 AM Pacific Time, the Workato scheduler fires the recipe.
  2. Truncate Lookup Table: All existing entries in the Payroll-Netsuite-Lookup Workato lookup table (ID 4776437) are deleted to prepare for a clean reload.
  3. Execute Snowflake SQL: A UNION ALL query runs against EXT_NETSUITE_EXPORT_PAYROLL to extract all four object types. For Departments, a regex extracts the leading numeric code as the key column; for other objects, the value_name is used directly as the key.
  4. Batch Insert to Lookup Table: The full SQL result set (~753 rows) is inserted as a batch into the lookup table with columns: Object, Values (internal ID), Key, and value_name.
  5. Stop: The recipe completes with a clean stop (no error).

9. KEY BUSINESS LOGIC

9. Key Business Logic

Department Key Extraction

The most significant piece of business logic is the differentiated treatment of Department objects versus all other object types in the SQL query:

Object TypeKey DerivationExample
Department REGEXP_SUBSTR(value_name, '^[0-9]+') "5215 Sales Assist - Consultative Support" → key = 5215
Account, Location, Class value_name (used directly) "1081USD" → key = 1081USD

Rationale: NetSuite department names follow a convention of "numeric_code + space + descriptive name". The downstream payroll journal-entry recipes use the numeric department code (from Workday) to look up the NetSuite internal ID. By extracting just the leading digits, the lookup key matches what Workday provides. For Accounts, Locations, and Classes, the full name is the natural key.

Full-Refresh Pattern

The recipe uses a truncate-and-reload strategy rather than an incremental upsert. This simplifies logic (no need to detect deletes or renames) and guarantees consistency, at the cost of a brief window where the lookup table is empty.

11. EDGE CASES

11. Edge Cases

Known Edge Cases and Risks

#ScenarioImpactMitigation
1 Department name without leading number: If a NetSuite department is created with a name that does not start with digits (e.g., "Unassigned"), REGEXP_SUBSTR returns NULL. The lookup key for that department will be NULL, making it unmatchable by downstream recipes. Enforce naming convention in NetSuite; add a COALESCE in the SQL to fall back to value_name.
2 Empty lookup table window: Between Step 1 (truncate) and Step 3 (batch insert), the lookup table is empty for ~10 seconds. If another recipe tries to look up a value during this window, it will get no match. Acceptable risk since the recipe runs Monday 8 AM before payroll processing. No other recipes run concurrently at that time.
3 Snowflake query failure after truncate: If the Snowflake query (Step 2) fails after the truncate (Step 1) has already completed, the lookup table remains empty. All downstream lookups fail until the recipe is rerun or the table is manually reloaded. Monitor recipe failures; consider reordering steps (query first, then truncate + insert) or adding error handling with rollback.
4 Duplicate department numeric codes: Two departments with the same leading number but different names would produce duplicate keys. Downstream lookup might return the wrong department's internal ID. NetSuite department numbering convention should ensure uniqueness; validate periodically.
5 Snowflake staging data staleness: If the Fivetran sync or dbt job feeding EXT_NETSUITE_EXPORT_PAYROLL fails, the recipe will reload stale data. The lookup table will appear refreshed but contain outdated NetSuite reference data. Monitor Fivetran sync health; consider adding a freshness check to the SQL query.

Improvement Recommendations

  • Reorder steps: Execute the Snowflake SQL query before truncating the lookup table to avoid an empty-table window if the query fails.
  • Add error handling: Wrap the truncate + insert in error-handling blocks so the recipe can alert on failure without leaving the lookup table empty.
  • Add NULL key check: Include a COALESCE(REGEXP_SUBSTR(value_name, '^[0-9]+'), value_name) for Department rows to handle edge cases where the name does not start with a number.
  • Freshness check: Consider adding a query to verify the Snowflake staging table was updated within the expected window before loading.
