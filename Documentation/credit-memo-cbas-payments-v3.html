2.0 Master Recipe — Credit Memo CBAs and Payments

Recipe ID: 58628332 Status: Active Author: venkatesh.manohar Generated: 2026-04-01 Last Run: 2026-04-01 Jobs Succeeded: 700 Jobs Failed: 3

1. Executive Summary

This integration is a master orchestrator within Intercom's Order-to-Cash (OTC) pipeline that automates three critical financial processes between Stripe (payment processor) and NetSuite (ERP system):

  1. Credit Memo creation — When a customer receives a credit note in Stripe, this system creates the corresponding Credit Memo in NetSuite so the books reflect the customer's reduced balance.
  2. Customer Balance Adjustment (CBA) creation — When Stripe issues a credit balance adjustment (e.g., a credit applied toward future invoices), this system records the matching CBA record in NetSuite.
  3. Customer Payment recording — When Stripe collects a payment, this system creates the Customer Payment in NetSuite, linking it to the correct invoice and sales order.
  4. Refunds, Voided Invoices, and Cancellations (async) — A secondary branch handles refunds, uncollectible invoices, voided invoices, and cancelled sales orders.
Why it matters: Without this automation, the Finance team would need to manually create thousands of credit memos, CBAs, and payments in NetSuite every day. This recipe processes them automatically, ensuring Stripe and NetSuite stay in sync for accurate revenue recognition and financial reporting.

2. Business Overview

What Problem Does This Solve?

Intercom bills customers through Stripe. When customers receive credits, refunds, or make payments, those financial events must be accurately recorded in NetSuite (the system of record for accounting). Manual entry would be error-prone and impossibly slow at Intercom's scale.

4. Architecture Overview

Layered Architecture

Source Layer (Stripe)

Business: All billing activity originates here — invoices, payments, credits, and refunds.

Technical: Fivetran replicates Stripe data into Snowflake raw tables. dbt models clean and reshape the data into NETSUITE_EXPORT_* tables.

Data Layer (Snowflake)

Business: A staging area that holds prepared records waiting to be sent to NetSuite.

Technical: Six key tables in SHARED_EXTERNAL_WORKATO schema. Dedup via WORKATO_LOGS table ensures no record is processed twice.

Orchestration Layer (Workato)

Business: The "traffic controller" that reads staged records and routes them to NetSuite in the correct order.

Technical: Hierarchical recipe structure: 1 master → 4 sub-masters → 4+ recipe functions. Sequential for Credit Memos and CBAs; async for Payments and Refunds.

Target Layer (NetSuite)

Business: Where financial records live for reporting, audit, and revenue recognition.

Technical: Records created via SuiteTalk REST API: Credit Memo, Customer Deposit (CBA), Customer Payment. Each write is logged back to Snowflake for deduplication.

6. End-to-End Flow

Level 0 — Simple View

Every day, Stripe billing events (credits, payments, refunds) flow through Snowflake into Workato, which creates the matching financial records in NetSuite. The system ensures every Stripe transaction has a corresponding NetSuite record, keeping the books balanced.

Level 1 — Detailed View

  1. Extraction: dbt models in Snowflake transform raw Stripe data into standardized export tables (NETSUITE_EXPORT_*).
  2. Deduplication Check: Each recipe queries Snowflake with a LEFT JOIN to WORKATO_LOGS, filtering out records that already have a netsuite_internal_id — meaning they were already processed.
  3. Sequential Processing: The master recipe calls child recipes in order: Credit Memos first (Rec 5.0), then CBAs (Rec 8.0), then Payments (Rec 6.0). This order matters because credit memos may need to exist before CBAs can reference them.
  4. Record-Level Processing: Each child "Rec" recipe queries Snowflake for unprocessed rows, then iterates (For Each) calling the corresponding "RF" recipe function to create the individual NetSuite record.
  5. Logging: After each NetSuite write, the RF recipe logs the result (success/failure, NetSuite internal ID) back to WORKATO_LOGS.

7. Processing Stages

Stage 1: Ingestion (dbt / Snowflake)

Business View

Stripe billing data (credit notes, payments, refunds) is automatically copied into Snowflake and transformed into clean, standardized tables ready for NetSuite.

Technical View

Fivetran extracts raw Stripe data; dbt models in SHARED_EXTERNAL_WORKATO schema create the NETSUITE_EXPORT_* tables. Each table is grain-specific (e.g., one row per payment, one row per credit adjustment).

Stage 2: Deduplication Query

Business View

Before processing, the system checks which records have already been sent to NetSuite, skipping duplicates. Only new records are processed.

Technical View

Each Rec recipe runs a SQL query with a LEFT JOIN to WORKATO_LOGS using external_id. The WHERE clause filters for rows where wl.netsuite_internal_id IS NULL and date is within range. Limit: 50,000 rows per run.

Stage 3: Routing / Orchestration

Business View

The master recipe ensures records are processed in the correct business order: Credit Memos first, then CBAs, then Payments, and finally Refunds.

Technical View

Recipe 58628332 calls children sequentially via call_recipe (sync). The last call (3.0 Refunds Master) is call_recipe_async so it does not block. A try/catch wrapper sends email alerts on timeout.

Stage 4: Record Processing

Business View

Each credit, payment, or adjustment is individually processed: the customer is verified/created in NetSuite, the matching invoice or sales order is located, and the financial record is written.

Technical View

RF recipes use NetSuite's SuiteTalk API: search_object_v2 to find existing records (customer, invoice, sales order), add_object to create new records, and update_object for updates. Custom code blocks handle edge cases and field mapping.

Stage 5: Logging

Business View

Every processed record is logged so the system knows it was handled, preventing duplicate creation on the next run.

Technical View

After each NetSuite write, the RF recipe inserts/updates a row in WORKATO_LOGS with the external_id, netsuite_internal_id, source_table_name, timestamp, and any error details.

9. Key Business Logic

Credit Memo Processing (Rec 5.0 / RF 5.0)

  • Grouping: Credit notes are grouped by credit_note ID in Snowflake. RF 5.0 receives one group and creates a single Credit Memo with multiple line items.
  • Invoice Linkage: RF 5.0 searches NetSuite for the original invoice using the Stripe invoice ID. If found, the Credit Memo is linked to it.
  • Customer Verification: If the customer doesn't exist in NetSuite, RF 5.0 calls Rec 1.01 (Customer Master) to create them before proceeding.

CBA Processing (Rec 8.0 / RF 8.0)

  • Sales Order Lookup: RF 8.0 queries NETSUITE_EXPORT_SALES_ORDER in Snowflake to find the matching sales order for the credit adjustment.
  • CBA Types: The type field from Snowflake determines the CBA type in NetSuite (e.g., "Adjustment Credit", "Credit Applied").
  • Customer Internal ID: The CBA query joins to NETSUITE_EXPORT_CUSTOMER to get the customer's NetSuite internal ID, avoiding an extra API call.

Payment Processing (Rec 6.0 / RF 6.0)

  • Invoice Search: RF 6.0 searches NetSuite for the invoice matching the Stripe invoice ID, then applies the payment against it.
  • Payment Methods: Supports Card, ACH, and Wire payments. Bank Transfer Payments are explicitly excluded from processing (check_reference_number <> 'Bank Transfer Payment').
  • Fee Tracking: Stripe processing fees (fee_amount) are passed to NetSuite for expense tracking.
  • Async Processing: RF 6.0 is called asynchronously to handle the high volume (804K+ records processed to date).

Refunds / Cancellations (3.0 Master)

  • Three Sub-Flows: Rec 7.0 handles refunds, Rec 10.1 handles voided invoices, Rec 9.0 handles cancelled sales orders.
  • Async Launch: The entire refunds branch runs asynchronously to avoid blocking the main Credit Memo / CBA / Payment flow.
  • Note: Rec 10.0 (Uncollectible Invoices) exists in the code but is currently skipped (disabled).

How Deduplication Works

Each export query uses a CTE called stg_workato_log that selects the most recent log entry per external_id (using ROW_NUMBER() ... PARTITION BY external_id ORDER BY job_run_completed_at DESC). The main query LEFT JOINs to this CTE and only processes rows where netsuite_internal_id IS NULL — meaning no successful NetSuite write has been recorded.

11. Edge Cases & Special Scenarios

Missing Customer in NetSuite

Scenario: A credit memo or CBA references a Stripe customer that doesn't yet exist in NetSuite.

Handling: Both RF 5.0 and RF 8.0 call Rec 1.01 Customer Master to create the customer on-the-fly before proceeding with the financial record.

Missing Sales Order

Scenario: RF 8.0 cannot find a matching sales order in NETSUITE_EXPORT_SALES_ORDER.

Handling: The CBA is still created, but without a sales order linkage. The log records this for reconciliation.

Bank Transfer Payments

Scenario: Some payments come via bank transfer (wire) rather than card/ACH.

Handling: Rec 6.0 explicitly excludes rows where check_reference_number = 'Bank Transfer Payment'. These are handled by a separate process. (Note: wire transfers were added to the model as of 2026-03-04.)

Timeout / Too Many Records

Scenario: A large backlog causes the recipe to exceed Workato's execution time limit.

Handling: The try/catch block in the master recipe catches the timeout error and sends an email alert to danny.mcgean@intercom.io, nan.lu@intercom.io, chase.clark@connorgp.com, and diana.sy@intercom.io.

Negative Amounts

Credit memos and CBAs naturally carry negative amounts (reducing a customer's balance). The RF recipes pass these amounts directly to NetSuite, which treats them as credits on the customer's account.

Skipped Rec 10.0 (Uncollectible Invoices)

The 3.0 Refunds Master includes a call to Rec 10.0 (Uncollectible Invoice Master), but this step is currently marked as skip: true in the recipe configuration. This likely indicates the uncollectible invoice process is handled elsewhere or is under development.

12. Operational Notes

Deduplication

  • Every recipe query includes a LEFT JOIN to WORKATO_LOGS filtered by source_table_name and environment_type = 'prod'.
  • Only the most recent log per external_id is considered (via ROW_NUMBER).
  • Records with a non-null netsuite_internal_id in the log are skipped.

Error Handling

  • Master level: Try/catch with email notification on failure.
  • Sub-orchestrator level: Each Rec recipe has its own try/catch (currently with empty catch blocks — errors propagate to the master).
  • RF level: Individual record failures are logged to WORKATO_LOGS with error details but do not stop the batch.

Execution Cadence

  • The master recipe (58628332) is called by a parent scheduler (0.0 Scheduler). It receives a Date parameter that sets the lookback window.
  • All date-filtered queries process records from the given start date up to DATEADD(day, -1, CURRENT_DATE) — i.e., up to yesterday.
  • Row limit per query: 50,000 records.

Key Limitations

  • Maximum 50,000 rows per recipe run. If more records accumulate, they will be picked up in subsequent runs.
  • The payment branch (RF 6.0) runs asynchronously, meaning failures may not be immediately visible in the master recipe's job report.
  • Rec 10.0 (Uncollectible Invoices) is currently disabled.
  • CBAs from manual invoices (proration/downgrade) may not appear in NetSuite when the originating invoice routing debug string prevents RF 8.0 from parsing the NS ID.
