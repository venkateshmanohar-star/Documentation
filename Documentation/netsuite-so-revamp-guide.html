1 The Problem

What's Going Wrong?

When a customer changes their contract (upsell, downsell, renewal), Salesforce CPQ generates a "rip-and-replace" transaction. Instead of saying "customer added 2 seats," it tears up the old order and recreates the entire thing with positive and negative lines.

-3,000
resolutions removed
+
+5,000
resolutions added
=
+2,000
net change (what actually happened)

~40% of amendments are "rip-and-replace" — creating noise instead of clarity

Core Issues

  • CPQ amendments generate full order replacements, not net changes
  • NetSuite doesn't support negative quantities for usage-based products
  • Multiple +/- lines create ambiguity in revenue attribution
  • Revenue recognition (ARM) for usage items can't be automated

Business Impact

  • Manual calculation & topside entries for usage items
  • Slower month-end close
  • Increased financial and audit risk
  • Finance teams manually interpret Sales Orders
  • Engineering relies on fragile Workato logic

4 The Netting Algorithm (5 Stages)

The netting algorithm lives in ext_netsuite_export_sales_order_v2.sql and transforms CPQ's rip-and-replace output into clean net changes through 5 stages:

Stage 1

Line-Level Netting

Purpose: Consolidate multiple CPQ lines for the same item within the same amendment into a single netted row.

How It Works

For each amendment, lines are grouped by (contract, opportunity, item, date range). Then quantities and amounts are summed to get the net change.

-3,000 qty
-$2,760
+
+5,000 qty
+$4,600
→
+2,000 qty
+$1,840
Technical Detail: GROUP BY & Aggregation
GROUP BY (contract_id, opportunity_id, item, rev_rec_start_date, rev_rec_end_date, billing params, prorate_multiplier, ...) AGGREGATE: actual_qty = SUM(actual_qty) -- net quantity amount = ROUND(SUM(amount), 2) -- net dollars discount_pct = AVG(discount_percent) -- known issue! unit_list_price = MAX(unit_list_price)
Stage 2

Amendment Splitting: Future Billing vs Proration

Purpose: After netting, each amendment row is split into two temporal components.

Future Billing

Full billing cycles remaining after the prorated stub. Always goes to Sales Order.

future_portion = total_term - prorated_portion future_billing_qty = future_portion / billing_freq

Proration (Stub)

The partial billing cycle at the start of the amendment. Routing depends on opp-level netting.

prorated_portion = total_term % billing_freq per_month = amount / total_term proration_amount = per_month * prorated_portion
Why Split?

NetSuite ARM needs the prorated stub separately because it creates a single revenue element for the partial period, while full billing cycles each create their own revenue elements on a recurring schedule.

Stage 3

Opportunity-Level Routing

Purpose: Decide whether an opportunity's proration lines go to a Sales Order or a Credit Memo.

FOR EACH opportunity_id: opp_total = SUM(proration amounts) IF opp_total >= 0: -- Route ALL proration lines to SALES_ORDER ELSE: -- Route ALL proration lines to CREDIT_MEMO
Critical: All-or-Nothing Routing

Individual lines within an opportunity are NOT split. The entire opportunity goes one way based on its net total. Even if one line is positive and another negative, they all route together.

Stage 4

Final Union

Purpose: Combine all components into the final output.

sales_order = new_renewal -- Non-amendment orders → SALES_ORDER UNION ALL future_billing -- Full cycles → always SALES_ORDER UNION ALL positive_proration -- Net-positive proration → SALES_ORDER UNION ALL negative_proration -- Net-negative proration → CREDIT_MEMO UNION ALL professional_services

Key rule: future_billing always goes to Sales Order regardless of opp-level netting. It represents ongoing subscription commitment, always positive.

Stage 5

Change Type Labeling

Purpose: Independent of dollar netting, label each item with what type of change it represents using a running quantity total.

FOR EACH (contract_id, product), ordered by date: current_qty = running_sum(net_qty_delta) prev_qty = LAG(current_qty) IF prev_qty IS NULL: → 'NEW' IF current_qty > prev_qty: → 'UPSELL' IF current_qty < prev_qty: → 'DOWNSELL' ELSE: → 'SAME' (price-only change)
5 Fin Bucket Pipeline (4 Models)

What is Fin Bucket?

When a customer opts into bucket pricing, their items (resolutions, voice, conversation) shift from per-unit billing to a prepaid capacity model. Instead of paying per resolution, they buy a "bucket" of, say, 5,000 resolutions per month. If they exceed the bucket, they pay overage. If they don't use it all, unused capacity expires.

ext_netsuite_export_sales_order_v2

Source of truth for contracts & SO lines (the netting table)

Model 1: netsuite_finbucket_usage

"How much was consumed?"

Model 2: netsuite_finbucket_expiration

"How much capacity was allocated?"

Model 3: ext_netsuite_finbucket_combined

Usage vs Cap → in-cap, overage, expired

Model 4: ext_netsuite_finbucket_revevent

Split into per-SO-line revenue events for NetSuite

Model 1

Usage Model

Question: "How many resolutions did each customer actually use, per calendar month, within each billing period?"

How it works (5 steps)
  1. Contract universe — Pulls all contracts where is_fillable = true, one row per contract-item pair
  2. Billing cycle generation — Uses generator(rowcount => 200) to create billing cycles anchored to contract start date
  3. Opt-in clipping — Drops cycles before opted_in_to_buckets_at, clips the first surviving cycle forward to opt-in date
  4. Calendar month slicing — Each billing cycle is sliced into calendar months (e.g., Jan 15-Apr 14 quarterly cycle → Jan 15-31, Feb 1-28, Mar 1-31, Apr 1-14)
  5. Usage join — Left-joins to Stripe meter data, sums daily usage into monthly totals

Output: One row per (contract, item, month) → resolution_usage

Model 2

Expiration (Capacity Allocation) Model

Question: "How much prepaid capacity is allocated to each calendar month, per SO line?"

This is the most complex model.

Three allocation types (different math each)
TypeLogicExample
BASE actual_qty * MIN(billing_freq, months_in_term) 5,000/mo on quarterly = 15,000 cap per cycle
PRORATED billing_qty placed directly into the overlapping monthly window One-time partial-period allocation
CONVERTED_FILLABLE Remainder allocation: full cycles get qty * freq, first partial gets remainder Mid-cycle conversion creates awkward first period

Output: One row per (contract, item, SO_line, component, month) → cap_allocated

Model 3

Combined Model

Question: "For each month, how much was used, how much was overage, and how much expired?"

The cumulative waterfall

The cap is per billing period, not per month. A running total determines when usage crosses from "in-cap" to "overage":

Customer has 15,000 quarterly cap: Month 1: Used 2,000 → in_cap: 2,000 | overage: 0 | remaining: 13,000 Month 2: Used 8,000 → in_cap: 8,000 | overage: 0 | remaining: 5,000 Month 3: Used 9,000 → in_cap: 5,000 | overage: 4,000 | expired: 0 -- Cap exhausted in month 3, 4,000 is overage

expired_cap only appears on the last month of each billing period — this is when NetSuite posts "breakage" revenue for unused prepaid capacity.

Model 4

Revenue Event Model

Question: "How do we split combined totals back to individual SO lines for NetSuite revenue recognition?"

Proportional allocation

If a billing period has 15,000 cap from SO line A (10,000) and SO line B (5,000):

allocation_ratio = line_cap / total_period_cap Line A ratio: 10,000 / 15,000 = 0.6667 Line B ratio: 5,000 / 15,000 = 0.3333 -- Creates 2 event types per SO line per month: Usage event: CEIL(in_cap_usage * ratio) Expiration event: CEIL(expired_cap * ratio) -- only on last month -- External ID: {so_line_id}-{month_end}-{event_type}

Why Is the Bucket Pipeline Complex?

  1. Three allocation populations with different math (BASE, PRORATED, CONVERTED_FILLABLE)
  2. Two time axes — Billing cycles vs calendar months, must slice one into the other
  3. Cumulative waterfall — Cap is per period, not per month. Running totals required.
  1. Fan-out and re-aggregation — Expiration fans out to SO-line detail, combined rolls up, then revevent fans back out with proportional allocation
  2. Opt-in timing — Each product can have different opt-in dates per contract, creating staggered conversion points
6 Edge Cases & Gotchas
Edge Case 1: Discount Averaging (Known Issue)

When netting lines with different discount rates (e.g., 9.27% and 7%), AVG() produces 8.135%. The true effective discount should be derived from the netted amounts. Business has acknowledged this — decision is to AVG for now if the item is returned at the same qty.

Edge Case 2: Free-Item-Only Opportunities

If an opportunity contains only free items, it has no rows after netting. The INNER JOIN in positive_proration silently drops these — which is correct behavior since there's nothing to route.

Edge Case 3: Co-Term Contracts

When contract term isn't evenly divisible by billing frequency, the last billing cycle is shorter. These may have a non-zero proration stub even for "clean" amendments.
