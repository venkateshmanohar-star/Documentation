Sales Order Pipeline — End-to-End System Documentation

Complete system-level documentation: Salesforce → Fivetran → Snowflake (dbt) → Workato → NetSuite

Type: End-to-End Author: venkatesh.manohar Generated: April 9, 2026 Systems: 5 (Salesforce, Fivetran, Snowflake, Workato, NetSuite) Workato Recipes: 7 active, 2 stopped Schedule: 4x daily (5:30, 10:30, 15:30, 20:30 PT)

1. Executive Summary

Business View

When a sales rep closes a deal in Salesforce and the contract is activated, a Sales Order must be created in NetSuite (our ERP) so that Finance can recognize revenue, manage billing schedules, and track order fulfillment. This pipeline automates that entire process — from the moment a contract activates in Salesforce to a fully-formed Sales Order (with correct line items, quantities, pricing, and rev-rec dates) appearing in NetSuite.

It also handles amendments (mid-contract changes like upgrades/downgrades), cancellations, and credit memo generation for prorated adjustments — all without manual Finance intervention.

Technical View

A 5-system pipeline: Salesforce (source of truth for contracts and order items) → Fivetran (CDC replication to Snowflake) → Snowflake (dbt staging/intermediate models + Workato export tables) → Workato (orchestration, business logic, API calls) → NetSuite (destination ERP).

Workato runs on a cron schedule (4x/day), queries Snowflake for unsynced activated contracts, groups order items by contract, and creates/updates Sales Orders in NetSuite via SuiteTalk API. A log table (WORKATO_LOGS) tracks sync state to ensure idempotency.

Reliability: The core recipe (RF 3.0) has processed 16,544 jobs with 0 failures (213,499 lifetime API tasks). The pipeline runs 4 times daily with email alerting on timeout errors.

2. Business Overview

What Problem Does This Solve?

Intercom's revenue operations require that every activated Salesforce contract results in a corresponding NetSuite Sales Order. Without automation, Finance teams would need to manually create thousands of sales orders per month, including complex amendment calculations, proration logic, and credit memo generation for mid-contract changes.

Business Flow

  1. Sales closes a deal — A Salesforce Opportunity is won, an Order is created, and a Contract is activated
  2. Order items are defined — Each contract has line items with products, quantities, pricing, billing schedules, and rev-rec dates
  3. Data flows to Snowflake — Fivetran replicates Salesforce data to Snowflake; dbt transforms it into clean staging/intermediate tables; a separate Snowflake table (NETSUITE_EXPORT_SALES_ORDER) is populated with the data needed for NetSuite
  4. Workato picks up unsynced orders — 4x daily, Workato queries Snowflake for activated contracts not yet in NetSuite
  5. Sales Order created in NetSuite — Workato creates the SO via SuiteTalk API with all custom fields, line items, and rev-rec schedules
  6. Amendments handled automatically — If an SO already exists (contract amendment), Workato updates it with new lines and creates a Credit Memo for prorated negative amounts
  7. Cancellations processed — When a contract is cancelled in Salesforce, Workato closes all SO lines and sets the cancellation date

4. Architecture Overview

Source Layer

Salesforce CRM holds the canonical contract and order data. When a sales rep activates a contract, Salesforce creates Order and OrderItem records with products, pricing, quantities, and billing schedules.

Source Layer — Technical

Fivetran CDC connector continuously replicates Order and OrderItem objects from Salesforce to FIVETRAN.FIVETRAN_SALESFORCE. Both production and sandbox environments are synced separately.

Data Layer

Snowflake serves as the central data platform. A single dbt pipeline transforms Salesforce data through staging and intermediate layers, then produces the export table that Workato reads. Analytics tables (FCT_ORDERS) branch off as a side path from the intermediate layer.

Data Layer — Technical

Sequential dbt pipeline: FIVETRAN_SALESFORCE (raw) → STG_SALESFORCE_ORDER_ITEM (view, dedup/filter) → INT_SALESFORCE_ORDER_ITEM (incremental, 495K rows) → NETSUITE_EXPORT_SALES_ORDER (340K rows, 48 cols, team-enterprise-systems). Side branch: INT → FCT_ORDERS (54K rows, analytics).
Workato reads the export table, joined with NETSUITE_EXPORT_CUSTOMER and NETSUITE_EXPORT_ITEM, cross-referenced against WORKATO_LOGS.

Orchestration Layer

Workato is the automation engine. It runs 4x daily, finds contracts that need syncing, applies business rules (amendments, prorations, cancellations), and creates the NetSuite records.

Orchestration Layer — Technical

Scheduler (0.0, cron 4x/day) → Master (Rec 3.0, Snowflake SQL) → Child (RF 3.0, NetSuite API). Concurrency managed via sleep(1) between API calls. Lookups for subsidiary, terms, and payment type via Workato lookup tables.

Target Layer

NetSuite receives the Sales Orders and Credit Memos. Finance uses these for revenue recognition, billing, and financial reporting. Custom fields link back to Salesforce for traceability.

Target Layer — Technical

NetSuite SuiteTalk API. New SOs use form 116; amendment Credit Memos use form 114. 16+ custom body fields and 11+ custom line fields. External ID = Salesforce Contract ID for idempotent upserts.

6. End-to-End Flow

The complete data journey for a Sales Order follows this sequential path across five systems:

  1. Salesforce CRM — Sales rep activates a contract, creating Order and OrderItem records with products, pricing, and billing schedules.
  2. Fivetran CDC — Continuously replicates Salesforce objects (Order, OrderItem, Contract, Opportunity, Account) to FIVETRAN.FIVETRAN_SALESFORCE in Snowflake. A separate connector also syncs NetSuite data back for reconciliation.
  3. Snowflake dbt Pipeline — A sequential dbt pipeline transforms the raw data:
    • STG_SALESFORCE_ORDER_ITEM — staging view (dedup, rename, filter sandbox)
    • INT_SALESFORCE_ORDER_ITEM — intermediate incremental table (495K rows, team-data-eng)
    • NETSUITE_EXPORT_SALES_ORDER — export table (340K rows, 48 cols, team-enterprise-systems) — this is what Workato reads
    • Side branch: FCT_ORDERS (54K rows) branches off from INT for analytics and reconciliation
  4. Workato Orchestration — Runs 4x daily via cron scheduler. Rec 3.0 queries the export table, groups order items by contract, and for each contract calls RF 3.0 which: looks up/creates the customer in NetSuite, determines new vs. amendment, builds SO line items, and calls the NetSuite API.
  5. NetSuite ERP — Receives Sales Orders (form 116), Credit Memos (form 114 for prorated amendments), and Customer records via SuiteTalk API. Revenue recognition begins once the SO is approved.
Feedback loops: Workato writes sync results (ns_internal_id, status) back to WORKATO_LOGS in Snowflake for dedup tracking. Fivetran’s NetSuite connector syncs transaction data back to FIVETRAN_NETSUITE for monthly reconciliation.

Key SQL: Rec 3.0 Master Query (Snowflake) 
SELECT so.external_id, customer, salesforce_contract_id, date,
  original_contract_start_date, salesforce_opportunity_id, so.status,
  po_number, memo, customer_payment_type, so.terms,
  item, item_description, actual_qty, amount, list_price, discount,
  rev_rec_start_date, rev_rec_end_date, sfdc_order_product_id,
  billing_schedule, total_contract_value, c.subsidiary, i.item_code,
  prorate_multiplier, so.netsuite_internal_id,
  default_subscription_term, order_type, proserve_type, so.reseller,
  is_co_term, contract_term, pricebook_name
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_SALES_ORDER so
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER c
  ON so.customer = c.external_id
LEFT JOIN SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_ITEM i
  ON so.item = i.billing_plan_id
LEFT JOIN stg_workato_log wl
  ON so.external_id = wl.external_id
WHERE so.date >= '{Start_Date}'
  AND wl.netsuite_internal_id IS NULL
  AND ((so.amount != 0 OR so.total_contract_value = 0)
       OR (so.amount = 0 AND discount = 100))
  AND contract_status = 'Activated'
ORDER BY so.date

9. Key Business Logic

New Sales Order vs. Amendment Detection

Business View

When a customer upgrades, downgrades, or modifies their subscription mid-contract, the system detects this as an amendment rather than creating a duplicate Sales Order. It adds the new lines to the existing SO and generates a Credit Memo for any prorated negative amounts.

Technical View

RF 3.0 searches NetSuite for an existing SO by externalId (= Salesforce Contract ID). If found → amendment path. It computes Future_Billing_Multiplier and Proration_Multiplier per line, builds amendment and prorated-negative line lists, then updates the SO and creates a CREDMEM (form 114) if negative proration lines exist.

Key Business Rules

RuleLogicWhy
Skip $0 lines (with exceptions)amount != 0 OR total_contract_value = 0 OR discount = 100$0 lines are noise unless the entire contract is $0 or fully discounted

Cancellation Logic

Rec 9.0 finds contracts where contract_status = 'Cancelled' AND an SO exists in NetSuite (has internal ID) AND the cancellation hasn't been logged yet. RF 9.0 then:

  1. Searches NetSuite for the SO by external ID
  2. Sets isClosed = true on all line items
  3. Sets custbody_cg_cancellation_close (cancellation date)
  4. Updates Rev Rec End Date on each line to the cancellation date

Reconciliation Model

The Streamlit recon portal (page 4_Sales_Orders.py) reconciles SF order items against NS sales order lines using a FULL OUTER JOIN on (opportunity_id, contract_id, billing_id, service_date). Key metrics:

  • Grain: (opportunity_id, contract_id, billing_id, service_date)
  • Matching: Quantity-based (tolerance = 0) — sf_actual_qty = ns_actual_qty
  • NS service_date: MIN(rev_rec_start_date) per (opportunity, contract)
  • SF source: INT_SALESFORCE_ORDER_ITEM (activated orders)
  • NS source: FIVETRAN.FIVETRAN_NETSUITE.TRANSACTION (SALESORD type)
