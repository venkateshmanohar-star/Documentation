Payment Pipeline — End-to-End System Documentation

Complete system-level documentation: Stripe → Fivetran → Snowflake (dbt) → Workato → NetSuite Customer Payments

Type: End-to-End Author: venkatesh.manohar Generated: April 9, 2026 Systems: 5 (Stripe, Fivetran, Snowflake, Workato, NetSuite) Workato Recipes: 6 active, 2 stopped Schedule: 4x daily (5:30, 10:30, 15:30, 20:30 PT)

1. Executive Summary

Business View

When a customer pays an invoice via Stripe (credit card, ACH, wire transfer), that payment must be recorded in NetSuite (our ERP) as a Customer Payment so that Finance can reconcile cash receipts, close open invoices, and maintain accurate accounts receivable balances. This pipeline automates the entire process — from a Stripe charge event to a fully-formed Customer Payment record in NetSuite with the correct customer, amount, invoice application, and posting period.

It also handles Customer Balance Adjustments (CBAs) for Stripe credit notes, and provides a manual resync path for failed or missing payments.

Technical View

A 5-system pipeline: Stripe (source of truth for charges and payment intents) → Fivetran (data share replication to Snowflake) → Snowflake (dbt staging/intermediate models + Workato export table) → Workato (orchestration, business logic, NetSuite API calls) → NetSuite (destination ERP).

Workato runs on a cron schedule (4x/day) via the 0.0 Scheduled Master chain. Rec 6.0 queries Snowflake for unsynced payments, then calls RF 6.0 (async, per payment) which creates Customer Payment records in NetSuite via SuiteTalk API. A log table (WORKATO_LOGS) tracks sync state to ensure idempotency.

Reliability: The core recipe (RF 6.0) has processed 819,410 jobs with 0 failures (6.5M lifetime API tasks). The pipeline runs 4 times daily with email alerting on timeout errors.

2. Business Overview

What Problem Does This Solve?

Intercom collects subscription payments via Stripe across multiple currencies and subsidiaries. Without automation, the Finance team would need to manually create hundreds of thousands of Customer Payment records in NetSuite per year, match each payment to the correct invoice, set posting periods, handle fee allocations, and manage multi-currency exchange rates. This pipeline automates all of that.

Business Flow

  1. Customer pays via Stripe — A Stripe charge is created when a customer pays an invoice (credit card, ACH, or wire). Stripe records the charge, payment intent, balance transaction, and payout information.
  2. Data flows to Snowflake — Stripe's data share and Fivetran replication land charge/payment data into Snowflake. dbt transforms it through staging and intermediate layers into the export table NETSUITE_EXPORT_PAYMENT.
  3. Workato picks up unsynced payments — 4x daily, Rec 6.0 queries Snowflake for payments not yet in NetSuite (via WORKATO_LOGS dedup check).
  4. Customer Payment created in NetSuite — RF 6.0 creates the payment via SuiteTalk API, applying it to the matching invoice, setting the correct subsidiary, payment method, and AR/cash accounts.
  5. CBAs processed — Stripe credit notes (customer balance adjustments) are synced separately via Rec 8.0 / RF 8.0, creating adjustment records in NetSuite.
  6. Sync logged — Success/failure is written back to WORKATO_LOGS in Snowflake for idempotency tracking.

4. Architecture Overview

Source Layer

Stripe holds the canonical payment data. When a customer's credit card is charged or an ACH payment clears, Stripe creates a Charge object linked to a PaymentIntent and BalanceTransaction. The charge carries the amount, currency, payment method, fee breakdown, and the associated invoice ID.

Source Layer — Technical

Stripe shares data via Stripe Data Share (Snowflake native data sharing) into STAGING_STRIPE_DATA_SHARE. Additionally, Stripe events are captured via the PREP_STRIPE_EVENTS schema. The primary raw objects are CHARGE, PAYMENT_INTENT (7.1M rows), and BALANCE_TRANSACTION.

Data Layer

Snowflake serves as the central data platform. A single dbt pipeline transforms Stripe charge data through staging and intermediate layers, then produces the export table (NETSUITE_EXPORT_PAYMENT) that Workato reads. The export table enriches raw charge data with subsidiary mapping, AR/cash account codes, payment method classification, and memo construction.

Data Layer — Technical

Sequential dbt pipeline: Stripe Data Share (raw) → STG_STRIPE_PAYMENTS (4.0M rows, 11 cols) → INT_STRIPE_PAYMENTS (4.0M rows, enriched with payment_method_type, stripe_customer_id) → NETSUITE_EXPORT_PAYMENT (420K rows, 29 cols, team-enterprise-systems).
Workato reads the export table, joined with NETSUITE_EXPORT_CUSTOMER (327K rows) for subsidiary lookup, cross-referenced against WORKATO_LOGS for dedup.

Orchestration Layer

Workato is the automation engine. It runs 4x daily via the 0.0 Scheduled Master chain, which calls the 2.0 Master Recipe (Credit Memo CBAs and Payments), which in turn calls Rec 6.0 (Payment Master). For each unsynced payment row, an async call to RF 6.0 creates the NetSuite Customer Payment record.

Orchestration Layer — Technical

0.0 Scheduler (cron 4x/day) → 2.0 Master (calls Rec 5.0 CREDMEMs, Rec 8.0 CBAs, Rec 6.0 Payments sequentially) → Rec 6.0 (Snowflake SQL query) → RF 6.0 (async, per payment: NetSuite customer lookup, payment creation via SuiteTalk API). Idempotency via WORKATO_LOGS.external_id = stripe_payment_id.

Target Layer

NetSuite receives Customer Payment records. Each payment is linked to the customer and applied to the correct invoice. Finance uses these for cash receipt reconciliation, AR aging, and financial reporting. The payment carries Stripe IDs in memo/custom fields for audit trail traceability.

Target Layer — Technical

NetSuite SuiteTalk API. Customer Payments (CustPymt) are created with subsidiary, payment method, AR account, cash account, check reference number (Stripe charge ID), and applied to the invoice by Stripe invoice ID. External ID = stripe_payment_id for idempotent upserts. Customer Balance Adjustments are created via RF 8.0 for Stripe credit notes.

6. End-to-End Flow

The complete data journey for a Customer Payment follows this sequential path across five systems:

  1. Stripe — Customer pays an invoice via credit card, ACH, or wire transfer. Stripe creates a Charge (ch_xxx) linked to a PaymentIntent (pi_xxx), a BalanceTransaction (txn_xxx, containing fees and net amount), and associates it with the Invoice (in_xxx). Funds settle into a Payout (po_xxx).
  2. Fivetran / Data Share — Stripe's native Snowflake data share replicates charge, payment intent, and balance transaction data into STAGING_STRIPE_DATA_SHARE. Event-based data flows through PREP_STRIPE_EVENTS. A separate NetSuite connector syncs payment records back for reconciliation.
  3. Snowflake dbt Pipeline — A sequential dbt pipeline transforms the raw Stripe data:
    • STG_STRIPE_PAYMENTS — staging table (4.0M rows, 11 cols: stripe_payment_id, customer_id, amount, status, stripe_invoice_id, payment_intent_id, etc.)
    • INT_STRIPE_PAYMENTS — intermediate table (4.0M rows, enriched with payment_method_type, stripe_customer_id, stripe_payment_id_2)
    • NETSUITE_EXPORT_PAYMENT — export table (420K rows, 29 cols, team-enterprise-systems) — this is what Workato reads
    • Side branch: STRIPE_INVOICE_PAYMENTS (9.4M rows) for analytics and reconciliation
  4. Workato Orchestration — Runs 4x daily via the 0.0 Scheduler chain. The 2.0 Master Recipe calls Rec 6.0 (Payment Master) which queries the export table joined with WORKATO_LOGS for dedup, then for each unsynced payment, calls RF 6.0 asynchronously. RF 6.0 looks up the customer in NetSuite, determines subsidiary/AR/cash accounts, builds the payment record, and calls the SuiteTalk API.
  5. NetSuite ERP — Receives Customer Payments (CustPymt, e.g., PAY189279) applied to the correct invoice. Custom fields carry Stripe IDs (charge ID as check reference, payment ID, payout ID, invoice ID) for full traceability.
Feedback loops: Workato writes sync results (stripe_payment_id as external_id, ns_internal_id, status) back to WORKATO_LOGS in Snowflake for dedup tracking. Fivetran's NetSuite connector syncs Customer Payment transaction data back for monthly reconciliation.

Key SQL: Rec 6.0 Master Query (Snowflake) 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id ORDER BY job_run_completed_at DESC
    ) = 1
)

SELECT SUBSIDIARY, CUSTOMER, DATE,
  STRIPE_INVOICE_ID, STRIPE_PAYMENT_ID,
  STRIPE_PAYOUT_ID, PAYMENT_AMOUNT,
  PAYMENT_METHOD, CHECK_REFERENCE_NUMBER, FEE_AMOUNT
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT
LEFT JOIN stg_workato_log wl
  ON stripe_payment_id = wl.external_id
WHERE date BETWEEN '{Start_Date}' AND DATEADD(day, -1, CURRENT_DATE)
  AND wl.netsuite_internal_id IS NULL
ORDER BY created_date
LIMIT 50000

9. Key Business Logic

Payment Creation Flow (RF 6.0)

Business View

When a Stripe charge is processed, the pipeline determines which NetSuite customer it belongs to, which subsidiary the payment should post to, the correct AR and cash accounts, the payment method classification (credit card, ACH, wire), and the applicable exchange rate for multi-currency transactions. The payment is then created in NetSuite and applied to the matching invoice.

Technical View

RF 6.0 receives per-payment parameters (customer_id, amount, subsidiary, date, stripe IDs, payment_method, fee_amount, check reference). It looks up the customer in NetSuite by external ID (Stripe cus_xxx). It determines the payment method code from the PAYMENT_METHOD field, sets AR account and cash account from the subsidiary mapping, constructs the check reference number from the charge ID, and creates a CustPymt record via SuiteTalk API. Success/failure is logged to WORKATO_LOGS with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT'.

Customer Balance Adjustment Flow (RF 8.0)

Business View

When Stripe issues a credit note (e.g., partial refund applied to customer balance), a Customer Balance Adjustment (CBA) must be created in NetSuite. This adjusts the customer's open balance without creating a separate refund transaction, keeping AR balances in sync between Stripe and NetSuite.

Technical View

Rec 8.0 queries NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS joined with NETSUITE_EXPORT_CUSTOMER (for subsidiary and NS internal ID) and WORKATO_LOGS (for dedup). For each unsynced CBA, RF 8.0 is called synchronously with entity, amount, type, currency, credit_note, invoice_id, and customer_netsuite_internal_id. RF 8.0 creates the adjustment in NetSuite.

Invoice Application

RF 6.0 applies the Customer Payment to the correct NetSuite invoice using the STRIPE_INVOICE_ID field. The recipe searches NetSuite for an open invoice matching the Stripe invoice ID and applies the payment amount against it. If no matching invoice is found, the payment is created as unapplied.
