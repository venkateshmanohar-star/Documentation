Snowflake Table Architecture — OTC Integration Pipeline

Table lineage & data flow: Fan-in DAG — Export tables pull from Fivetran RAW, STG, INT & other EXPORT tables simultaneously

Type: Architecture Author: venkatesh.manohar Generated: April 9, 2026 Database: INTERCOM_PROD Export Tables: 14 (SHARED_EXTERNAL_WORKATO) Layers: 5 (RAW, STG, INT, EXPORT, CORE) — fan-in DAG, not linear

1. Executive Summary

Business View

Intercom's Order-to-Cash (OTC) integration pipeline transforms data from two upstream source systems — Salesforce (CRM / contracts) and Stripe (billing / payments) — through Snowflake. The pipeline produces export tables in SHARED_EXTERNAL_WORKATO that Workato reads to create records in NetSuite (ERP).

Six transaction types flow through this architecture: Sales Orders, Invoices, Payments, Credit Memos, Refunds, and Customer Balance Adjustments (CBAs).

Technical View

Five logical layers exist: RAW (Fivetran-managed), STG (dbt views), INT (dbt materialized tables), EXPORT (SHARED_EXTERNAL_WORKATO), and CORE (analytics). However, data does NOT flow in a strict linear pipeline. Export tables use a fan-in DAG pattern — each export model pulls from whichever upstream layers it needs: some sources come through the full RAW → STG → INT path, others skip straight from Fivetran RAW or STG into the export table, and some export tables reference other export tables within the same schema.

All export tables are TRANSIENT BASE TABLEs tagged with OWNER_TEAM='team-enterprise-systems' and APP='dbt'. CORE analytics tables branch off from INT — they are not downstream of export tables.

Scale: The largest export table is NETSUITE_EXPORT_INVOICE at 6.7M rows (1.07 GB). The integration tracking table WORKATO_LOGS holds 67M rows (10.2 GB) across all pipelines. Total export layer: 21 tables (14 core + 7 supplementary).

2. Business Overview

What Problem Does This Solve?

Finance requires accurate, timely records in NetSuite for every transaction in the OTC cycle. Without this architecture, six separate data pipelines would need to be manually maintained, each with different source systems, transformation logic, and NetSuite record types. This unified Snowflake architecture provides:

  • Single source of truth — All export tables live in one schema (SHARED_EXTERNAL_WORKATO), queried by Workato recipes
  • Dual-purpose data — Same transformations serve both ERP integration and analytics reporting
  • Idempotent sync — WORKATO_LOGS tracks every synced record to prevent duplicates
  • Clear ownership boundaries — team-data-eng owns STG/INT/CORE; team-enterprise-systems owns EXPORT

4. Architecture Overview

Key architectural insight (from Select Star lineage): Data does NOT flow in strict sequential stages. Export tables use a fan-in DAG pattern where each dbt model pulls from whichever upstream sources it needs — Fivetran RAW, Staging views, Intermediate tables, and even other Export tables — all simultaneously. The "layers" below describe where tables live, not a mandatory sequential flow.

RAW Layer (Fivetran)

Fivetran continuously replicates data from Salesforce and Stripe into Snowflake. These are untransformed, schema-for-schema copies of the source systems. Some RAW tables feed directly into EXPORT models, bypassing STG and INT entirely (e.g., PRICEBOOK_2, ORDER_ITEM, PRICEBOOK_ENTRY for Sales Orders).

RAW Layer — Technical

Fivetran manages 232+ raw tables across 3 schemas. Salesforce raw: FIVETRAN_SALESFORCE (includes ORDER_ITEM, PRICEBOOK_2, PRICEBOOK_ENTRY, SRQQ_QUOTE_C). Stripe raw: BASE (135 tables, including STRIPE_INVOICES 4.4M rows, STRIPE_CUSTOMER 1.8M rows, STRIPE_REFUND 237K rows). Some tables replicate to STAGING_SALESFORCE (76 tables).

Staging Layer (dbt Views)

Staging views rename columns, cast types, and apply light filtering. They are live views with zero storage cost. Some staging tables are consumed directly by export models (e.g., STG_WORKATO_LOG and STG_BILLING_CUSTOMERS feed directly into NETSUITE_EXPORT_SALES_ORDER).

Staging Layer — Technical

Schemas: STAGING_SALESFORCE, STAGING_PRODUCT, STAGING_BILLING, STAGING_STRIPE_DATA_SHARE. Key views: STG_SALESFORCE_ORDER_ITEM (27 columns), STG_STRIPE_INVOICES (4.4M rows), STG_STRIPE_PAYMENTS (4.0M rows), STG_WORKATO_LOG, STG_BILLING_CUSTOMERS. Owner: team-data-eng.

Intermediate Layer (dbt Tables)

Business logic, joins, deduplication, and enrichment happen here. These are materialized tables (not views) for performance. Export tables pull from intermediate tables across multiple schemas (GTM, FINANCE, COMMON), not just from one.

Intermediate Layer — Technical

Schemas: INTERMEDIATE_GTM (INT_SALESFORCE_ORDER, INT_SALESFORCE_CONTRACT, INT_OPPORTUNITIES, INT_SALESFORCE_APP_TO_SUBSCRIPTION), INTERMEDIATE_FINANCE (INT_STRIPE_INVOICE_LINES 35.3M rows, INT_STRIPE_PAYMENTS 4.0M, INT_STRIPE_METERING_FEATURES), INTERMEDIATE_COMMON (INT_SALESFORCE_ACCOUNT, INT_SNAP_STRIPE_CUSTOMER). Owner: team-data-eng.

Export Layer (Workato)

The SHARED_EXTERNAL_WORKATO schema contains all NetSuite-ready export tables. These are the fan-in convergence point — each export model's dbt SQL joins sources from RAW, STG, INT, and even other export tables. Export tables also have downstream dependents within the same schema (e.g., NETSUITE_EXPORT_SALES_ORDER feeds NETSUITE_EXPORT_CUSTOMER and NETSUITE_EXPORT_ADJUSTMENT_RECORDS).

Export Layer — Technical

14 TRANSIENT tables in SHARED_EXTERNAL_WORKATO. All tagged APP='dbt', OWNER_TEAM='team-enterprise-systems'. TRANSIENT = 1-day Time Travel (vs 90-day default). Largest: NETSUITE_EXPORT_INVOICE (6.7M rows, 1.07 GB). Self-referencing: some export tables depend on other export tables (e.g., NETSUITE_EXPORT_PRODUCTS feeds into NETSUITE_EXPORT_SALES_ORDER).

Core Layer (Analytics)

Analytics-ready models consumed by Looker, Mode, and downstream dbt models. These branch off from the intermediate layer — they are not downstream of the export tables.

Core Layer — Technical

Schemas: CORE_GTM (98 tables), CORE_FINANCE (40 tables), CORE_COMMON (38 tables). Key tables: FCT_ORDERS (54.7K rows), FCT_INVOICES (6.8M rows, combines Stripe + Zuora). Owner: team-data-eng.

6. Sales Order Pipeline Lineage (Corrected via Select Star)

Business View

The Sales Order pipeline is the most complex, with a fan-in of 15 source tables from 4 layers. It sources from Salesforce CRM (contracts, orders, opportunities), Stripe (customer & metering data), billing metadata, and Workato integration logs — all converging into a single export table for NetSuite.

Technical View

Fan-in DAG, NOT a linear chain. The dbt model NETSUITE_EXPORT_SALES_ORDER (340K rows, 22.5 MB, 48 cols) joins directly from: 4 Fivetran RAW tables (SRQQ_QUOTE_C, PRICEBOOK_2, ORDER_ITEM, PRICEBOOK_ENTRY — these skip STG/INT entirely), 2 Staging tables (STG_WORKATO_LOG, STG_BILLING_CUSTOMERS — skip INT), 7 Intermediate tables from 3 schemas (GTM, FINANCE, COMMON), and 1 other Export table (NETSUITE_EXPORT_PRODUCTS — self-reference). Side branch: INT → FCT_ORDERS (54.7K rows, CORE_GTM).

7. Invoice Pipeline Lineage

Business View

The Invoice pipeline is the highest-volume pipeline, processing 6.7M rows from Stripe. Invoices and their 35.3M line items flow through staging and intermediate enrichment before reaching the export layer for Workato to sync to NetSuite.

Technical View

Fan-in DAG with 71 columns. Primary chain: BASE.STRIPE_INVOICES (4.4M) → STG → INT_STRIPE_INVOICES (4.4M, 297 MB), plus INT_STRIPE_INVOICE_LINES (35.3M, 2.0 GB). Additional fan-in sources: Salesforce contracts (SALESFORCE_CONTRACT_ID, CONTRACT_STATUS), Salesforce orders (SALESFORCE_ORDER_ID), Stripe subscriptions, payment intents, charges, and discount data. The export table NETSUITE_EXPORT_INVOICE (6.7M, 1.07 GB) expands from 4.4M source rows due to invoice line-level grain. V2 table: EXT_NETSUITE_EXPORT_INVOICE_V2 (6.6M, 695 MB).

8. Payment Pipeline Lineage

Business View

The Payment pipeline joins multiple Stripe sources to build a unified payment record. It handles regular charges, out-of-band payments, and bank transfer payments (ACH/wire). The export model's comment explicitly references the OTC mapping spreadsheet.

Technical View

Five source tables merge into INT_STRIPE_PAYMENTS (4.0M rows, 386 MB), then filter to 420K export rows. Grain: stripe_payment_id. Partial payments create multiple rows per invoice. Bank transfer support added 2026-03-04.

9. Supporting Pipelines (Refund, Credit Memo, CBA)

These three pipelines have shorter lineage chains, typically transforming directly from Stripe raw/staging data to the export layer without deep intermediate processing.

Refund Pipeline

Business View

When a Stripe refund is issued and linked to a NetSuite invoice, a refund record is created in NetSuite. Only 0.8% of Stripe refunds (1,926 of 237K) are exported — the high filter ratio is because most refunds are for invoices not tracked in NetSuite.

Technical View

Cross-pipeline fan-in with 18 columns. Sources: BASE.STRIPE_REFUND (237K rows, 16 MB) + Stripe charges (STRIPE_CHARGE_ID) + Stripe balance transactions (STRIPE_AUTOMATIC_PAYOUT_ID, FEE_AMOUNT) + credit notes (CREDIT_NOTE_ID). Cross-pipeline dependency: joins against NETSUITE_EXPORT_INVOICE to match only refunds with corresponding NetSuite invoices. Output: NETSUITE_EXPORT_REFUNDS (1,926 rows, 0.5 MB) — 99.2% of source rows filtered out.

Credit Memo Pipeline

Business View

Credit memos in NetSuite represent credit notes from Stripe. Two export tables exist: a header table (NETSUITE_EXPORT_CREDIT_MEMO) and a line-item table (NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES with 3,849 rows).

Technical View

Cross-pipeline fan-in with 34 columns. Sources: Stripe credit notes (STRIPE_CREDIT_NOTE_ID) + Stripe invoices (STRIPE_INVOICE_ID, STRIPE_INVOICE_STATUS) + invoice line items (STRIPE_INVOICE_LINE_ITEM) + Stripe refunds (REFUND_ID) + Stripe charges (CHARGE_ID) + Salesforce contracts (SALESFORCE_CONTRACT_ID). Two distinct credit memo paths exist: (1) Stripe-sourced credit notes via this pipeline, and (2) amendment-driven credit memos generated by the Sales Order pipeline's RF 3.0 recipe directly in NetSuite (form 114).

CBA (Customer Balance Adjustment) Pipeline

Business View

CBAs track customer credit balance adjustments from Stripe — typically from prorations, downgrades, or credit applications. These create journal entries in NetSuite to properly account for the credit balance changes.

Technical View

Cross-pipeline fan-in with 14 columns. Sources: Stripe balance transactions (EXTERNAL_ID, TYPE, ENDING_BALANCE) + credit notes (CREDIT_NOTE) + invoices (INVOICE_ID). Export: NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS (20,924 rows, 1.8 MB). Note: uses ENTITY instead of CUSTOMER for the Stripe customer ID column. Known edge case: CBAs originating from manual invoices (proration/downgrade) may not sync to NetSuite due to routing debug strings in the NS ID field preventing RF 8.0 from parsing.

12. Ownership & Governance

Ownership Transition

Ownership transitions at the EXPORT layer. All tables in RAW, STG, INT, and CORE are owned by team-data-eng. The NETSUITE_EXPORT_* tables are owned by team-enterprise-systems, verified by the dbt tag OWNER_TEAM on each table's DDL.

Storage Optimization

TRANSIENT tables as a cost strategy: All NETSUITE_EXPORT_* tables are created as TRANSIENT, meaning Snowflake does not maintain Time Travel beyond 1 day (vs 90 days for permanent tables). This is intentional — the export tables are fully rebuildable from dbt at any time. This saves significant storage costs, especially for the 1+ GB Invoice table.
