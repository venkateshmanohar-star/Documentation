Rec 1.01 - Customer Master Recipe By Customer ID

Workato Integration Documentation — NetSuite ARM Customer Sync (v3)
Recipe ID: 54665259 Status: Active / Running Author: venkatesh.manohar Created: 2024-10-10 Last Run: 2026-04-01 Generated: 2026-04-01

1. Executive Summary

Rec 1.01 - Customer Master Recipe By Customer ID is a Workato recipe function that synchronizes customer records from Snowflake to NetSuite. It is a core building block of the NetSuite ARM (Accounts Receivable Management) integration, invoked by upstream recipes whenever a customer record must exist in NetSuite before processing sales orders, invoices, or credit memos.

The recipe accepts a customer_id parameter, queries the SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER staging table in Snowflake, and for each matching row calls a child recipe function (RF 1.0 Create/Update Customer, ID: 54251831) that upserts the customer into NetSuite, manages multi-subsidiary relationships, and logs results back via Fivetran.

2. Business Overview

Purpose

Intercom's Order-to-Cash (OTC) pipeline requires every customer that will appear on a Sales Order, Invoice, or Credit Memo in NetSuite to first have a corresponding Customer record. This recipe ensures that requirement is met by creating or updating the NetSuite Customer record on-demand, driven by data staged in Snowflake from upstream Stripe, Salesforce, and billing systems.

Business Context

  • Revenue Recognition: Accurate customer records are a prerequisite for NetSuite ASC-606 revenue recognition schedules.
  • Multi-subsidiary Support: Intercom operates through multiple legal entities (US, IE). The integration automatically creates subsidiary relationships when a customer transacts across subsidiaries.
  • Billing Accuracy: Customer billing addresses, payment terms, and currency are synchronized to ensure invoices are correct.
  • Audit Trail: Every create/update is logged back to Snowflake via Workato Logs + Fivetran, maintaining a complete integration audit trail.

4. Architecture Overview

The architecture follows a hub-and-spoke pattern with Snowflake as the central data hub and Workato as the orchestration layer:

  1. Data Staging (Snowflake): Upstream dbt models aggregate customer data from Stripe, Salesforce, and billing systems into NETSUITE_EXPORT_CUSTOMER.
  2. Orchestration (Workato): A scheduled master recipe (0.0) triggers the pipeline 4x daily. Downstream recipes call Rec 1.01 on-demand when customer records are needed before processing sales orders or CBAs.
  3. Processing (Workato → NetSuite): RF 1.0 upserts the customer, manages subsidiary relationships, and logs results.
  4. Feedback Loop (Fivetran → Snowflake): Workato Logs are synced back to Snowflake via Fivetran, updating staging rows with NetSuite Internal IDs, error messages, and timestamps.
Key Design Decision: This recipe is a callable function, not a standalone scheduled recipe. It is invoked synchronously by RF 3.0 (Sales Orders) and RF 8.0 (CBAs) whenever they detect a customer that does not yet exist in NetSuite, ensuring just-in-time customer creation.

6. End-to-End Flow

Flow Narrative

  1. Scheduled Trigger: The root recipe 0.0 Scheduled Master (ID: 58628330) fires at 05:30, 10:30, 15:30, and 20:30 Pacific daily via cron.
  2. Upstream Pipeline: The master recipe calls Rec 2.0 (Items), Rec 3.0 (Sales Orders), and an async invoice/credit memo pipeline. Both Rec 3.0 and RF 8.0 (CBA) call Rec 1.01 synchronously when they need to ensure a customer exists in NetSuite.
  3. Customer Lookup: Rec 1.01 receives a customer_id and queries Snowflake: SELECT * FROM NETSUITE_EXPORT_CUSTOMER WHERE external_id = :customer_id.
  4. Row Iteration: For each row returned (typically 1), Rec 1.01 calls RF 1.0 passing all customer fields (name, address, subsidiary, Salesforce ID, billing info, terms, etc.).
  5. NetSuite Upsert: RF 1.0 upserts the customer in NetSuite using the External ID. If the customer already exists, it updates; otherwise, it creates.
  6. Subsidiary Management: RF 1.0 searches for existing subsidiary relationships. If none exists for the current subsidiary, it creates one.
  7. Logging: RF 1.0 logs the result (success or error) to Workato Logs, which Fivetran syncs back to Snowflake for auditability.
  8. Error Handling: If the master recipe (Rec 1.01) itself fails, the catch block updates the Integration Tracker lookup table with the failure timestamp.

Stage 4: RF 1.0 Processing (Child Recipe)

  1. Upsert Customer: netsuite.upsert_object creates or updates the Customer record in NetSuite using External ID as the match key.
  2. Subsidiary Logic: Custom Ruby code determines if a subsidiary relationship is needed. A search checks existing relationships; if missing, netsuite.add_object creates one.
  3. Success Logging: A mapper formats the result, and logger.log_message writes it to Workato Logs (picked up by Fivetran).
  4. Error Handling: On error, the catch block distinguishes error types via conditional logic, formats an error log, and writes it to Workato Logs with error_type and error_message.

Stage 5: Error Handling (Rec 1.01 Level)

If the overall recipe fails (e.g., Snowflake connectivity issue), the catch block:

  1. Searches the Integration Tracker lookup table (ID: 3347261) for the "Customer" object entry.
  2. Updates Last_Runtime to the current timestamp (but does NOT update Last_Successful_Runtime), creating an observable gap for monitoring.
Note: Steps 6 and 7 (LUT search and update) are currently marked as skip: true in the recipe configuration, meaning the error tracking via the lookup table is disabled. Errors at the RF 1.0 level are still logged to Workato Logs.

9. Key Business Logic

9.1 Company Name Fallback

When passing CompanyName to RF 1.0, the recipe uses conditional logic:

CompanyName = customer_name.present? ? customer_name : external_id

If the customer name is blank or null, the Stripe customer ID (external_id) is used as a fallback to ensure the NetSuite record always has a name.

9.2 Upsert Strategy

RF 1.0 uses NetSuite's upsert_object action, which matches on the External ID field. This means:

  • If a customer with that External ID already exists in NetSuite, all fields are updated.
  • If no customer exists with that External ID, a new customer record is created.
  • This is idempotent — running the same customer_id multiple times is safe.

9.3 Multi-Subsidiary Relationship Management

After upserting the customer, RF 1.0:

  1. Uses custom Ruby code to determine the subsidiary internal ID.
  2. Searches NetSuite for existing Customer Subsidiary Relationships.
  3. If the customer does not already have a relationship for the current subsidiary, creates one via netsuite.add_object.

This allows a single customer to transact across both the US and IE subsidiaries.

9.4 Integration Tracker (Error Monitoring)

The recipe's catch block references a Workato Lookup Table ("Integration Tracker- SB", ID: 3347261) to track runtimes. The design pattern is:

  • On success: Both Last_Runtime and Last_Successful_Runtime would be updated.
  • On failure: Only Last_Runtime is updated; the gap between the two timestamps signals a problem.
These lookup table steps are currently skipped in production (skip: true). Error tracking relies on Workato Logs and Fivetran-synced audit data in Snowflake.

9.5 Error Handling in RF 1.0

RF 1.0's catch block categorizes errors with conditional logic (if/else), formatting different log messages depending on the error type. Both success and error outcomes are logged to Workato Logs for Fivetran to pick up, ensuring the Snowflake staging table is always updated with the latest integration status.

11. Edge Cases

#ScenarioBehaviorRisk
1 Customer not found in Snowflake Query returns 0 rows; foreach loop does not execute; recipe completes successfully with no action. Low — but caller (RF 3.0 / RF 8.0) may fail if it expects the customer to exist in NetSuite after this call.
2 Multiple rows for same external_id Foreach iterates all rows, calling RF 1.0 for each. Since upsert is idempotent, the last row's data wins. Medium — if rows have conflicting data (e.g., different addresses), the final state may be non-deterministic.
3 Blank company name Fallback logic uses external_id as company name. Customer is created in NetSuite with Stripe ID as name. Low — customer will exist but may require manual name correction.
4 Missing required billing address fields NetSuite may reject the upsert if required address fields are null. Error is caught and logged by RF 1.0. Medium — customer will not be created; downstream SO/CBA processing will fail.
5 Subsidiary change RF 1.0 searches existing subsidiary relationships and adds new ones if needed. Does not remove old relationships. Low — subsidiary relationships are additive.
6 NetSuite API timeout/rate limit RF 1.0's catch block logs the error. The customer row's error_type and error_message are updated in Snowflake via Fivetran. Medium — retry is not automatic; requires re-trigger from upstream.
7 Snowflake connectivity failure Rec 1.01's catch block fires. Integration Tracker LUT update is currently skipped. High — no automatic retry; entire batch for that customer_id fails silently.
8 Concurrent calls for same customer_id Multiple RF 3.0/RF 8.0 jobs may call Rec 1.01 for the same customer simultaneously. NetSuite upsert is idempotent, so both succeed. Low — potential for wasted API calls but no data corruption.

12. Operational Notes

Scheduling & Frequency

  • Root recipe (0.0) runs 4 times daily: cron: 30 5,10,15,20 * * * (Pacific time).
  • Rec 1.01 is not scheduled directly; it is called on-demand by RF 3.0 and RF 8.0.
  • Typical volume: ~178K+ successful executions over the recipe's lifetime (since Oct 2024).
  • Last run: 2026-04-01 11:32 PT.

Monitoring & Alerting

  • Workato Job Reports: Custom column "Customer ID" is included in job reports for easy lookup.
  • Email Alerting: The root master recipe (0.0) sends email alerts to danny.mcgean@intercom.io, nan.lu@intercom.io, chase.clark@connorgp.com, diana.sy@intercom.io on timeout errors.
  • Snowflake Audit: Integration results (including errors) are queryable in staging_product.stg_workato_log via Fivetran sync.

Error Recovery

  • Rec 1.01 has only 2 failed jobs out of 178,481 total (99.999% success rate).
  • Individual record failures in RF 1.0 are caught and logged; they do not cause the parent recipe to fail.
  • To re-process a failed customer, re-trigger the upstream recipe (RF 3.0 or RF 8.0) which will call Rec 1.01 again.

Performance Considerations

  • Snowflake query has a 5,000-row safety limit, though a single customer_id typically returns 1 row.
  • RF 1.0 makes 2-3 NetSuite API calls per customer (upsert + subsidiary search + optional subsidiary create), so high-volume batches should account for NetSuite rate limits.
  • The recipe uses Workato's Lifetime Task Count of 355,786 tasks, indicating ~2 tasks per job (Snowflake query + RF 1.0 call).
