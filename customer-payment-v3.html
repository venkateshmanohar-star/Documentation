1. Overview

Rec 6.0 Create Customer Payment is a production Workato recipe that automates the creation of Customer Payment records in NetSuite based on Stripe payment data staged in Snowflake. It is part of Intercom's Order-to-Cash (OTC) integration pipeline, specifically handling the payment application step where Stripe charges are matched to NetSuite invoices and recorded as Customer Payments.

The recipe operates on a parent-child pattern: this parent recipe (54665264) reads unprocessed payments from Snowflake, then dispatches each payment row asynchronously to a child recipe (RF 6.0 Create/Update Customer Payment, ID 54665274) that performs the actual NetSuite create/update operations.

Business Purpose

When a customer pays a Stripe invoice, the payment data flows through Fivetran into Snowflake, where a dbt model (NETSUITE_EXPORT_PAYMENT) stages the record for NetSuite integration. This recipe picks up unprocessed payments and creates corresponding Customer Payment records in NetSuite, applying them to the correct invoices. This ensures the Accounts Receivable ledger accurately reflects cash received.

Integration Family: This recipe is part of the Rec 6.x payment integration family within the OTC mapping framework. See the OTC Mapping Spreadsheet for the full specification.

2. Architecture

The integration follows a three-tier architecture: Snowflake serves as the data source (staging layer), Workato provides the orchestration and transformation engine, and NetSuite is the target ERP system. The WORKATO_LOGS table in Snowflake closes the feedback loop by tracking which records have been successfully processed.

WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT'
        AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id
        ORDER BY job_run_completed_at DESC
    ) = 1
)

SELECT SUBSIDIARY, CUSTOMER, DATE, STRIPE_INVOICE_ID,
       STRIPE_PAYMENT_ID, STRIPE_PAYOUT_ID, PAYMENT_AMOUNT,
       PAYMENT_METHOD, CHECK_REFERENCE_NUMBER, FEE_AMOUNT
FROM SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_PAYMENT
LEFT JOIN stg_workato_log wl
    ON stripe_payment_id = wl.external_id
WHERE date BETWEEN '{date_parameter}' AND DATEADD(day, -1, CURRENT_DATE)

9. Error Handling

Parent Recipe Error Handling

The parent recipe wraps all logic in a Try/Catch block (Steps 1/6). The catch block has 0 retries with a 2-second interval, meaning errors are caught but not retried at the parent level. Individual payment failures in the child recipe do not stop the parent's for-each loop since child recipes are called asynchronously.

Child Recipe Error Handling

The child recipe (RF 6.0) implements a more nuanced error handling strategy:

Error ScenarioHandlingWORKATO_LOGS Entry
Invoice not found in NetSuite Soft error: logged with message INT_ERROR_CREATE_FAILED: Stripe Invoice does not exist in NetSuite is_successful = false, error logged
NetSuite API error (catch block) Hard error: full error message captured is_successful = false, error_message from exception
Duplicate payment (already exists) Not an error: updates existing payment with current invoice list is_successful = true, existing NS internal ID logged
Idempotency: The recipe is idempotent by design. If re-run with the same date range, it will skip payments already in WORKATO_LOGS (via the source SQL filter) or update existing NS records (via the external ID search in the child). This enables safe re-runs for catch-up processing.

12. Operational Notes

Key Operational Considerations

  • Batch Size Limit: The Snowflake query returns a maximum of 50,000 rows per execution. For high-volume periods, multiple runs may be needed.
  • Bank Transfer Exclusion: Records with check_reference_number = 'Bank Transfer Payment' are excluded from automated processing and must be handled manually.
  • Async Dispatch: The parent calls the child asynchronously, meaning the parent job completes before all child jobs finish. Monitor child recipe job counts separately.
  • Rate Limiting: The child recipe includes 1-second sleep calls between NS API operations. At high volume, this limits throughput to approximately 1 payment per second per concurrent job.
  • Currency Hardcoding: Currency is hardcoded to 1 (USD). Multi-currency payments may require manual intervention or a separate recipe.
  • Custom Form: All payments use custom form ID 115. Changes to this form in NetSuite could impact the integration.

Reprocessing / Catch-up

To reprocess payments for a specific date range, invoke the recipe function with the desired date parameter. The WORKATO_LOGS filter ensures already-processed payments are skipped. For payments that failed with soft errors (e.g., missing invoice), fix the underlying issue in NetSuite first, then either:

  1. Wait for the next scheduled run to pick up the record automatically, or
  2. Manually trigger the recipe with the appropriate date range.
