Rec 2.0 — Items Master Recipe

Workato Integration Documentation — NetSuite Inventory Item Sync
Recipe ID: 54620963 Status: Active Author: venkatesh.manohar Generated: 2026-04-01

1. Overview

The Rec 2.0 - Items Master Recipe is an orchestrator recipe responsible for synchronizing product/item records from Intercom's Snowflake data warehouse into NetSuite as Inventory Item records. It queries the NETSUITE_EXPORT_ITEM staging table for items that have not yet been synced (no NetSuite Internal ID and no last integration date), then iterates over each record and dispatches it to the child recipe RF 2.0 Create/Update Item (ID: 54620964) for individual record processing.

This recipe is part of the broader NetSuite Order-to-Cash (OTC) integration suite managed on Workato, ensuring that billing plan items from Intercom's product catalog are accurately represented in NetSuite's item master for revenue recognition, invoicing, and financial reporting.

Key Capabilities

  • Batch Discovery: Identifies unsynced items via Snowflake SQL query joining NETSUITE_EXPORT_ITEM with WORKATO_LOGS
  • Iterative Processing: Loops through each discovered item and dispatches to child recipe for NetSuite upsert
  • Idempotent Design: Only processes items where netsuite_internal_id IS NULL and last_integration_date IS NULL
  • Error Containment: Try/catch block wraps all processing to prevent recipe failure on transient errors

2. Business Context

Intercom's billing system maintains product items as billing plans. When new products are created or existing products are updated, the item records must be mirrored in NetSuite to enable:

  • Sales Order Line Items: Items referenced on sales orders must exist in the NetSuite item master
  • Invoice Generation: Stripe invoices mapped to NetSuite invoices require matching item records
  • Revenue Recognition: NetSuite's Rev Rec engine uses item classification (class codes) for proper revenue scheduling
  • Financial Reporting: Accurate item categorization ensures correct P&L and balance sheet presentation

Integration Ecosystem

This recipe sits within the Workato "Rec 2.0" family of recipes that handle the full Order-to-Cash flow:

  • Customer Master — Syncs customer/account records to NetSuite
  • Items Master (this recipe) — Syncs product items to NetSuite
  • Sales Orders — Creates sales order transactions in NetSuite
  • Invoices — Syncs Stripe invoices to NetSuite
  • Credit Memos — Handles refunds and credit adjustments

Step 0: Trigger — Recipe Function Execute

Step 1: TRY Block

Step 2: Snowflake SQL Query

Executes a custom SQL query against the Snowflake INTERCOM_PROD warehouse to identify items requiring sync:

View SQL Query 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE
        external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_ITEM'
        AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id
        ORDER BY job_run_completed_at DESC
    ) = 1
)
SELECT
    BILLING_PLAN_ID,
    product_name,
    item_code,
    netsuite_class_code
FROM
    SHARED_EXTERNAL_WORKATO.netsuite_export_item i
LEFT JOIN
    stg_workato_log wl ON i.billing_plan_id = wl.external_id
WHERE
    wl.netsuite_internal_id IS NULL
    AND last_integration_date IS NULL

Query Logic:

  • The CTE stg_workato_log gets the most recent log entry per external_id from WORKATO_LOGS, filtered to the NETSUITE_EXPORT_ITEM source table in production
  • The main query selects items from NETSUITE_EXPORT_ITEM that have no successful NetSuite ID logged AND no last integration date
  • Returns four columns: BILLING_PLAN_ID, product_name, item_code, netsuite_class_code

10. Error Handling

Master Recipe Error Strategy

MechanismConfigurationBehavior
Try/Catch BlockWraps Steps 2-5Catches all exceptions from Snowflake query and for-each processing
Max Retries0No automatic retries at the master recipe level
Retry Interval2 secondsConfigured but not used (retries = 0)
Catch Block ActionsNoneEmpty catch block — errors are silently absorbed

Idempotency Guarantees

  • Safe to Re-run: The query only selects items where netsuite_internal_id IS NULL AND last_integration_date IS NULL, so previously synced items are automatically excluded
  • External ID Matching: NetSuite uses BILLING_PLAN_ID as the external ID, enabling upsert behavior in the child recipe
  • Log-Based Deduplication: The join to WORKATO_LOGS provides an additional deduplication layer by checking if any log entry exists with a NetSuite Internal ID

Known Considerations

  • Silent Failures: The empty catch block means errors in the Snowflake query or child recipe calls are not surfaced via alerts or notifications
  • No Partial Failure Tracking: If some items in the for-each loop fail while others succeed, there is no explicit mechanism to track which items failed at the master recipe level (this is handled by the child recipe's own error logging)
  • Zero Failed Jobs: The 874 succeeded / 0 failed record suggests the try/catch effectively prevents recipe-level failures, but individual item failures may be tracked in WORKATO_LOGS
