Rec 4.0 — Invoice Master Recipe

Automated Stripe-to-NetSuite invoice creation for Self-Serve and Sales-Led customers

Recipe ID: 54665262 Status: Active Version: 59 Author: diana.sy Last Run: 2026-04-01 Success: 886 | Failed: 141 Lifetime Tasks: 2,487,130

1. Executive Summary

The Invoice Master Recipe is the central orchestrator for creating invoices in NetSuite from Stripe billing data. Every time Stripe finalizes an invoice for an Intercom customer, this recipe ensures a corresponding invoice (or credit memo) is created in NetSuite — the company's financial system of record.

What it does: Reads finalized Stripe invoices from a Snowflake staging table, determines whether the customer is on a self-serve or sales-led contract, and routes each invoice to the appropriate processing recipe that creates the correct NetSuite record.

Why it exists: Without this automation, finance teams would need to manually create thousands of invoices per month in NetSuite. This recipe ensures revenue is recognized accurately, on time, and with full audit traceability from Stripe through to the general ledger.

High-level flow:

  1. Query Snowflake for finalized Stripe invoices that have not yet been synced to NetSuite
  2. Deduplicate against the Workato log to skip invoices already processed
  3. Group invoices by Stripe Invoice ID for batch processing
  4. Route each invoice to Self-Serve or Sales-Led processing based on the sales channel
  5. Each child recipe creates the appropriate NetSuite records (Invoice, Credit Memo, or both)

2. Business Overview

The Problem

Intercom bills customers through Stripe (for self-serve) and through Salesforce-originated contracts (for sales-led). In both cases, once a Stripe invoice is finalized, a matching invoice must be created in NetSuite for revenue recognition, tax compliance, and financial reporting. This must happen reliably for hundreds of invoices daily, with correct line items, discounts, tax amounts, and customer mappings.

What Happens from a Business Perspective

Business View

  • A Stripe invoice is finalized — the customer has been charged or an invoice has been issued
  • Data lands in Snowflake — Fivetran/dbt pipelines extract and transform the Stripe invoice into a staging table (NETSUITE_EXPORT_INVOICE)
  • Workato picks it up — this recipe reads new invoices and creates corresponding NetSuite transactions
  • NetSuite invoice is created — with correct customer, line items, amounts, rev rec dates, and tax

4. Architecture Overview

Layer Model

Source Layer (Stripe)

Business: Stripe generates invoices when customers are charged for subscriptions, usage, or one-time items.

Technical: Fivetran extracts raw Stripe data into Snowflake. dbt models transform it into the NETSUITE_EXPORT_INVOICE staging table with all fields mapped to NetSuite requirements.

Data Layer (Snowflake)

Business: Acts as the "waiting room" for invoices before they're sent to NetSuite. Also stores logs of what has already been processed.

Technical: The master recipe queries NETSUITE_EXPORT_INVOICE joined with NETSUITE_EXPORT_ITEM (for NetSuite item IDs) and left-joined with WORKATO_LOGS (for deduplication). Only invoices where wl.netsuite_internal_id IS NULL are fetched — meaning they have not been successfully written yet.

Orchestration Layer (Workato)

Business: The "traffic controller" that reads invoices and sends them to the right processing path.

Technical: Rec 4.0 is a recipe function triggered by a scheduler. It loops through results, groups lines by stripe_invoice_id, and calls either RF 4.01 (Self-Serve) or RF 4.00 (Sales-Led) based on sales_channel.

Target Layer (NetSuite)

Business: The financial system where invoices become part of the official books. Revenue recognition starts here.

Technical: Records created include Invoice (standard and standalone), Credit Memo (for amendments/proration), and Customer (if missing). SuiteScript saved searches are used for existence checks.

6. End-to-End Flow

Level 0 — Simple View

A Stripe invoice is finalized. The data flows through Snowflake (staging), gets picked up by Workato (orchestration), and lands as an Invoice or Credit Memo in NetSuite (financial system). The entire process is automated and runs continuously.

Level 1 — Detailed View

  1. Extraction: The master recipe queries Snowflake to fetch all finalized invoices between a configurable start date and yesterday, filtering out zero-amount lines and invoices already synced (via WORKATO_LOGS left join).
  2. Grouping: Invoice line items are grouped by stripe_invoice_id into a deduplicated list. Multiple line items for the same Stripe invoice are batched together.
  3. Routing: Each grouped invoice is checked for sales_channel. If self-serve, it goes to RF 4.01. Otherwise (including sales-contract), it goes to RF 4.00.
  4. Processing: The child recipe handles customer lookup/creation, item mapping, invoice creation, and (for sales-led) sales order matching and credit memo generation.
  5. Logging: Every processed invoice is logged to WORKATO_LOGS with success/failure status, NetSuite internal ID, and error details.

Level 2 — Branching by Scenario

Self-Serve Path (RF 4.01)

  1. Search NetSuite for existing customer by external ID
  2. If customer missing, call Rec 1.01 to create customer
  3. Search for existing invoice (dedup check)
  4. Build invoice line items with custom Ruby code
  5. Create invoice in NetSuite (add_object: Invoice)
  6. Log result to WORKATO_LOGS

Sales-Led Path (RF 4.00)

  1. Search NetSuite for Sales Order by Salesforce Contract ID
  2. Declare variables for invoice lines, credit memo lines, and tracking
  3. Iterate through line items and classify (invoice vs. credit)
  4. Search for existing invoices on the Sales Order
  5. Apply netting logic (match new invoice amounts against existing credit memos)
  6. Create Invoice and/or Credit Memo in NetSuite
  7. Update Credit Memo references if needed
  8. If no Sales Order found, fall back to Self-Serve path (RF 4.01)
  9. If credit lines exist, call RF 5.0 to create Credit Memo
  10. Log result to WORKATO_LOGS

7. Processing Stages

Stage 1: Ingestion

Business View

The recipe reads all finalized Stripe invoices that haven't been processed yet. It looks at invoices finalized between a start date (passed as a parameter) and yesterday.

Technical View

Executes a SQL query against Snowflake joining NETSUITE_EXPORT_INVOICE with NETSUITE_EXPORT_ITEM (for NS item internal IDs) and left joining WORKATO_LOGS (filtered to successful writes for this source table). Only rows where wl.netsuite_internal_id IS NULL and amount != 0 are returned. Results ordered by invoice_finalized_at.

Stage 2: Grouping

Business View

Multiple line items from the same Stripe invoice are grouped together so they can be created as a single NetSuite invoice with multiple lines.

Technical View

A Unique_Invoice_List variable is populated with distinct stripe_invoice_id values. The forEach loop iterates over unique IDs. For each ID, matching line items are batch-inserted into a working list (Invoices) for processing.

Stage 3: Routing

Business View

Each invoice is sent to the correct processing path based on how the customer was acquired. Self-serve customers (who signed up online) follow a simpler path. Sales-led customers (with negotiated contracts) follow a more complex path involving sales order matching.

Technical View

An IF condition checks sales_channel == 'self-serve'. If true, calls RF 4.01 (recipe 54665272). Otherwise, calls RF 4.00 (recipe 54665271). The condition is evaluated on the first line item of each grouped invoice.

Stage 4: Processing

Business View

The child recipe creates the actual NetSuite records — looking up or creating the customer, building the invoice with correct line items and amounts, and handling any credits or amendments.

Technical View

Self-Serve (RF 4.01): Searches NS for customer, calls Rec 1.01 if missing. Searches for existing invoice (dedup). Uses Ruby code to build line item array. Creates Invoice via add_object.

Sales-Led (RF 4.00): Searches NS for Sales Order by SF Contract ID. Classifies lines into invoice vs credit buckets. Checks for existing invoices. Applies netting logic. Creates Invoice and/or Credit Memo. Falls back to Self-Serve path if no Sales Order found.

Stage 5: Output & Logging

Business View

Every processed invoice — whether successful or failed — is logged for audit and deduplication. Successful records include the NetSuite internal ID so they won't be reprocessed.

Technical View

A mapper step constructs a log entry with the Stripe invoice ID, NetSuite internal ID, source table name, success/error status, and timestamp. This is written to WORKATO_LOGS via a logger action. The catch block logs failures with error type and message.

9. Key Business Logic

Invoice Classification

The routing decision is based on a single field: sales_channel.

ValueMeaningPath
self-serveCustomer signed up directly online; no sales contractRF 4.01 — standalone invoice
sales-contractCustomer on a negotiated sales contract via SalesforceRF 4.00 — invoice against Sales Order

Sales Order Matching (Sales-Led)

For sales-led invoices, the recipe searches NetSuite for a Sales Order using the salesforce_contract_id from the Snowflake data. This is critical because:

  • NetSuite invoices for sales-led deals must reference the originating Sales Order for revenue recognition
  • Credit memos for amendments/prorations are matched against existing invoices on the same Sales Order
  • If no Sales Order is found, the recipe falls back to the Self-Serve path (creating a standalone invoice)

Credit Memo / Amendment Handling

When a sales-led customer modifies their subscription (upgrade, downgrade, or cancellation), the Stripe invoice may contain negative line items representing credits. The Sales-Led recipe:

  1. Separates positive (invoice) and negative (credit) line items
  2. Checks for existing invoices on the Sales Order for netting
  3. Creates an Invoice for positive amounts
  4. Creates a Credit Memo (or calls RF 5.0) for negative amounts
  5. Links the Credit Memo to the relevant Invoice

Customer Creation Logic

Before creating any invoice, the recipe checks if the customer exists in NetSuite. If not:

  1. RF 4.01 / RF 4.00 calls Rec 1.01 (Customer Master Recipe)
  2. Rec 1.01 queries NETSUITE_EXPORT_CUSTOMER for the customer details
  3. For each matching customer row, it calls RF 1.0 (Create/Update Customer) which creates or updates the NetSuite customer record with address, subsidiary, currency, and Salesforce ID

Fallback Behavior

Important: When a sales-led invoice cannot find a matching Sales Order in NetSuite, the recipe does not fail. Instead, it falls back to the Self-Serve path (RF 4.01), creating a standalone invoice. This prevents processing failures while still ensuring revenue is captured.

Why Grouping Is Needed

Snowflake returns one row per line item. A single Stripe invoice may have 5-10 line items (subscription, usage, discount, tax). These must be grouped by stripe_invoice_id so they can be created as a single NetSuite invoice with multiple lines. Without grouping, each line would create a separate invoice, causing incorrect financials.

11. Edge Cases & Special Scenarios

Missing Customer in NetSuite

What Happens

If a Stripe customer has never been synced to NetSuite, the invoice recipe detects this and automatically creates the customer first. This happens transparently — the invoice is still created in the same run.

How

The child recipe (RF 4.01 or RF 4.00) searches NetSuite for the customer by external ID. If not found, it calls Rec 1.01, which queries NETSUITE_EXPORT_CUSTOMER and calls RF 1.0 to create the customer with full address, subsidiary, and Salesforce mapping.

Missing Sales Order (Sales-Led)

What Happens

If a sales-led invoice arrives but the Sales Order hasn't been synced to NetSuite yet, the invoice is processed through the self-serve path instead. A standalone invoice is created, and the finance team can manually link it later if needed.

How

RF 4.00 searches for the Sales Order using salesforce_contract_id. If the search returns no results, the recipe falls through the else branch at the end, which calls RF 4.01 (Self-Serve) as a fallback.

Negative Amounts (Credits / Prorations)

What Happens

When a customer downgrades or receives a credit, Stripe generates an invoice with negative line items. These become Credit Memos in NetSuite, not negative invoices.

How

RF 4.00 separates line items by sign. Positive amounts create Invoice lines. Negative amounts create Credit Memo lines. If the amounts net to zero, the recipe may skip creation. RF 5.0 is called for standalone credit memo creation.

Duplicate Prevention

What Happens

If the recipe runs twice for the same date range, invoices that were already successfully synced are automatically skipped.

How

The initial SQL query left-joins with WORKATO_LOGS and only returns rows where wl.netsuite_internal_id IS NULL. This means any invoice that was successfully created (and logged) in a previous run will not appear in subsequent queries.

Zero-Amount Lines

Lines with amount = 0 are filtered out at the SQL level (amount != 0). These represent $0 line items that don't need financial records in NetSuite.

Rate Limiting / Throttling

Technical Note

The Sales-Led recipe includes sleep(1) calls between NetSuite API operations. This prevents hitting NetSuite's API rate limits, especially during high-volume processing periods.

12. Operational Notes

Deduplication

Deduplication is handled at two levels:

  1. SQL-level: The main query joins with WORKATO_LOGS to exclude already-processed invoices. The log CTE uses ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) to get only the latest log entry per invoice.
  2. Application-level: Within the forEach loop, a Unique_Invoice_List variable ensures each stripe_invoice_id is processed only once, even if multiple line items exist.

Error Handling

Every recipe uses a try/catch pattern:

  • Try block: Contains the main processing logic
  • Catch block: Logs the error to WORKATO_LOGS with error type and message, then continues to the next record
  • Individual record failures do not stop the batch — one invoice failing does not prevent other invoices from being processed

Key Limitations

  • Date range dependency: The recipe processes invoices between the input date and yesterday. If not run for several days, the backlog will process on the next run.
  • Sales Order timing: If a Sales Order hasn't been synced to NetSuite before the invoice arrives, the fallback creates a standalone invoice that may need manual reconciliation.
  • NetSuite rate limits: High-volume days may experience slower processing due to sleep(1) throttling between NS API calls.
  • Item mapping dependency: If an item is missing from NETSUITE_EXPORT_ITEM, the join returns a NULL netsuite_internal_id, which may cause the NS invoice creation to fail.

Monitoring Recommendations

  • Track the daily count of new invoices vs. successfully synced invoices
  • Alert on master recipe failures (recipe ID 54665262, job_failed_count increasing)
  • Check for orphaned standalone invoices created via the Sales-Led fallback path
