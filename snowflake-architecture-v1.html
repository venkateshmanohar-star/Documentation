Snowflake Table Architecture — OTC Integration Pipeline

Table lineage & data flow: Source Systems → Fivetran → Snowflake (STG → INT → EXPORT) → Workato → NetSuite

Type: Architecture Author: venkatesh.manohar Generated: April 9, 2026 Database: INTERCOM_PROD Export Tables: 14 (SHARED_EXTERNAL_WORKATO) Layers: 6 (RAW → STG → INT → EXPORT → CORE)

1. Executive Summary

Business View

Intercom's Order-to-Cash (OTC) integration pipeline transforms data from two upstream source systems — Salesforce (CRM / contracts) and Stripe (billing / payments) — through a multi-layer dbt pipeline in Snowflake. The pipeline produces export tables that Workato reads to create records in NetSuite (ERP).

Six transaction types flow through this architecture: Sales Orders, Invoices, Payments, Credit Memos, Refunds, and Customer Balance Adjustments (CBAs).

Technical View

Data flows through 6 layers: RAW (Fivetran-managed) → STG (dbt views) → INT (dbt materialized tables) → EXPORT (SHARED_EXTERNAL_WORKATO, Workato-readable) → CORE (analytics side branch). All export tables are TRANSIENT BASE TABLEs tagged with OWNER_TEAM='team-enterprise-systems' and APP='dbt'.

The same STG/INT layers feed two consumers: (1) export tables for Workato/NetSuite integration, and (2) CORE analytics tables for Looker/Mode.

Scale: The largest export table is NETSUITE_EXPORT_INVOICE at 6.7M rows (1.07 GB). The integration tracking table WORKATO_LOGS holds 67M rows (10.5 GB) across all pipelines. Total export layer: 14 production tables.

2. Business Overview

What Problem Does This Solve?

Finance requires accurate, timely records in NetSuite for every transaction in the OTC cycle. Without this architecture, six separate data pipelines would need to be manually maintained, each with different source systems, transformation logic, and NetSuite record types. This unified Snowflake architecture provides:

  • Single source of truth — All export tables live in one schema (SHARED_EXTERNAL_WORKATO), queried by Workato recipes
  • Dual-purpose data — Same transformations serve both ERP integration and analytics reporting
  • Idempotent sync — WORKATO_LOGS tracks every synced record to prevent duplicates
  • Clear ownership boundaries — team-data-eng owns STG/INT/CORE; team-enterprise-systems owns EXPORT

4. Architecture Overview

Layer 0 — Raw (Fivetran)

Fivetran continuously replicates data from Salesforce and Stripe into Snowflake. These are untransformed, schema-for-schema copies of the source systems. Salesforce data lands in STAGING_SALESFORCE; Stripe data lands in SNAPSHOT_STRIPE and BASE.

Layer 0 — Technical

Fivetran manages 232+ raw tables across 3 schemas. Key raw tables: BASE.STRIPE_INVOICES (4.4M rows, 448 MB), BASE.STRIPE_CUSTOMER (1.8M rows, 99 MB), BASE.STRIPE_REFUND (237K rows, 16 MB). Salesforce tables are in STAGING_SALESFORCE (76 tables).

Layer 1 — Staging (dbt Views)

Staging views rename columns, cast types, and apply light filtering. They are live views with zero storage cost — always in sync with the raw data. These are the "clean interface" to raw data.

Layer 1 — Technical

Schemas: STAGING_SALESFORCE, STAGING_PRODUCT, STAGING_BILLING, STAGING_STRIPE_DATA_SHARE. Key views: STG_SALESFORCE_ORDER_ITEM (27 columns), STG_STRIPE_INVOICES (4.4M rows), STG_STRIPE_PAYMENTS (4.0M rows). Owner: team-data-eng.

Layer 2 — Intermediate (dbt Tables)

Business logic, joins, deduplication, and enrichment happen here. These are materialized tables (not views) for performance. The intermediate layer is shared by both the export path (to NetSuite) and the analytics path (to CORE).

Layer 2 — Technical

Schemas: INTERMEDIATE_GTM, INTERMEDIATE_FINANCE, INTERMEDIATE_ACCOUNTING. Largest: INT_STRIPE_INVOICE_LINES (35.3M rows, 2.0 GB), INT_STRIPE_PAYMENTS (4.0M rows, 386 MB). Owner: team-data-eng.

Layer 3 — Export (Workato)

The SHARED_EXTERNAL_WORKATO schema contains all NetSuite-ready export tables. These are the tables that Workato recipes query to create/update ERP records. Each table maps to a specific NetSuite record type.

Layer 3 — Technical

12 TRANSIENT tables in SHARED_EXTERNAL_WORKATO. All tagged APP='dbt', OWNER_TEAM='team-enterprise-systems'. TRANSIENT = 1-day Time Travel (vs 90-day default) to save storage. Largest: NETSUITE_EXPORT_INVOICE (6.7M rows, 1.07 GB).

Layer 4 — Core (Analytics)

Analytics-ready models consumed by Looker, Mode, and downstream dbt models. These branch off from the intermediate layer — they are not downstream of the export tables.

Layer 4 — Technical

Schemas: CORE_GTM (98 tables), CORE_FINANCE (40 tables), CORE_COMMON (38 tables). Key tables: FCT_ORDERS (54.7K rows), FCT_INVOICES (6.8M rows, combines Stripe + Zuora). Owner: team-data-eng.

8. Payment Pipeline Lineage

Business View

The Payment pipeline joins multiple Stripe sources to build a unified payment record. It handles regular charges, out-of-band payments, and bank transfer payments (ACH/wire). The export model's comment explicitly references the OTC mapping spreadsheet.

Technical View

Five source tables merge into INT_STRIPE_PAYMENTS (4.0M rows, 386 MB), then filter to 420K export rows. Grain: stripe_payment_id. Partial payments create multiple rows per invoice. Bank transfer support added 2026-03-04.

9. Supporting Pipelines (Refund, Credit Memo, CBA)

These three pipelines have shorter lineage chains, typically transforming directly from Stripe raw/staging data to the export layer without deep intermediate processing.

Refund Pipeline

Business View

When a Stripe refund is issued and linked to a NetSuite invoice, a refund record is created in NetSuite. Only 0.8% of Stripe refunds (1,926 of 237K) are exported — the high filter ratio is because most refunds are for invoices not tracked in NetSuite.

Technical View

Direct transform: BASE.STRIPE_REFUND (237K rows, 16 MB) → NETSUITE_EXPORT_REFUNDS (1,926 rows, 0.5 MB). No intermediate table. Filtering joins against NETSUITE_EXPORT_INVOICE to match only refunds with corresponding NetSuite invoices.

Credit Memo Pipeline

Business View

Credit memos in NetSuite represent credit notes from Stripe. Two export tables exist: a header table (NETSUITE_EXPORT_CREDIT_MEMO) and a line-item table (NETSUITE_EXPORT_CREDIT_MEMO_CREDIT_NOTES with 3,849 rows).

Technical View

Source: Stripe invoice credit notes. Two distinct credit memo paths exist: (1) Stripe-sourced credit notes via this pipeline, and (2) amendment-driven credit memos generated by the Sales Order pipeline's RF 3.0 recipe directly in NetSuite (form 114).

CBA (Customer Balance Adjustment) Pipeline

Business View

CBAs track customer credit balance adjustments from Stripe — typically from prorations, downgrades, or credit applications. These create journal entries in NetSuite to properly account for the credit balance changes.

Technical View

Source: Stripe balance transactions. Export: NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS (20,924 rows, 1.8 MB). Known edge case: CBAs originating from manual invoices (proration/downgrade) may not sync to NetSuite due to routing debug strings in the NS ID field preventing RF 8.0 from parsing.

12. Ownership & Governance

Ownership Transition

Ownership transitions at the EXPORT layer. All tables in RAW, STG, INT, and CORE are owned by team-data-eng. The NETSUITE_EXPORT_* tables are owned by team-enterprise-systems, verified by the dbt tag OWNER_TEAM on each table's DDL.

Storage Optimization

TRANSIENT tables as a cost strategy: All NETSUITE_EXPORT_* tables are created as TRANSIENT, meaning Snowflake does not maintain Time Travel beyond 1 day (vs 90 days for permanent tables). This is intentional — the export tables are fully rebuildable from dbt at any time. This saves significant storage costs, especially for the 1+ GB Invoice table.
