1. EXECUTIVE SUMMARY

1. Executive Summary

The Adhoc Payroll Report integration enables Finance/Payroll teams to trigger an on-demand payroll general-ledger extract from Workday via a Slack mention, and have the resulting data forwarded to the downstream Payroll-WD-Netsuite-v2 recipe which creates NetSuite Journal Entries. This is the ad-hoc companion to the scheduled payroll integration, used for re-runs, corrections, or one-off payroll postings outside the standard pay cycle.

2. BUSINESS OVERVIEW

2. Business Overview

Business Purpose

Intercom's Payroll team uses Workday as the system of record for payroll processing. After each pay run, the payroll general ledger data must be posted to NetSuite as journal entries for financial reporting and reconciliation. While the standard flow runs on a scheduled basis, certain scenarios require ad-hoc re-extraction:

  • Payroll corrections -- retroactive adjustments or off-cycle payments
  • Re-runs -- when the scheduled run missed data or failed
  • Specific pay group extractions -- pulling data for a single pay group (e.g., US Salaried, Ireland Hourly)
  • Audit requests -- re-posting for a specific payment date

Business Trigger

A Finance/Payroll team member mentions the Workato bot in a designated Slack channel with a structured message containing:

  • Start-Date -- the payment date to extract (used for both start and end date)
  • Report Type -- the pay run group/type identifier (e.g., "US Salaried", "Ireland Monthly")

Business Outcome

The Workday GL report data is extracted, wrapped in a SOAP envelope, and forwarded to the downstream Payroll-WD-Netsuite-v2 recipe, which parses the XML, maps GL accounts to NetSuite internal IDs via a lookup table, and creates journal entries in NetSuite.

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

The integration follows a trigger-extract-forward pattern. The Adhoc Payroll Report recipe acts as a lightweight orchestrator that:

  1. Receives a Slack @mention event with structured parameters
  2. Parses the Start-Date and Report Type from the Slack message using regex
  3. Resolves the Report Type to a Workday WID using a project-level lookup table
  4. Calls the Workday custom report API to extract GL data in XML format
  5. Validates whether the report returned data or is empty
  6. Forwards the XML payload (wrapped in a SOAP envelope) to the Payroll-WD-Netsuite-v2 recipe via an internal webhook

The downstream recipe then handles the heavy lifting: XML parsing, data mapping via lookup tables, and NetSuite journal entry creation.

Design Pattern: This is a fan-out pattern where one trigger recipe (adhoc or scheduled) feeds into a shared processing recipe. This avoids duplicating the complex NetSuite mapping logic.

6. END-TO-END FLOW

6. End-to-End Flow

Flow Narrative

  1. Slack Trigger: A Payroll team member @mentions the Workato bot in the designated Slack channel. The message contains structured fields: *Start-Date*\n2026-03-15 and *Report Type*\nUS Salaried.
  2. Variable Extraction: The recipe parses the Slack message text using regex patterns to extract Start_Date and Report Type.
  3. WID Lookup: The Report Type string (e.g., "US Salaried") is mapped to a Workday WID using the Workday_Payroll_Netsuite_Lookup project property lookup table (filtered by Identification = 'Report_type').
  4. Workday API Call: An HTTP GET request is sent to the Workday custom report endpoint PAY_General_Ledger_by_Payment_Date_-_W_Pay_Component_-_WD_NS, passing the WID and payment date. Response is in XML format.
  5. Empty Report Check: The recipe compares the raw XML body to the known empty-report XML string. If empty, a danger-colored Slack alert is posted to #strategic-updates (channel C0ABDPEK2DV) with instructions to check Workday or rerun.
  6. Log and Forward: If data exists, the XML is cleaned (XML declaration stripped), logged, wrapped in a SOAP env:Envelope/env:Body, and POSTed to the Payroll-WD-Netsuite-v2 webhook endpoint.
  7. Downstream Processing: The Payroll-WD-Netsuite-v2 recipe receives the SOAP payload, parses the XML, maps GL data to NetSuite fields using the Payroll-Netsuite-Lookup table, and creates journal entries in NetSuite.

9. KEY BUSINESS LOGIC

9. Key Business Logic

9.1 Slack Message Parsing

The recipe relies on a specific Slack message format. The user must format their message with Slack bold markers (*field*) followed by a newline and the value:

*Start-Date*
2026-03-15
*Report Type*
US Salaried

Regex patterns extract these values:

  • /\*Start-Date\*\n([^\n]+)/ -- captures the date string
  • /\*Report Type\*\n([^\n]+)/ -- captures the report type string
Important: If the Slack message does not match the expected format (missing bold markers, wrong field names, extra whitespace), the regex will fail with no method '[]' for nil, which has been the most common failure mode (3 of 5 failures).

9.2 Report Type to WID Mapping

The recipe uses a project-level lookup table (Workday_Payroll_Netsuite_Lookup) to resolve human-readable report type names to Workday WIDs. The lookup is filtered by:

  • Identification = 'Report_type'
  • Key = {user-provided Report Type}

The Values field contains the corresponding Workday WID used in the API call.

9.3 Same-Day Date Range

Both Payment_Date__Start_ and Payment_Date__End_ are set to the same value (Start_Date), meaning the report always pulls data for a single payment date. This is by design for ad-hoc extraction of specific pay runs.

9.4 Empty Report Detection

The recipe performs an exact string comparison against the known empty XML response. If matched, the user is alerted via Slack with an error-colored message. Otherwise, processing continues.

9.5 XML Cleanup and SOAP Wrapping

Before forwarding, the XML declaration is stripped using .gsub(), then the cleaned XML is wrapped in a SOAP envelope:

<?xml version="1.0" encoding="UTF-8"?>
<env:Envelope xmlns:env="http://schemas.xmlsoap.org/soap/envelope/">
  <env:Body>
    {cleaned_report_xml}
  </env:Body>
</env:Envelope>

9.6 Downstream NetSuite Mapping (Payroll-WD-Netsuite-v2)

The downstream recipe performs the heavy lifting with 31 processing steps:

  • XML Parsing: Extracts Report_Entry elements containing GL line items
  • List Processing: Builds Workday-Report-Input and Netsuite-Input lists
  • Lookup Table Mapping: Maps Workday cost centers, departments, locations, subsidiaries, accounts, and classes to NetSuite internal IDs
  • Journal Entry Creation: Creates NetSuite journal entries with debit/credit lines, custom fields, and proper GL coding
  • Slack Notification: Posts success/failure alerts

11. EDGE CASES

11. Edge Cases and Error Handling

Known Failure Modes

Error TypeFrequencyRoot CauseResolution
Regex Parse Failure 3 of 5 failures no method '[]' for nil -- Slack message format mismatch Ensure message uses exact *Start-Date* and *Report Type* bold headers
Channel Not Found 1 of 5 failures channel_not_found for the Slack notification channel Verify channel ID C0ABDPEK2DV exists and bot has access
Base URL Restriction 1 of 5 failures REST connector only allows services1.myworkday.com base URL Fixed by using rest_secondary connector for webhook POST

Edge Cases

  • Empty Workday Report: Handled explicitly with Slack alert. Occurs when the payment date has no GL data.
  • Stale Lookup Table: If Pull_Netsuite_Account fails, the downstream recipe may fail on unmapped accounts.
  • Duplicate Runs: No deduplication logic exists. Running the same date/type combination multiple times will create duplicate journal entries in NetSuite.
  • Large Payroll Reports: The 3600-second timeout accommodates large reports, but very large XML payloads could hit Workato payload size limits.
  • Date Format: The Start_Date is parsed as a date type. Non-standard date formats may cause silent failures.
  • Concurrent Runs: Multiple ad-hoc triggers fired simultaneously could cause race conditions in the downstream recipe.
Risk: There is no guard against duplicate journal entry creation. If a user triggers the same date/type combination multiple times, duplicate JEs will be created in NetSuite.
