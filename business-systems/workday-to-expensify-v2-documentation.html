1 Executive Summary

This integration automates the daily synchronization of employee records from Workday (Intercom's Human Capital Management system) into Expensify (the corporate expense reimbursement platform). Running as a parent-child recipe pattern, it extracts all employees from a Workday custom report, partitions them by country, resolves the correct Expensify expense policy per country, and upserts each country-batch into Expensify via its Integration Server API.

2 Business Overview

Business Purpose

Intercom employees across all global offices need to submit expense reports through Expensify. For this to work, each employee must have an up-to-date Expensify profile mapped to the correct country-specific expense reimbursement policy. This integration ensures:

  • New hires are automatically provisioned in Expensify on their hire date or shortly after, with the correct country policy.
  • Existing employees have their profile data (name, manager, status) kept current as Workday records change.
  • Terminated employees are flagged (isTerminated = true) so Expensify can restrict further expense submissions.
  • Manager approvals are mapped correctly, with the final fallback approver always being accountspayable@intercom.io.

Business Stakeholders

  • Finance / Accounts Payable — Rely on accurate employee-to-policy mapping for expense report routing and approval.
  • People Operations / HR — Source of truth in Workday for employee lifecycle events.
  • All Employees — End-users who submit expense reports in Expensify.
  • Enterprise Systems / Integrations — Owns and monitors the automation.

Business Impact of Failure

If this integration fails, new hires cannot submit expenses, terminated employees may retain access, and manager approval chains become stale. Finance would need to manually provision Expensify accounts, which is error-prone at scale across multiple country subsidiaries.

4 Architecture Overview

The integration follows a parent-child asynchronous pattern:

  • Parent Recipe (59113812) — Orchestrates the full flow: pulls all employees from Workday, creates an in-memory Smart List collection, extracts distinct countries, and then iterates over each country. For each country, it queries employees belonging to that country (plus their cross-country managers), constructs the Account_Type variable, looks up the Expensify Policy ID, and calls the child recipe asynchronously with the employee batch.
  • Child Recipe (59113811) — Receives a list of employees and a country, then POSTs the batch to the Expensify Integration Server API to upsert employee records. On error, it publishes to the AOF_Topic Pub/Sub topic for centralized error handling.

Key Design Decisions

  • Full daily load (not incremental) ensures all changes -- including retroactive corrections -- are captured.
  • Country-based partitioning ensures each Expensify policy receives only the employees that belong to it.
  • Async child calls allow parallel processing of country batches without blocking the parent.
  • Smart List with SQL joins enables manager-country lookup via self-join on employee email.
  • Email domain normalization converts hq.intercom.com to intercom.io for legacy address compatibility.

6 End-to-End Flow

1

Scheduled Trigger (Daily 1:00 AM PT)

The Workato Clock trigger fires once per day at 1:00 AM Pacific time, initiating the parent recipe.

2

Pull Workday Report

The parent calls Workday's custom report API endpoint Workato_Integration_Report_-_Navan_All_EE_s to retrieve all employee records in JSON format. Fields include: email, name (preferred + legal), employee ID, active status, hire/termination dates, cost center, location, country, manager email, job level, business title, NetSuite subsidiary, base pay currency, and cost allocation segment.

3

Parse and Store in Smart List

The raw JSON payload is parsed via the JSON Parser step, then loaded into a Workato Smart List named Employee_Collection with Employee_ID as the primary index. The overwrite flag is set to true, so each run replaces the entire collection with fresh data.

4

Extract Distinct Countries

A SQL query runs against the Smart List: SELECT DISTINCT(Primary_Work_Address_-_Country) FROM Employee_Collection. This yields the unique set of countries across all employees.

5

For Each Country (Loop)

The parent iterates over each distinct country. Within each iteration:

  • Country Employee Query: A SQL query with a LEFT JOIN retrieves employees whose Primary_Work_Address_-_Country matches the current country, plus any managers from other countries who supervise employees in this country.
  • Set Account_Type: The variable is set to {Country} - Expense Reimbursement.
  • Lookup PolicyID: The Workato lookup table Expensify Mapping - PolicyID is queried using the Account_Type to retrieve the correct Expensify Policy ID for that country.
6

Call Child Recipe (Async)

The parent invokes the child recipe (ID 59113811) asynchronously, passing the employee list with mapped fields: employeeID, employeeEmail, managerEmail, policyID, firstName, lastName, isTerminated, customField1 (full preferred name), customField2 (employee ID), and approvesTo (accountspayable@intercom.io).

7

Child: POST to Expensify API

The child recipe POSTs to https://integrations.expensify.com/Integration-Server/ExpensifyIntegrations with the employee batch as form-encoded data. The request type is update with entity type employees. Authentication uses a service account (aa_expensify_workato_workday_service_account_intercom_io) and a secret stored in Workato account properties.

8

Error Handling and Notification

On success, Expensify sends an onFinish email to the configured notification recipient. On failure, the child's try/catch block publishes an error message to the AOF | PS-001 | AOF_Topic Pub/Sub topic for centralized alerting.

9 Key Business Logic

9.1 Country-Based Employee Partitioning

Employees are grouped by their Primary_Work_Address_-_Country. Each country group maps to a single Expensify expense reimbursement policy. The variable Account_Type is constructed as "{Country} - Expense Reimbursement" and used to look up the PolicyID.

9.2 Manager-Employee Cross-Country Join

A critical SQL query uses a LEFT JOIN on the Smart List to include managers who are in a different country than their direct reports. The join matches e.Manager_Email = m.Email, pulling the manager's country into the result set.

SELECT e.Email, e.Employee_ID, e.Preferred_FirstName, e.Legal_FirstName,
       e.Preferred_LastName, e.Legal_LastName, e.Country,
       e.Manager_Email, m.Country AS Manager_Country, e.Active_Status
FROM Employee_Collection e
LEFT JOIN Employee_Collection m ON e.Manager_Email = m.Email
WHERE e.Country = '{current_country}'
   OR e.Email IN (SELECT Manager_Email FROM Employee_Collection
                  WHERE Country = '{current_country}')

The OR clause also includes managers whose direct reports are in the current country, even if the manager themselves is based elsewhere.

9.3 Email Domain Normalization

Some Workday records contain the legacy domain hq.intercom.com. The recipe normalizes these to intercom.io:

email.include?("hq.intercom.com")
  ? email.gsub("hq.intercom.com", "intercom.io")
  : email

This logic applies to both employee email and manager email fields.

9.4 Name Fallback Logic

Preferred names take priority, falling back to legal names when preferred names are blank:

firstName = Preferred_FirstName.blank? ? Legal_FirstName : Preferred_FirstName
lastName  = Preferred_LastName.blank?  ? Legal_LastName  : Preferred_LastName

9.5 Termination Flag

The Active_Status field from Workday is checked: if it contains "0", the employee is marked as isTerminated = true. This instructs Expensify to deactivate the employee's expense submission privileges.

9.6 Approval Chain

All employees have their approvesTo field set to accountspayable@intercom.io, meaning the final approval authority for all expense reports is Accounts Payable. Manager email is also passed but serves as the direct approver before AP.

9.7 Custom Fields

  • customField1 = Full preferred name ("{FirstName} {LastName}")
  • customField2 = Employee ID from Workday

9.8 Expensify API Authentication

The child recipe authenticates to Expensify using:

  • partnerUserID: aa_expensify_workato_workday_service_account_intercom_io
  • partnerUserSecret: Stored in Workato account property expensify_user_secret

11 Edge Cases

⚠
Missing Preferred Name: If an employee has no preferred first or last name in Workday, the recipe falls back to the legal name. If both are blank, the employee will be created with an empty name field in Expensify.
⚠
New Country Not in Lookup Table: If Intercom hires in a country not yet mapped in the Expensify Mapping - PolicyID lookup table, the PolicyID lookup returns nil. The child recipe will send a nil PolicyID to Expensify, which may cause the upsert to fail or assign the employee to a default policy.
⚠
Legacy Email Domain: Employees with hq.intercom.com email addresses are normalized to intercom.io. If Workday contains other non-standard domains, they will pass through unchanged.
⚠
Manager Not in Workday: If a manager's email does not exist in the Employee_Collection (e.g., external contractor), the LEFT JOIN will return NULL for Manager_Country. The employee is still included in the batch.
ⓘ
Cross-Country Managers: Managers in a different country than their direct reports appear in multiple country batches (once for their own country, and once for each country where they have direct reports). This means managers may be upserted multiple times per run, but the upsert is idempotent.
ⓘ
Full Daily Overwrite: The Smart List is fully overwritten every run (force: true). If the Workday report returns fewer records than expected (e.g., API issue), the Smart List will contain fewer employees, and Expensify may not receive updates for missing employees. However, existing Expensify records are not deleted -- they are only upserted.
⚠
Expensify API Rate Limits: Since country batches are processed asynchronously via child calls, multiple POST requests may hit Expensify simultaneously. If Expensify applies rate limiting, some batches may fail. The try/catch in the child handles this by publishing to the error topic.

Monitoring and Alerting

  • Expensify onFinish Email: Each successful batch triggers an email to the configured Notification_Recipient account property.
  • AOF Error Topic: Child recipe errors are published to the AOF | PS-001 | AOF_Topic Pub/Sub topic, which feeds into the centralized Automation Operations Framework (AOF) error handling pipeline.
  • Workato Job History: Both parent and child job histories are available in the Workato console for debugging.

Error Handling Strategy

  • The child recipe wraps the Expensify API call in a try/catch block with 0 retries and a 2-second retry interval.
  • On catch, the error is published to the AOF error topic with full context: recipe ID, job ID, parent job ID, error message, stack trace, and a transaction description including the country.
  • The error topic message includes "trigger_async": true, indicating the error originated from an asynchronous child call.
