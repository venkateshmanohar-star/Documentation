2. BUSINESS OVERVIEW

2. Business Overview

Business Context

Intercom uses Navan as its corporate travel and expense management platform. For Navan to correctly enforce travel policies, manage approvals, and route expenses, it must maintain an accurate employee directory that mirrors the source of truth in Workday. This integration automates that synchronization, eliminating manual data entry and ensuring policy compliance.

Key Business Outcomes

  • Policy Compliance: Travel policy levels (e.g., Executive, Manager, Individual Contributor) are automatically assigned based on Workday job level, ensuring correct spending limits and approval workflows in Navan.
  • Employee Lifecycle: New hires are automatically provisioned in Navan; terminated employees are automatically disabled, preventing unauthorized travel bookings.
  • Cost Center Tracking: Cost allocation segments, cost centers, and subsidiary data flow through to Navan for accurate financial reporting and expense categorization.
  • Manager Hierarchy: Manager email addresses are synced so that Navan approval chains match the organizational structure in Workday.

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

The integration follows a straightforward Extract → Transform → Load pattern with a file-based delivery mechanism. Data is extracted from Workday as JSON via a custom report API, transformed and mapped into a CSV format that conforms to Navan's employee import schema, and then uploaded to Navan's SFTP server for ingestion.

Pattern: Scheduled ETL with SFTP file delivery. Single-recipe, no child recipes. Error handling via AOF (Automated Operations Framework) Pub/Sub topic with retry logic.

Design Principles

  • Full Snapshot: Every run extracts the complete employee roster (not incremental), ensuring Navan always receives a complete and accurate picture.
  • Idempotent Delivery: Each daily file has a date-stamped name (Intercom_MMDDYYYY.csv), making each upload distinct and traceable.
  • Fail-Safe Error Handling: Two separate try/catch blocks cover the Workday extraction and the SFTP upload independently, with AOF notifications for both.
  • Lookup-Driven Policy: Travel policy levels are derived from a Workato lookup table, allowing changes without modifying recipe logic.

6. END-TO-END FLOW

6. End-to-End Flow

Flow Summary

  1. Trigger: Workato scheduler fires at 8:15 PM PT every day.
  2. Extract: The recipe calls Workday's custom report API (Workato_Integration_Report_-_Navan) and retrieves all employee records as JSON. If the call fails, it retries up to 3 times before publishing an error to the AOF topic and stopping.
  3. Parse: The raw JSON response is parsed using the JSON Parser action to produce a structured array of employee objects.
  4. Transform: Each employee record is mapped to a 13-column CSV format. Key transformations include:
    • Preferred name falls back to legal name if blank (truncated to 30 characters)
    • enabled flag is computed from termination/hire dates
    • Policy level is looked up from the Policylevel-Navan table based on Workday job level
  5. Generate CSV: The CSV Parser connector creates a CSV file with headers and quoted fields.
  6. Upload: The CSV file is uploaded to Navan's SFTP server with filename Intercom_MMDDYYYY.csv. If upload fails, an error is published to AOF.
  7. Notify: A success email is sent to enterprise.integrations@intercom.io with the uploaded filename.

9. KEY BUSINESS LOGIC

9. Key Business Logic

9.1 Name Resolution (Preferred vs. Legal)

Rule: Use the employee's preferred name. If preferred name is blank, fall back to legal name. All names are truncated to a maximum of 30 characters.

# First Name Logic (pseudocode):
IF Preferred_Name_First_Name IS BLANK
  THEN Legal_Name_First_Name.slice(0, 30)
  ELSE Preferred_Name_First_Name.slice(0, 30)

# Last Name Logic (same pattern):
IF Preferred_Name_Last_Name IS BLANK
  THEN Legal_Name_Last_Name.slice(0, 30)
  ELSE Preferred_Name_Last_Name.slice(0, 30)

9.2 Employee Enabled/Disabled Flag

Rule: An employee is marked enabled = true unless they have a non-blank termination date where the hire date is before the termination date AND the termination date is on or before today.

# Enabled Logic (pseudocode):
IF termination_date IS BLANK
  THEN enabled = TRUE
ELSE
  IF Hire_Date < termination_date AND termination_date <= TODAY
    THEN enabled = FALSE
  ELSE
    enabled = TRUE

This logic handles edge cases such as:

  • Future-dated terminations (employee still enabled until termination date arrives)
  • Rehired employees (hire date after termination date keeps them enabled)
  • Employees with no termination date (always enabled)

9.3 Travel Policy Level Mapping

Rule: The employee's Workday job level (e.g., E7, M5, IC3) is looked up in the Policylevel-Navan Workato lookup table to determine their Navan travel policy level.

# Policy Level Logic:
policyLevel = LOOKUP('Policylevel-Navan',
                      key: 'Joblevel',
                      value: Job_Level_Primary_Position)['Policylevel']

This allows travel policy assignments to be updated centrally in the lookup table without modifying the recipe itself. Common mappings typically include Executive, Manager, and Individual Contributor tiers.

9.4 Region / Company Office

Both the region and companyOffice CSV columns are populated from the same Workday field: location (e.g., "US CA San Francisco"). This means both fields will contain the full Workday location string.

9.5 Department Mapping

The department CSV column is mapped from Workday's Cost_Allocation_Segment (e.g., "Core", "Growth"), not from the organizational department. The costCenter column uses the actual Cost_Center value (e.g., "1403 Recruiting").

11. EDGE CASES

11. Edge Cases

Handled Edge Cases

Future-dated termination Employee remains enabled = true until termination date arrives Employee can still book travel until their last day
Rehired employee (hire date after termination date) enabled = true because hire_date > termination_date condition is not met Rehired employee is correctly re-enabled

SFTP upload failure No retries; immediate AOF error notification No file delivered to Navan; team notified
Job level not in lookup table Lookup returns blank/null for policyLevel Employee synced with empty policy level (Navan may apply default)

Potential Risk Areas

Full Snapshot Risk: Because this is a full-file replacement (not incremental), if the Workday report returns an empty or partial dataset due to an upstream issue, Navan could disable all employees. There is no row-count validation or threshold check before upload.

No Duplicate Detection: If the Workday report contains duplicate employee IDs or email addresses, both rows will be included in the CSV. Navan's import behavior for duplicates depends on its own deduplication logic.

SFTP No-Retry: The SFTP upload try/catch has 0 retries configured. A transient network issue will cause the upload to fail without any retry attempt, though the AOF notification will trigger.
