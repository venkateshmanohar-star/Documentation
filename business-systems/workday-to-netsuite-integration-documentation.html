1. EXECUTIVE SUMMARY

1. Executive Summary

This Workato recipe automates the daily synchronization of employee data from Workday (Intercom's HR system of record) to NetSuite (Intercom's ERP/financial system). The integration runs every morning at 6:00 AM Pacific Time, pulling a custom Workday report containing all active and terminated employee records, transforming the data through lookup-based field mappings (department, class, location, subsidiary, currency), and upserting employee records into NetSuite in batch mode.

Key Value: Eliminates manual employee data entry in NetSuite, ensures consistent employee attributes across HR and Finance systems, and enables accurate expense reporting, approval routing, and financial reporting by subsidiary.

2. BUSINESS OVERVIEW

2. Business Overview

Business Problem

Intercom employs hundreds of people across multiple geographies and legal entities. The Finance team needs accurate, up-to-date employee records in NetSuite to support expense management, approval workflows, financial consolidation by subsidiary, and regulatory reporting. Without automation, the People Operations team would need to manually replicate every hire, termination, transfer, and title change from Workday into NetSuite -- a process that is error-prone and does not scale.

Solution

This integration acts as a nightly data pipeline that:

  • Extracts the complete employee roster from a custom Workday report
  • Transforms HR attributes (cost center, location code, currency) into NetSuite-compatible internal IDs via a centralized lookup table
  • Handles special subsidiary routing for EMEA EOR/Contractor employees in the United Kingdom
  • Upserts employee records in NetSuite using the Workday Employee ID as the external ID (idempotent -- safe to re-run)
  • Captures and emails errors to the Enterprise Integrations team for rapid triage

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

The integration follows a classic Extract-Transform-Load (ETL) pattern orchestrated by Workato:

  1. Extract: A scheduled trigger fires daily at 6:00 AM PT and pulls the full employee dataset from a Workday custom report via REST API.
  2. Transform: The raw JSON is parsed, loaded into a Workato Smart List (in-memory database), and queried via SQL to join employee records with manager records (self-join on Employee_ID). Field values are mapped from Workday text labels to NetSuite internal IDs using a centralized Data_Mapping_Lookup table.
  3. Load: Transformed records are batch-upserted into NetSuite's Employee record type in groups, keyed on the Workday Employee ID as the external ID. A conditional branch handles EMEA EOR/Contractor employees in the UK with a hardcoded location override.
  4. Notify: Errors from failed upserts and exceptions are collected into lists and emailed to the Enterprise Integrations team.

6. END-TO-END FLOW

6. End-to-End Flow

Flow Summary

  1. 6:00 AM PT -- Scheduler triggers the recipe.
  2. Three empty Workato lists are initialized: ErrorList, Change-Subsidiary, and Sampling-Employee-Netsuite.
  3. The Workday custom report (Netsuite_Employee_Reporting) is fetched via REST API, returning the full employee roster as JSON.
  4. The JSON payload is parsed and loaded into a Employee_Collection Smart List (in-memory database).
  5. A SQL self-join query is executed against the Smart List to match each employee with their manager's record (joining on Manager Email = Employee Email), producing a denormalized dataset.
  6. The result set is iterated in batch mode using a For-Each loop.
  7. For each batch, a Try/Catch block wraps the upsert logic.
  8. An exception filter skips specific executive email addresses (e.g., eoghan@intercom.io, katherine.fischer@intercom.io, etc.).
  9. Subsidiary routing: If the employee's location is "EMEA - EOR/Contractor" AND country is "United Kingdom", the NetSuite location field is hardcoded to internal ID 15. Otherwise, all fields are dynamically mapped via the Data_Mapping_Lookup table.
  10. The NetSuite batch upsert creates or updates Employee records keyed on external_id (= Workday Employee_ID).
  11. Successfully upserted records are sampled into the Sampling-Employee-Netsuite list for audit/logging.
  12. Failed records are captured into the ErrorList with the external ID, error description, and email.
  13. If any errors exist in ErrorList, a summary email is sent to enterprise.integrations@intercom.io and the team.
  14. If a batch-level exception occurs (caught by Try/Catch), an immediate error email is sent to diana.sy@intercom.io.
  15. Final logging captures the full ErrorList and Sampling list for the job report.

7. Processing Stages

Stage 1: Initialization (Steps 1-3)

Stage 2: Data Extraction (Step 4)

The Workday custom report is fetched via the URL:

https://services1.myworkday.com/ccx/service/customreport2/intercom/ISU_Workato/Netsuite_Employee_Reporting?format=json

This returns the complete employee roster with ~20 attributes per employee. The report is maintained by the ISU_Workato service account in Workday.

Stage 3: Parsing and In-Memory Loading (Steps 5-7)

The JSON response is parsed and loaded into a Workato Smart List called Employee_Collection, indexed on Employee_ID with force=true (overwrites existing data). This creates an in-memory database that can be queried with SQL.

Stage 4: Self-Join Query (Step 8)

A SQL query joins the Employee_Collection to itself to denormalize the employee record with their manager's data. The join matches e.Manager_Email = m.Email (self-referential on the same collection). The query also deduplicates by Employee_ID to handle any duplicates in the source data.

Stage 5: Batch Processing Loop (Step 10)

The query results are iterated in batches using a For-Each loop. Each batch passes through:

  1. Try/Catch wrapper -- catches batch-level exceptions
  2. Exception filter -- skips specified executive email addresses
  3. Subsidiary routing -- routes EOR/UK employees to a specific upsert path
  4. NetSuite batch upsert -- creates or updates employee records
  5. Error capture -- failed records added to ErrorList

Stage 6: Notification (Steps 11-13)

After all batches complete, the recipe checks if the ErrorList has any items. If so, an error summary email is sent to:

enterprise.integrations@intercom.io, nick.barton@intercom.io, nan.lu@intercom.io, pedro.cruz@intercom.io, venkatesh.manohar@intercom.io

Batch-level exceptions (caught by Try/Catch) trigger immediate emails to diana.sy@intercom.io.

9. Key Business Logic

9.1 Executive Exception Filter

The recipe skips specific email addresses from being upserted. These are executive/founder accounts that are managed manually in NetSuite:

Maintenance Note: When executives change or new exceptions are needed, this list must be updated in the recipe's IF condition (Step 10b). There is no external configuration for this -- it is hardcoded in the recipe logic.

9.2 Subsidiary Routing (EOR/UK Override)

For employees classified as EMEA EOR/Contractors in the United Kingdom, the NetSuite location field is hardcoded to internal ID 15 instead of being dynamically resolved via the lookup table. This addresses a specific business requirement where UK-based EOR employees belong to a different NetSuite location than what the standard country-based mapping would assign.

The Data_Mapping_Lookup project-level lookup table maps Workday text values to NetSuite internal IDs for four dimensions:

9.4 Email Domain Normalization

The recipe normalizes email addresses by replacing hq.intercom.com with intercom.io. This handles legacy email domain formats that may still exist in Workday for older employee records.

email = email.present? ? email.gsub("hq.intercom.com", "intercom.io") : email

9.5 Active/Inactive Status Mapping

Workday's Active_Status flag (1 = active, 0 = inactive) is inverted for NetSuite's isInactive field:

isInactive = ActiveStatus.is_true? ? "0" : "1"

Active employees get isInactive = 0 (not inactive); terminated employees get isInactive = 1 (inactive).

9.6 Upsert Key Strategy

The NetSuite batch upsert uses external_id = Workday Employee_ID as the match key. This is also stored in the custom field custentity_workday_id. This makes the operation idempotent -- running the recipe multiple times for the same data produces the same result without creating duplicate records.

11. EDGE CASES

11. Edge Cases & Known Limitations

Edge Case 1: Lookup Table Mismatch
If a new Workday cost center, location, or country is added that does not have a corresponding entry in the Data_Mapping_Lookup table, the lookup will return null. This causes the NetSuite upsert to either fail or set the field to blank, which may violate mandatory field requirements. Mitigation: The error is captured in the ErrorList and emailed to the team for manual lookup table update.
Edge Case 2: Duplicate Employee IDs
The SQL self-join query deduplicates on Employee_ID. However, if Workday returns multiple records with the same Employee_ID (e.g., concurrent employment), only one record will be processed. The dedup strategy is implicit in the SQL query logic.
Edge Case 3: Manager Not Found
The self-join matches employees to managers via email. If a manager's email in Workday does not match any employee's work email, the supervisor field will be empty in the NetSuite upsert. This is a left-join behavior.
Edge Case 4: Executive Exception List Staleness
The hardcoded executive email exception list does not auto-update. If an executive listed leaves the company or a new executive needs exclusion, the recipe must be manually edited. There is no alerting for this.
Edge Case 5: Email Domain Migration
The hq.intercom.com to intercom.io domain replacement handles legacy email formats. If Intercom ever changes its primary email domain again, this logic must be updated.
Edge Case 6: EOR/UK Location Hardcode
NetSuite location internal ID 15 is hardcoded for EMEA EOR/Contractor employees in the UK. If this location is reorganized or deactivated in NetSuite, the recipe will fail silently for these employees. The hardcoded value should ideally be replaced with a lookup table entry.
Edge Case 7: Batch-Level Failure Isolation
The Try/Catch block wraps the entire batch. If a single record in a batch causes an unrecoverable error (not a per-record validation failure), the entire batch fails and the error email is sent to diana.sy@intercom.io. Individual record failures within a successful batch are captured separately in the ErrorList.
