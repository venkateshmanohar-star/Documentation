1. EXECUTIVE SUMMARY

1. Executive Summary

The EXPENSIFY-Child recipe (ID: 59113811) is a callable child recipe within Intercom's Expensify V2 integration suite. It is invoked asynchronously by its parent recipe, EXPENSE - Workday to Expensify Parent V2 (ID: 59113812), which runs on a daily schedule at 1:00 AM PT.

This child recipe receives a batch of employee records (grouped by country) and upserts them into the Expensify Integration Server via a REST API POST call. It handles employee provisioning and deprovisioning, ensuring that Intercom's Expensify workspace stays synchronized with Workday as the source of truth for HR data.

2. BUSINESS OVERVIEW

2. Business Overview

Business Purpose

Intercom uses Expensify as its expense reimbursement platform. When employees join, leave, or change roles, their Expensify profiles must be updated to reflect the correct expense policies, manager approval chains, and termination status. This recipe automates the employee lifecycle management in Expensify, eliminating manual provisioning by the Finance/People Operations teams.

Business Process

  1. Workday serves as the authoritative HR system containing all employee data
  2. The parent recipe (59113812) extracts employee data from Workday daily, groups by country, and dispatches batches to this child recipe
  3. This child recipe takes each country batch and upserts employees into Expensify via API
  4. Expensify policies are mapped per country (via lookup table) ensuring correct expense approval workflows
  5. Terminated employees are flagged with isTerminated: true to disable their Expensify access

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

This recipe follows a Parent-Child Callable Pattern. The parent recipe (59113812) is scheduled to run daily at 1:00 AM PT. It fetches all employees from Workday via a custom report, groups them by country using a Smart List in-memory database, and dispatches each country batch asynchronously to this child recipe.

The child recipe acts as a stateless processor: it receives a country-specific employee list, constructs the Expensify API payload, and performs a single POST to the Expensify Integration Server to upsert all employees in that batch. Error handling follows the AOF (Automation Operations Framework) pattern, publishing structured error events to a centralized Pub/Sub topic.

6. END-TO-END FLOW

6. End-to-End Flow

Flow Narrative

  1. Trigger (Callable): The parent recipe (59113812) calls this child recipe asynchronously, passing a List_Of_Employees array and a Country string parameter.
  2. Log Input: The incoming payload is logged to the job report for traceability.
  3. Try Block - Upsert Employee: A POST request is sent to the Expensify Integration Server endpoint with:
    • requestJobDescription: JSON containing credentials, data source type ("employees"), and an onFinish email notification action
    • data: The employee list serialized as JSON
  4. Catch Block - Error Handling: If the API call fails, an error event is published to the AOF Pub/Sub topic (37790) with structured error details including job ID, recipe ID, stack trace, and error message.

9. KEY BUSINESS LOGIC

9. Key Business Logic

Employee Email Domain Normalization (Parent)

The parent recipe normalizes email domains before passing to this child:

  • If employee email contains hq.intercom.com, it is replaced with intercom.io
  • Same normalization applies to manager emails
  • This handles legacy domain references in Workday

Name Resolution (Parent)

  • If Preferred_FirstName is blank, falls back to Legal_FirstName
  • If Preferred_LastName is blank, falls back to Legal_LastName

Termination Detection (Parent)

  • isTerminated is set to true when ActiveStatus contains "0"
  • Terminated employees are still synced (not excluded) so Expensify can disable their accounts

Country-Based Policy Assignment (Parent)

  • Each country maps to a specific Expensify policyID via lookup table: Expensify Mapping - PolicyID
  • The lookup key is {Country} - Expense Reimbursement
  • This ensures employees are assigned to the correct country-specific expense policy

Manager Cross-Country Inclusion (Parent)

The parent's Smart List query includes managers who are in a different country than their reports. The SQL uses an OR clause:

  • Include employees where Primary_Work_Address_Country = {Country}
  • OR include employees whose email is in the manager email list for that country's employees
  • This ensures the approval chain is complete even when a manager is in a different country

Approval Routing

  • All employees have approvesTo set to accountspayable@intercom.io
  • This means final expense approval routes to the central Accounts Payable team

Custom Fields

  • customField1: Full preferred name (FirstName LastName)
  • customField2: Employee ID (same as employeeID)

Expensify API Behavior

  • API type is "update" (upsert semantics: creates if not exists, updates if exists)
  • dry-run: false means changes are applied immediately
  • An onFinish email notification is sent to the configured Notification_Recipient account property

11. EDGE CASES

11. Edge Cases

#ScenarioHandlingRisk
1 Expensify API timeout or 5xx error Caught by try/catch block. Error published to AOF topic. No retry (max_retry_count = 0). High - entire country batch fails with no automatic retry
2 Expensify returns non-2xx but ignoreHttpErrors is toggled The toggle config shows response.ignore_http_errors: true but the response schema checks ignore_http_errors: false. May silently pass on partial failures. Medium - conflicting config may mask API errors
3 Empty employee list for a country The child recipe will still fire and POST an empty array. Expensify API should handle gracefully but may return an error. Low
4 Country not found in PolicyID lookup table Handled by parent: the lookup returns null/empty, which means policyID is blank. Expensify may reject or use a default policy. Medium
5 Manager not found in Workday (no manager email) The LEFT JOIN in the parent query means employees without managers still appear. Manager email and manager country may be null. Low
6 Duplicate employee IDs across batches Managers from other countries are included in a country batch. The same employee can appear in multiple batches. Since the API is "update" (upsert), the last write wins. Low - idempotent upsert
7 Expensify partner credentials expire The secret is stored as a Workato account property. If rotated and not updated, all child invocations will fail simultaneously. High - global outage risk
8 Large country batch exceeds Expensify payload limit No chunking logic exists in the child. The entire country employee list is sent as a single POST. Medium - depends on Expensify limits
12. OPERATIONAL NOTES

12. Operational Notes

Scheduling

  • Parent Trigger: Daily at 1:00 AM Pacific Time
  • Child Trigger: Asynchronous callable - fires once per distinct country in the Workday report
  • Expected Concurrency: Multiple child instances run in parallel (one per country)

Monitoring and Alerting

  • Error Handling: All errors are published to the AOF Pub/Sub topic (PS-001, ID: 37790)
  • Email Notifications: Expensify sends a completion email to the configured Notification_Recipient account property via the onFinish callback
  • Job Logging: Incoming payload is logged to the Workato job report (Step 1)
  • No Retry: max_retry_count = 0 in the catch block; failed batches require manual re-run

Maintenance Notes

  • New Country: When Intercom expands to a new country, a row must be added to the Expensify Mapping - PolicyID lookup table with the country name and corresponding Expensify policy ID.
  • Credential Rotation: If the Expensify partner secret is rotated, update the expensify_user_secret account property in Workato immediately to prevent global failure.
  • Workday Report: The parent pulls from Workato_Integration_Report_-_Navan_All_EE_s. If the report URL or schema changes, the parent recipe must be updated.
  • Version History: Current version is 7, last edited by diana.sy@intercom.io on 2025-12-10.
