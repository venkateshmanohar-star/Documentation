1. Executive Summary

This Workato recipe is a callable function (Recipe Function) that acts as a reusable lookup service for Outreach prospect records. Given a prospect email address, it queries the Outreach API, determines whether a matching prospect exists, and returns the full prospect profile along with owner (user) details back to the calling recipe.

2. BUSINESS OVERVIEW

2. Business Overview

Purpose

The integration supports Intercom's GTM (Go-to-Market) operations by bridging the gap between Intercom conversations and Outreach sales engagement data. When a conversation is assigned in Intercom, the parent recipe needs to know whether the contact already exists as a prospect in Outreach and, if so, who owns them. This function encapsulates that lookup logic into a clean, reusable service.

Business Context

  • Sales-Support Alignment: Enables the sales team's Outreach engagement data to be linked with Intercom customer support conversations, ensuring that assigned reps in Intercom are aware of existing outreach sequences and ownership.
  • Data Enrichment: Returns a comprehensive prospect profile (100+ fields) including company information, engagement scores, social profiles, and custom fields.
  • Owner Resolution: When a prospect is found, fetches the Outreach user (owner) details so downstream processes can route or assign accordingly in Salesforce.

4. ARCHITECTURE OVERVIEW

4. Architecture Overview

This recipe follows a callable function pattern within Workato's recipe orchestration framework. It is not triggered independently -- it is invoked by a parent recipe that passes an email address and expects a structured response containing prospect lookup results.

Architectural Principles

  • Separation of Concerns: The Outreach lookup logic is isolated in this function, keeping the parent recipe clean and focused on its Intercom-to-SFDC sync workflow.
  • Reusability: As a callable function, any Workato recipe in the workspace can invoke this prospect search without duplicating the Outreach query logic.
  • Standardized Response: Returns a consistent schema (found_matching_prospect, number_of_matching_prospects, results array) regardless of whether a match is found.
  • Owner Enrichment: When a prospect is found, the function makes a second API call to fetch the Outreach user/owner record, providing ownership context to callers.

6. END-TO-END FLOW

6. End-to-End Flow

The end-to-end flow begins when the parent recipe Intercom to SFDC Conversation Assignee Sync receives a webhook event (Intercom conversation assignment) and calls this function to look up the prospect in Outreach before syncing to Salesforce.

Flow Narrative

  1. Trigger: Parent recipe (15918702) receives a webhook from Intercom indicating a conversation assignee change.
  2. Function Call: Parent recipe extracts the contact email and invokes this function, passing prospect_email as the input parameter.
  3. Search Outreach: This function queries the Outreach Prospect Search API, filtering by the email address (up to 1,000 results limit).
  4. Initialize Variables: Two tracking variables are created: found_matching_prospect (boolean) and number_of_matching_prospects (integer).
  5. Evaluate Results: An IF/ELSE condition checks whether the search returned zero results or an empty set.
  6. No Match Path: If no prospects found, variables are set to false and 0.
  7. Match Found Path: If prospects are found, the owner's Outreach User record is fetched via a second API call, and variables are set to true and the match count.
  8. Return: The function returns a structured payload containing the boolean flag, match count, and the full prospect/owner data array to the calling recipe.
  9. Downstream: The parent recipe uses this data to sync conversation assignee information to Salesforce.

1 Search Prospects in Outreach

Queries the Outreach API's Prospect Search endpoint, filtering by the emails field using the input prospect_email parameter. Limit set to 1,000 results.

Declares two recipe-level variables to track the search outcome:

3 Conditional: Evaluate Search Results

IF the result set size equals 0 OR is blank (no results returned):

Condition: results.size == 0 OR results.size is blank
3a IF TRUE (No Match): Update Variables

Sets found_matching_prospect = false and number_of_matching_prospects = 0.

Returns a structured JSON response containing:

9. KEY BUSINESS LOGIC

9. Key Business Logic

Search Strategy

The recipe searches Outreach Prospects using the email address as the primary filter. The search endpoint returns all prospects matching the email, with a configurable limit of 1,000 records. This means a single email could return multiple prospect records if duplicates exist in Outreach.

Match Determination Logic

// Condition: No match found IF results.size == 0 OR results.size is BLANK found_matching_prospect = false number_of_matching_prospects = 0 ELSE // Prospect found - also fetch owner details owner = Outreach.get_user(results[0].relationships.owner.data.id) found_matching_prospect = true number_of_matching_prospects = results.size

Owner Enrichment

When a matching prospect is found, the function makes a second API call to Outreach to fetch the User (owner) record. This uses the relationships.owner.data.id from the first matching prospect. The owner's details are available downstream for assignment routing decisions.

Note: The owner lookup uses only the first matching prospect's owner. If multiple prospects match the email (duplicates), only the first result's owner is fetched.

Data Pass-Through

The function returns an extensive prospect profile including 100+ standard Outreach fields (name, company, engagement scores, social URLs, etc.) plus up to 150 custom fields (custom1 through custom150). The full relationship graph (account, sequences, mailings, opportunities, etc.) is also included.

Maintenance Recommendations

  • Input Validation: Consider adding a check for blank/null email before calling the Outreach search API.
  • Error Handling: Add try/catch around the Outreach API calls to gracefully handle API failures and return a meaningful error to the caller.
  • Duplicate Handling: If multiple prospects match, consider logic to determine the "best" match rather than defaulting to the first result.
  • Owner Null Check: Add a null check on relationships.owner.data.id before the User lookup to avoid potential failures on unowned prospects.
  • Response Optimization: The 150 custom fields in the return payload may not all be needed; consider trimming to only fields used by downstream processes.
