2. Business Overview

▼

Purpose

Intercom's Finance team requires that all subscription billing data — originating from Stripe and Salesforce — be reflected in NetSuite for ASC 606 revenue recognition. This integration automates the Order-to-Cash (OTC) pipeline, creating and updating NetSuite records so that revenue schedules, invoices, and payments stay in sync without manual journal entries.

Business Impact

  • Revenue Recognition Compliance: Ensures ASC 606 multi-element arrangements are properly booked with correct rev rec dates.
  • Billing Accuracy: Creates Sales Orders with discount, proration, and co-term calculations matching Salesforce contracts.
  • Cash Application: Automates payment matching, CBA creation, and refund processing to keep AR aging accurate.
  • Audit Trail: Every write to NetSuite is logged in WORKATO_LOGS (66M+ rows) for full traceability.

4. Architecture Overview

▼

The integration follows a hierarchical master-child recipe pattern. The top-level scheduler (Rec 0.0) uses synchronous calls for Items and Sales Orders (which must complete in order), then asynchronous calls for Invoices/Credit Memos/Payments (which can run in parallel).


Execution Model

PhaseModeRecipe ChainRationale
1. ItemsSync0.0 → 2.0 → RF 2.0Items must exist before Sales Orders reference them
2. Sales OrdersSync0.0 → 3.0 → RF 3.0SOs must exist before Invoices can reference them
3. Invoices & DownstreamAsync0.0 → 1.0 → 4.0 → (2.0 Master → 5.0, 8.0, 6.0 → 3.0 Master)Parallelism improves throughput once dependencies satisfied

Error Handling Pattern

Every master recipe wraps its logic in a try/catch block. On failure, an HTML email is sent to the Finance and Integration teams with the error message. Retry count is set to 0 (no auto-retry) to prevent duplicate NetSuite records.

6. End-to-End Flow & Data Flow Diagram

▼

Step-by-Step Flow

  1. Cron fires at 05:30 / 10:30 / 15:30 / 20:30 PT daily.
  2. Variable declared: Date set to lookback anchor (currently 2026-02-01).
  3. Log the Date variable for audit.
  4. Call Rec 2.0 (sync): Items Master queries NETSUITE_EXPORT_ITEM joined with WORKATO_LOGS to find un-synced items (where netsuite_internal_id IS NULL and last_integration_date IS NULL). For each row, calls RF 2.0 to create/update the Inventory Item in NetSuite.
  5. Call Rec 3.0 (sync): Sales Order Master queries NETSUITE_EXPORT_SALES_ORDER similarly, creating Sales Orders in NetSuite via child recipe functions.
  6. Call 1.0 Master Invoice (async): This triggers the invoice chain:
    • Rec 4.0: Invoice creation in NetSuite
    • 2.0 Master CBA/Payments (async): Triggers Credit Memos (Rec 5.0), CBAs (Rec 8.0), and Customer Payments (Rec 6.0) sequentially
    • 3.0 Master Refunds (async): Handles refunds, uncollectible invoices, and contract cancellations
  7. Catch block: Any unhandled exception triggers an email with subject "NetSuite ARM Integration Timeout Error".

9. Business Logic

▼

Item Sync Logic (Rec 2.0)

The SQL query that drives item creation uses an anti-join pattern against WORKATO_LOGS:

WITH stg_workato_log AS (
    SELECT external_id,
           job_run_completed_at AS last_successful_job_run_completed_at,
           netsuite_internal_id, source_table_name,
           error_type, error_message
    FROM SHARED_EXTERNAL_WORKATO.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_ITEM'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id ORDER BY job_run_completed_at DESC
    ) = 1
)
SELECT BILLING_PLAN_ID, product_name, item_code, netsuite_class_code
FROM SHARED_EXTERNAL_WORKATO.netsuite_export_item i
LEFT JOIN stg_workato_log wl ON i.billing_plan_id = wl.external_id
WHERE wl.netsuite_internal_id IS NULL
  AND last_integration_date IS NULL

This ensures only new, un-synced items are sent to NetSuite. Each item row is passed to RF 2.0 with parameters: External_ID, Item_Type (item_code), Class (netsuite_class_code), and product_name.


Sales Order Sync Logic (Rec 3.0)

Similar anti-join pattern against WORKATO_LOGS for NETSUITE_EXPORT_SALES_ORDER. The Sales Order recipe joins with NETSUITE_EXPORT_CUSTOMER (for subsidiary) and NETSUITE_EXPORT_ITEM (for item_code). Key fields mapped:

  • Header: customer, salesforce_contract_id, date, PO number, memo, terms, order_type, currency
  • Lines: item (billing_plan_id), quantity, amount, discount, rev_rec_start/end dates, billing_schedule
  • Pricing: list_price, modified_list_price, discounted_unit_price, prorate_multiplier
  • Contract: contract_term, contract_end_date, default_subscription_term, is_co_term, proserve_type

Invoice & Downstream Logic (1.0 → 2.0 Master)

  • Invoices (Rec 4.0): Matches Stripe invoices to NetSuite Sales Orders. Filters by sales_channel IN ('self-serve', 'sales-contract') and amount != 0.
  • Credit Memos (Rec 5.0): Created for adjustments, refunds, and contract modifications.
  • CBAs (Rec 8.0): Customer Balance Adjustments applied from Stripe customer balance transactions.
  • Payments (Rec 6.0): Maps Stripe payment_id, payout_id, fee_amount to NetSuite Customer Payments.
  • Refunds / Cancellations (3.0 Master): Handles uncollectible invoices and contract termination records.

Dependency Ordering

The sync/async pattern enforces dependencies: Items must exist before Sales Orders reference them (sync). Sales Orders must exist before Invoices apply against them (sync to async boundary). Credit Memos and Payments can process in parallel with each other but must follow Invoices.

11. Edge Cases & Known Issues

▼
ScenarioBehaviorImpact
Timeout on large batch Catch block fires; HTML email sent with "NetSuite ARM Integration Timeout Error" Un-synced records remain and will be picked up on next run (idempotent anti-join)
Duplicate item attempt RF 2.0 uses external_id for upsert; NetSuite deduplicates by external ID No duplicate Items created
Missing customer record Sales Order creation fails in NetSuite; error logged to WORKATO_LOGS SO line skipped; will retry on subsequent runs once customer exists
Zero-amount lines Filtered out by amount != 0 conditions in most export queries Exception: total_contract_value = 0 lines ARE included for SOs
CBA from manual invoices CBAs not created in NetSuite when originating invoice was manual (proration/downgrade) Routing debug string in NS ID prevents RF 8.0 from parsing correctly
Async recipe failure Invoice/CBA/Payment failures do not block the parent recipe (fire-and-forget) Parent shows success; check child recipe jobs for actual errors
Sandbox vs Production WORKATO_LOGS filtered by environment_type = 'prod' Sandbox runs do not interfere with production dedup logic
Date variable as lookback anchor Currently hardcoded to 2026-02-01; not dynamically computed Must be manually updated for historical reprocessing
