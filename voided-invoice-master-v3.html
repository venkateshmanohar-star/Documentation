1. Executive Summary

The Voided Invoice Master Recipe (Rec 10.1) automates the creation of Credit Memos in NetSuite whenever an invoice is voided in Stripe. When a customer's invoice is voided (typically due to bad debt or billing errors), this recipe reads the voided invoice data from Snowflake, groups line items by Stripe invoice ID, and calls a child recipe that creates a corresponding Credit Memo in NetSuite to reverse the original charge.

This integration ensures that voided invoices in the billing system (Stripe) are properly reflected as credit memos in the financial system of record (NetSuite), maintaining accurate revenue recognition and financial reporting.

Business View
When Stripe voids an invoice, NetSuite automatically gets a Credit Memo created against that invoice. This keeps the books balanced and ensures bad-debt write-offs are properly recorded.
Technical View
A Workato Recipe Function queries Snowflake for unprocessed voided invoices (filtering against the WORKATO_LOGS deduplication table), groups line items by invoice, then iterates and calls child RF 10.1 which searches NetSuite for the original invoice and creates a Credit Memo with matching line details.

2. Business Overview

What Problem Does This Solve?

When a customer's invoice is voided in Stripe (the billing platform), the corresponding financial record in NetSuite must be adjusted. Without automation, the Finance team would need to manually create credit memos in NetSuite for every voided invoice — a time-consuming and error-prone process.

What Happens from a Business Perspective?

  1. A Stripe invoice is voided (e.g., bad debt, billing error, customer dispute)
  2. The voided invoice data lands in a Snowflake staging table via the ETL pipeline (Fivetran + dbt)
  3. This Workato recipe picks up unprocessed voided invoices
  4. For each voided invoice, it creates a Credit Memo in NetSuite that reverses the original invoice
  5. The Credit Memo includes all original line items (products, quantities, amounts, rev-rec dates)

4. Architecture Overview

Layered Architecture

Source Layer — Stripe
Stripe is the origin of truth for invoice lifecycle events. When an invoice is voided, this event flows through the ETL pipeline into Snowflake. The recipe does not connect directly to Stripe.
Data Layer — Snowflake
The NETSUITE_EXPORT_VOIDED_INVOICE table is a dbt-built staging table containing voided invoice line items with customer, item, amount, and rev-rec details. Joined with NETSUITE_EXPORT_ITEM for item mapping and WORKATO_LOGS for deduplication.
Orchestration Layer — Workato
Two recipes work together: the Master Recipe (10.1) extracts and groups data, then the child Recipe Function (RF 10.1) handles the NetSuite search-and-create logic for each individual invoice.
Target Layer — NetSuite
Credit Memos are created with: matching subsidiary, customer (via external ID), Stripe Invoice ID (custom field), memo "Voided Invoice", custom form 114, and line items with quantities, amounts, discounts, and rev-rec dates.

7. Processing Stages

Stage 1: Ingestion (Snowflake Query)

What Happens
The recipe reads all voided invoice line items from Snowflake that haven't been processed yet. It joins with item data to get item IDs, and checks the log table to skip already-processed invoices.
How It Works
A custom SQL query joins NETSUITE_EXPORT_VOIDED_INVOICE with NETSUITE_EXPORT_ITEM (on billing_plan_id) and WORKATO_LOGS (on stripe_invoice_id = external_id). Filters: INVOICE_VOIDED_AT >= {Date parameter}, wl.netsuite_internal_id IS NULL, amount != 0. Limit: 50,000 rows.

Stage 2: Grouping (Unique Invoice Extraction)

What Happens
Since one voided invoice may have multiple line items, the recipe identifies all unique invoice IDs and processes them one at a time.
How It Works
Uses Ruby formula .pluck('stripe_invoice_id').uniq to extract distinct invoice IDs. Creates a Unique_Invoice_List variable. Then iterates using a For Each loop indexed by list position.

Stage 3: List Building (Per-Invoice Assembly)

What Happens
For each invoice, all its line items are gathered into a structured list with all the data needed for the Credit Memo.
How It Works
Uses .where(stripe_invoice_id: ...) to filter the full dataset. Batch-inserts matching rows into an "Invoices" list with 16 fields (stripe_invoice_id, subsidiary, customer, invoice_voided_at, invoice_total, item, item_description, amount, tax_amount, line_type, actual_qty, discount_percentage, rev_rec_start_date, rev_rec_end_date, total_tax, item_internal_id). The list is cleared after each iteration.

Stage 4: NetSuite Processing (RF 10.1)

What Happens
The child recipe validates the invoice, checks NetSuite for duplicates, and creates a Credit Memo if one doesn't already exist. It handles edge cases like missing invoices with a retry mechanism.
How It Works
  1. Check invoice_total > 0
  2. Search NS for Invoice by custbodystripeinvoiceid
  3. Search NS for existing CreditMemo by externalId = VOID_{invoice_id}
  4. If no CM found AND invoice found: create CreditMemo (custom form 114, memo "Voided Invoice")
  5. If invoice not found initially: execute concurrency wait (Ruby sleep), re-search with status = CustInvc:A, then create
  6. Map result to NetSuite_Integration_Log schema (ID 78962) and log

Stage 5: Logging & Deduplication

What Happens
Every processed invoice is logged so it won't be picked up again on the next run. Both successes and failures are recorded.
How It Works
The Workato mapper writes to the NetSuite_Integration_Log schema. The logger publishes to the job report. These entries feed into the WORKATO_LOGS Snowflake table with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_VOID_INVOICE'. The dedup CTE in the extraction query filters on netsuite_internal_id IS NULL.

9. Key Business Logic

Credit Memo Creation Logic

The core business logic determines when and how a Credit Memo is created in NetSuite to reverse a voided Stripe invoice.

Invoice Classification

ConditionActionRationale
invoice_total > 0Process → Create Credit MemoStandard voided invoices with positive balances need reversal
invoice_total < 0Alternative handlingNegative totals indicate credits/adjustments that may need different treatment
amount = 0Filtered out at SQL levelZero-amount lines carry no financial impact

Credit Memo Structure

Each Credit Memo is created with the following key attributes:

  • External ID: VOID_{stripe_invoice_id} — ensures idempotency and prevents duplicates
  • Custom Form: 114 — the designated NetSuite form for voided invoice credit memos
  • Memo: "Voided Invoice" — clear audit trail
  • Subsidiary: Looked up from a NetSuite Subsidiaries table (US or IE)
  • Customer: Linked via Stripe customer external ID
  • Stripe Invoice ID: Set via custom field custbodystripeinvoiceid
  • Line Items: Replicate original invoice lines with amounts, quantities, items, discounts, and rev-rec dates

Deduplication Strategy

Two levels of deduplication protect against duplicate Credit Memos:

  1. Snowflake-level: The extraction SQL LEFT JOINs against WORKATO_LOGS and only includes rows where netsuite_internal_id IS NULL (not yet processed)
  2. NetSuite-level: RF 10.1 searches for an existing Credit Memo with externalId = VOID_{invoice_id}. If found, the invoice is skipped.

Concurrency Handling

If the initial search for the NetSuite invoice returns no results (possible when another recipe is mid-creation), the recipe executes a Ruby concurrency wait (sleep) and re-searches with a status filter (CustInvc:A = open invoices). This handles race conditions with the main invoice creation pipeline.

11. Edge Cases & Special Scenarios

Invoice Not Found in NetSuite

What Happens
Sometimes the original invoice hasn't been created in NetSuite yet when the voided invoice recipe runs. This can happen if the main invoice pipeline is still processing.
How It's Handled
RF 10.1 executes a Ruby concurrency wait (sleep), then re-searches NetSuite for the invoice with status CustInvc:A (Approved/Open). If found on retry, it proceeds to create the Credit Memo.

Credit Memo Already Exists

What Happens
If the recipe runs twice for the same invoice (e.g., due to a rerun), it should not create a duplicate Credit Memo.
How It's Handled
The recipe searches NetSuite for a CreditMemo with externalId = VOID_{stripe_invoice_id}. If found (count > 0), the recipe skips creation and logs the skip event. Additionally, the Snowflake-level dedup prevents the invoice from appearing in the extraction query after the first successful run.

Negative Invoice Totals

What Happens
Some voided invoices may have negative totals, indicating they were credits or adjustments rather than charges.
How It's Handled
The ELSIF branch (invoice_total < 0) provides a separate handling path. Currently this appears to pass through to the logging step without creating a Credit Memo, as reversing a credit doesn't make sense.

Zero-Amount Line Items

What Happens
Some invoice lines have zero amounts (e.g., trial periods, $0 add-ons). These don't need financial reversal.
How It's Handled
Filtered out at the SQL level with amount != 0. They never enter the processing pipeline.

Missing Customer in NetSuite

What Happens
The Credit Memo references a customer by their Stripe external ID. If the customer hasn't been created in NetSuite, the creation will fail.
How It's Handled
The error is caught by the Try/Catch block and logged to WORKATO_LOGS with the error type and message. The invoice will be retried on the next run (since no netsuite_internal_id will be recorded).

12. Operational Notes

Deduplication Mechanism

The integration uses a two-tier deduplication approach:

  1. Pre-processing (Snowflake): The extraction SQL LEFT JOINs to WORKATO_LOGS filtered by source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_VOID_INVOICE'. Only rows where wl.netsuite_internal_id IS NULL are included, meaning only unprocessed invoices are extracted.
  2. In-processing (NetSuite): RF 10.1 searches for an existing CreditMemo with externalId = VOID_{stripe_invoice_id}. If found, the invoice is skipped.

Error Handling

  • The master recipe wraps all processing in a Try/Catch block with 0 retries and a 2-second interval
  • The child recipe (RF 10.1) also has its own Try/Catch that logs errors via the mapper (schema 78962) and logger
  • Failed records remain in the extraction pool (no log entry) and will be retried on the next run
