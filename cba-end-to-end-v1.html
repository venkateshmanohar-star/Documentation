CBA Pipeline — End-to-End System Documentation

Complete system-level documentation: Stripe → Fivetran → Snowflake (dbt) → Workato → NetSuite

Type: End-to-End Author: venkatesh.manohar Generated: April 9, 2026 Systems: 5 (Stripe, Fivetran, Snowflake, Workato, NetSuite) Workato Recipes: 4 active, 2 stopped Schedule: Via 2.0 Master (CM, CBA, Payments)

1. Executive Summary

Business View

When a customer has a credit balance in Stripe — from refunds, credit notes, subscription downgrades, or proration adjustments — that credit needs to be reflected in NetSuite (our ERP) as a Customer Balance Adjustment (CBA). This pipeline automates that process, ensuring every Stripe credit balance transaction that represents a real financial adjustment gets recorded as a deposit or journal entry in NetSuite for accurate accounts receivable tracking.

CBAs are critical for revenue reconciliation: they represent money owed back to the customer or credit applied to future invoices. Without automation, Finance would need to manually process thousands of balance adjustments monthly.

Technical View

A 5-system pipeline: Stripe (source of customer_balance_transactions via API) → Fivetran (Stripe data share replication to Snowflake) → Snowflake (dbt models + export table NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS) → Workato (Rec 8.0 master queries Snowflake, RF 8.0 child creates NetSuite records) → NetSuite (customer deposit/journal entry).

The pipeline is invoked as step 2 within the 2.0 Master Recipe (Credit Memo, CBAs and Payments), which runs on a scheduled basis. A WORKATO_LOGS table provides idempotency by tracking which external IDs have been synced.

Reliability: RF 8.0 (the core child recipe) has processed 66,869 jobs with 0 failures (900,253 lifetime API tasks). Rec 8.0 (the master) has completed 873 jobs with 9 failures. The export table contains 20,924 CBA records.
Known Issue: CBAs originating from manual invoices (proration/downgrade scenarios) may not appear in NetSuite. The routing debug string embedded in the NetSuite ID prevents RF 8.0 from parsing the external ID correctly. See Section 11 for details.

2. Business Overview

What Problem Does This Solve?

Intercom bills customers via Stripe. When events occur that generate credit balances — such as credit notes being issued, subscription downgrades creating prorated refunds, or overpayments — Stripe creates customer_balance_transaction records. These represent real financial obligations that must be mirrored in NetSuite for accurate AR (Accounts Receivable) tracking, revenue recognition, and financial reporting.

Without this pipeline, Finance would need to manually identify thousands of Stripe balance transactions, determine the correct NetSuite subsidiary and customer, convert currencies, and create the corresponding records — a process prone to errors and omissions.

Business Flow

  1. Credit event occurs in Stripe — A credit note is issued, an invoice is voided, a downgrade creates a proration credit, or a refund generates a customer balance
  2. Stripe records the balance transaction — A customer_balance_transaction object is created with entity (customer ID), amount, type, currency, and linked invoice/credit note references
  3. Data flows to Snowflake — Fivetran replicates Stripe data to Snowflake; dbt transforms it through staging layers into the export table NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS (20,924 rows)
  4. Workato picks up unsynced CBAs — Rec 8.0 queries Snowflake for records where the WORKATO_LOGS entry has no netsuite_internal_id, meaning it has not yet been synced
  5. CBA created in NetSuite — RF 8.0 creates the appropriate record in NetSuite (customer deposit or journal entry) with the correct subsidiary, currency, amount, and customer linkage
  6. Log updated — Success/failure is written back to WORKATO_LOGS with the NetSuite internal ID for idempotent dedup

4. Architecture Overview

Source Layer

Stripe Billing holds the canonical customer balance transaction data. When a credit event occurs (credit note, refund, proration), Stripe creates a customer_balance_transaction object with the amount, currency, type (adjustment/applied_to_invoice), and references to the originating credit note or invoice.

Source Layer — Technical

Stripe exposes data via its Data Share product and API. Fivetran's Stripe connector replicates balance transactions, credit notes, and customer data to Snowflake. The raw data lands in STAGING_STRIPE_DATA_SHARE and SNAPSHOT_STRIPE schemas. Key raw table: SNAP_STRIPE_BALANCE_TRANSACTION (10M+ rows).

Data Layer

Snowflake transforms Stripe balance data through dbt models, enriches it with customer metadata (subsidiary, NetSuite internal ID, category), and produces the export table that Workato reads. The export table (NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS) contains 20,924 rows with all the data needed for NetSuite record creation.

Data Layer — Technical

Sequential dbt pipeline: Stripe Data Share (raw) → STG_STRIPE_BALANCE_TRANSACTIONS (staging, 3.4M rows) → STRIPE_BALANCE_TRANSACTION (base, 3.3M rows) → NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS (export, 20,924 rows, 14 cols, team-enterprise-systems).
Workato reads the export table joined with NETSUITE_EXPORT_CUSTOMER (327K rows) for subsidiary/NetSuite ID, cross-referenced against WORKATO_LOGS (67M rows) for idempotency.

Orchestration Layer

Workato is the automation engine. The 2.0 Master Recipe triggers Rec 8.0, which queries Snowflake for unsynced CBAs. For each CBA, it calls RF 8.0, which creates the appropriate record in NetSuite with correct customer linkage, subsidiary assignment, and currency handling.

Orchestration Layer — Technical

2.0 Master → Rec 8.0 (Snowflake SQL query, foreach loop) → RF 8.0 (NetSuite API: netsuite.add for customer deposit/journal entry, ruby custom code for field mapping, workato_list for data manipulation). RF 8.0 uses 7 Workato connectors: netsuite, workato_custom_code, logger, workato_list, workato_mapper, workato_variable, workato_recipe_function.

Target Layer

NetSuite receives the CBA records. Finance uses these for AR reconciliation, ensuring Stripe credit balances are properly reflected in the general ledger. Customer deposit records link back to Stripe via the external ID for traceability.

Target Layer — Technical

NetSuite SuiteTalk API. CBA records are created as customer deposits or journal entries depending on the transaction type. External ID = Stripe customer_balance_transaction ID (e.g., cbtxn_xxx) for idempotent upserts. Subsidiary is resolved from the NETSUITE_EXPORT_CUSTOMER lookup.

6. End-to-End Flow

The complete data journey for a Credit Balance Adjustment follows this sequential path across five systems:

  1. Stripe Billing — A credit event occurs: a credit note is issued (cn_xxx), a subscription downgrade generates a proration credit, or an invoice is voided. Stripe creates a customer_balance_transaction (cbtxn_xxx) recording the amount, currency, type, and references to the originating credit note or invoice.
  2. Fivetran ETL — Stripe Data Share connector continuously replicates balance transactions, credit notes, customers, and invoices to Snowflake. Data lands in SNAPSHOT_STRIPE and STAGING_STRIPE_DATA_SHARE schemas.
  3. Snowflake dbt Pipeline — A sequential dbt pipeline transforms the raw data:
    • SNAP_STRIPE_BALANCE_TRANSACTION — raw snapshot (10M rows, SNAPSHOT_STRIPE)
    • STG_STRIPE_BALANCE_TRANSACTIONS — staging table (3.4M rows, STAGING_PRODUCT)
    • STRIPE_BALANCE_TRANSACTION — base table (3.3M rows, BASE schema)
    • NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS — export table (20,924 rows, 14 cols, SHARED_EXTERNAL_WORKATO) — this is what Workato reads
    • Side branch: INT_STRIPE_BALANCE_STATEMENT_BY_INVOICE_LINE_ITEM (1.6M rows) for analytics/recon
  4. Workato Orchestration — The 2.0 Master Recipe calls Rec 8.0, which queries the export table joined with NETSUITE_EXPORT_CUSTOMER for subsidiary/NS ID info, filtered against WORKATO_LOGS to exclude already-synced records. For each unsynced CBA, RF 8.0 is called with entity, amount, type, currency, credit note, description, ending balance, invoice ID, subsidiary, and customer NetSuite internal ID.
  5. NetSuite ERP — RF 8.0 creates the appropriate record (customer deposit or journal entry) via SuiteTalk API with the correct subsidiary, currency, and customer linkage. The Stripe cbtxn_xxx ID is set as the external ID for idempotent upserts.
Feedback loop: After each NetSuite API call, Workato writes the result (ns_internal_id, error_type, error_message) back to WORKATO_LOGS in Snowflake (source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS'). This prevents re-processing on subsequent runs.

Key SQL: Rec 8.0 Master Query (Snowflake) 
WITH stg_workato_log AS (
    SELECT
        external_id,
        job_run_completed_at AS last_successful_job_run_completed_at,
        netsuite_internal_id,
        source_table_name,
        error_type,
        error_message
    FROM intercom_prod.shared_external_workato.WORKATO_LOGS
    WHERE
        external_id IS NOT NULL
        AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS'
        AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1
)
SELECT
    entity,
    amount,
    created,
    credit_note,
    currency,
    description,
    ending_balance,
    cba.external_id,
    invoice_id,
    c.subsidiary,
    c.netsuite_internal_id AS customer_netsuite_internal_id,
    cba.netsuite_internal_id AS cba_netsuite_internal_id,
    type,
    c.category
FROM
    SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS cba
LEFT JOIN
    SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER c ON cba.entity = c.external_id
LEFT JOIN
    stg_workato_log wl ON wl.external_id = cba.external_id
WHERE
    created >= '{date}' AND created <= DATEADD(day, -1, CURRENT_DATE)
    AND wl.netsuite_internal_id IS NULL
ORDER BY created

9. Key Business Logic

CBA Type Routing

Business View

Stripe balance transactions come in different types. Adjustment types represent direct credits to a customer's balance (from credit notes, refunds). Applied_to_invoice types represent credits being consumed against a future invoice. Each type maps to a different NetSuite record type to maintain proper accounting treatment.

Technical View

RF 8.0 reads the type field from the Snowflake export row. Using workato_custom_code (Ruby), it maps Stripe types to NetSuite record types and constructs the appropriate API payload. The workato_mapper handles field-level transformations (currency codes, date formats, amount sign conventions). External ID = cbtxn_xxx for idempotent upserts.

2.0 Master Recipe Orchestration Sequence

The 2.0 Master Recipe orchestrates four sub-pipelines in a specific order. CBAs are step 2, intentionally placed after credit memos (to ensure the credit note records exist in NetSuite first) and before customer payments:

  1. Step 1: Rec 5.0 Credit Memo Master — Creates credit memos in NetSuite (sync call)
  2. Step 2: Rec 8.0 Create CBA — Creates balance adjustments in NetSuite (sync call) — this pipeline
  3. Step 3: Rec 6.0 Customer Payment — Creates customer payments in NetSuite (sync call)
  4. Step 4: 3.0 Master — Refunds, Uncollectible, Cancellations — Handles refunds, uncollectible invoices, and cancellations (async call)
Error handling: The 2.0 Master Recipe wraps all calls in a try/catch block. On failure, it sends an email to danny.mcgean@intercom.io, nan.lu@intercom.io, chase.clark@connorgp.com, and diana.sy@intercom.io with subject "NetSuite ARM Integration Timeout Error" and the error message.

WORKATO_LOGS Filter Strategy

Business View

The WORKATO_LOGS table is a shared log across all ARM integration pipelines (Sales Orders, Credit Memos, CBAs, Payments, Refunds). With 67 million rows, precise filtering is essential. The CBA pipeline uses the source_table_name column to isolate only CBA-related log entries.

Technical View

The CTE stg_workato_log filters to source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS' and environment_type = 'prod'. It then deduplicates using QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id ORDER BY job_run_completed_at DESC) = 1, keeping only the most recent log entry per external ID. The LEFT JOIN with wl.netsuite_internal_id IS NULL identifies un-synced records.

11. Edge Cases & Special Scenarios

CRITICAL: CBAs from Manual Invoices Not Syncing to NetSuite
When a CBA originates from a manual Stripe invoice (typically created during proration/downgrade scenarios), the routing debug string embedded in the NetSuite ID field prevents RF 8.0 from parsing the external ID correctly. The netsuite_internal_id column in the export table contains a debug string from a previous routing attempt rather than a clean NS internal ID or NULL, causing the WORKATO_LOGS check to incorrectly believe the record has already been synced.

Impact: These CBAs silently fail to appear in NetSuite, creating reconciliation gaps between Stripe and NetSuite.
Detection: Look for records in NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS where netsuite_internal_id contains non-numeric text or routing debug strings rather than a valid NetSuite internal ID number.
Workaround: Use the manual recipe 5.02.8 Create CBA Master — Manual (ID: 54968410) to re-process these records by providing the specific external ID.

    DATE_TRUNC('month', cba.created) AS month,
    COUNT(*) AS total_cbas,
    COUNT(wl.netsuite_internal_id) AS synced,
    COUNT(*) - COUNT(wl.netsuite_internal_id) AS unsynced
FROM INTERCOM_PROD.SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_CUSTOMER_CREDIT_ADJUSTMENTS cba
LEFT JOIN log wl ON wl.external_id = cba.external_id
GROUP BY 1
ORDER BY 1 DESC;
