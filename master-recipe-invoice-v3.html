1. Executive Summary

The 1.0 Master Recipe — Invoice is the top-level orchestrator for Intercom's NetSuite ARM (Accounts Receivable Management) integration. It coordinates the automated creation and synchronization of financial transactions from Stripe billing data (staged in Snowflake) into NetSuite ERP. The recipe operates as a three-branch parallel pipeline handling invoices, credit memos, CBAs, payments, refunds, voided invoices, uncollectible invoices, and cancelled sales orders.

2. Business Overview

Purpose

This integration automates Intercom's order-to-cash financial operations by synchronizing Stripe billing events with NetSuite ERP. Without this automation, the Finance team would need to manually create thousands of invoices, credit memos, payments, refunds, and journal entries in NetSuite each day.

Business Value

  • Revenue Recognition: Ensures timely and accurate creation of invoices with rev-rec schedules in NetSuite ARM
  • Cash Application: Automates matching of Stripe payments to NetSuite invoices
  • Credit Management: Handles credit memos, CBAs (Customer Balance Adjustments), and refunds automatically
  • Compliance: Maintains audit trail via WORKATO_LOGS with idempotent processing (checks for prior successful writes)
  • Multi-subsidiary Support: Routes transactions to correct NetSuite subsidiaries based on customer entity mapping

4. Architecture Overview

The integration uses a three-tier master-recipe orchestration pattern:

Tier 1: Top-Level Orchestrator

Recipe 58628331 (1.0 Master Recipe - Invoice) accepts a Date parameter and dispatches work to two branches:

  • Synchronous call to Rec 4.0 (Invoice Master) — must complete before CBAs/Payments branch starts
  • Asynchronous call to Recipe 58628332 (2.0 Master Recipe - Credit Memo CBAs and Payments)

Tier 2: Domain Masters

  • Rec 4.0 (54665262) — Invoice creation (self-serve + sales-led)
  • Recipe 58628332 (2.0) — Credit Memos, CBAs, Payments, then async to Recipe 58628333
  • Recipe 58628333 (3.0) — Refunds, Voided Invoices, Uncollectible Invoices, Cancelled Sales Orders

Tier 3: Processing Recipes

Each domain master queries Snowflake for unprocessed records (using WORKATO_LOGS for idempotency), then iterates and dispatches to RF (Recipe Function) sub-recipes that perform the actual NetSuite API calls.

Execution Order: Invoices must be created first (sync) because credit memos, CBAs, and payments reference invoice IDs. Once invoices are done, the remaining transaction types can process in parallel.

9. Business Logic

Idempotency Pattern

Every processing recipe uses the same idempotency pattern via a CTE against WORKATO_LOGS:

WITH stg_workato_log AS (
    SELECT external_id,
           netsuite_internal_id,
           source_table_name
    FROM SHARED_EXTERNAL_WORKATO.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = '{table_name}'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY external_id
        ORDER BY job_run_completed_at DESC
    ) = 1
)
SELECT ...
FROM {source_table}
LEFT JOIN stg_workato_log wl ON ...
WHERE wl.netsuite_internal_id IS NULL  -- Only unprocessed records

Invoice Routing Logic

Rec 4.0 routes invoices based on the sales_channel field:

  • Self-Serve: Dispatches to RF 4.01 — creates standalone NetSuite invoices
  • Sales-Led: Dispatches to RF 4.00 — creates invoices linked to existing sales orders

Date Filtering

All queries use the pattern: created >= '{Date}' AND created <= DATEADD(day, -1, CURRENT_DATE). This ensures same-day records (which may still be updating in Stripe) are not processed prematurely.

Payment Exclusion

Rec 6.0 excludes bank transfer payments: check_reference_number IS NULL OR check_reference_number <> 'Bank Transfer Payment'. These are handled through a separate process.

Multi-Subsidiary Routing

Customer subsidiary information flows from NETSUITE_EXPORT_CUSTOMER through to each NetSuite transaction, ensuring transactions are created in the correct legal entity.

Invoice Line Grouping

Recipes 4.0, 5.0, 10.0, and 10.1 group individual line items into invoice-level batches using Workato's list variables. The pattern:

  1. Query returns all line items ordered by date
  2. Extract unique invoice IDs: .pluck('stripe_invoice_id').uniq
  3. Create a list sized to the unique count
  4. For each unique invoice, filter and batch-insert matching lines
  5. Call the RF recipe with the complete invoice + line items
