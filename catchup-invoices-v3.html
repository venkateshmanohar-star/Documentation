Rec 4.1 — SystemMigration-Create Catchup Invoices

Recipe ID: 64257157 Status: Stopped Author: venkatesh.manohar Last Run: 2025-12-03 Generated: 2026-04-01

1. Executive Summary

What it does: This recipe automatically creates "catchup invoices" in NetSuite for customers who have been migrated from a legacy billing system. When a customer's sales order has been successfully created in NetSuite but the corresponding invoice has not yet been generated, this recipe identifies the gap and creates the missing invoice.

Why it exists: During Intercom's system migration from its legacy billing platform, sales orders were created in NetSuite for existing contracts. However, invoices -- the actual billing documents customers receive -- were not always generated. This recipe closes that gap by running daily to detect unmatched contracts and producing the corresponding NetSuite invoices with correct line items, billing schedules, and revenue recognition dates.

High-level flow: Every day at 11:15 AM, the recipe queries Snowflake to find Salesforce contracts that have a completed sales order in NetSuite but no corresponding catchup invoice. For each contract found, it calls a child recipe that looks up the sales order in NetSuite, creates an invoice from it, computes billing frequency for each line item, and updates the invoice with the correct detail lines.

2. Business Overview

The Problem

When Intercom migrated customers from its legacy billing system to NetSuite, the migration process created sales orders (the internal record of what was sold) but did not always create invoices (the billing document sent to customers). Without invoices, revenue cannot be properly recognized in NetSuite, and customers may not be billed correctly for their ongoing subscriptions.

The Solution

This integration acts as a daily "safety net" that:

  • Identifies all migrated contracts that have a sales order but no catchup invoice
  • For each contract, looks up the NetSuite sales order and creates a matching invoice
  • Computes the correct billing frequency (monthly, quarterly, annual) for each line item
  • Updates the invoice with properly computed line items including amounts, rev-rec dates, and billing schedules

Business Impact

This recipe ran 91 successful jobs with only 1 failure over its active lifetime (July 2025 - December 2025), processing 19,422 tasks. The child recipe (RF 4.1a) processed 9,610 contracts with zero failures, generating 123,333 tasks. It was stopped on December 4, 2025, indicating the migration backlog was cleared.

4. Architecture Overview

Business View

Source Layer: Salesforce contract data, already staged in Snowflake during the migration process, represents the "what was sold" records.

Data Layer: Snowflake holds the staging tables that track which contracts have been processed and which still need invoices.

Orchestration Layer: Workato runs a daily check, identifies gaps, and coordinates the invoice creation process across multiple recipes.

Target Layer: NetSuite receives the completed invoices, enabling proper revenue recognition and customer billing.

Technical View

Source: NETSUITE_EXPORT_SALES_ORDER table filtered by data_type='legacy_migration'. Joined with WORKATO_LOGS to check processing status.

Data: CTE-based SQL with deduplication via ROW_NUMBER() windowing. Checks both sales order logs and catchup invoice logs.

Orchestration: Clock trigger (daily 11:15 AM) → SQL query → For-Each loop over distinct contracts → Recipe Function call.

Target: NetSuite API via Workato connector: search_object_v2 (Sales Order, Invoice), add_object (Invoice), update_object (Invoice lines).

6. End-to-End Flow

Level 0 -- Simple View

Every day, the system checks Snowflake for migrated contracts that have a sales order in NetSuite but no corresponding invoice. For each missing invoice, it creates one in NetSuite using data from the sales order, then updates it with properly computed line items.

Level 1 -- Detailed View

  1. Extraction: A SQL query joins NETSUITE_EXPORT_SALES_ORDER with WORKATO_LOGS to find contracts where the sales order succeeded (wl.netsuite_internal_id IS NOT NULL) but the catchup invoice log entry does not exist (ci.netsuite_internal_id IS NULL). Only data_type='legacy_migration' rows are considered.
  2. Grouping: Results are grouped by distinct salesforce_contract_id, migration_date, and pricebook_name.
  3. Iteration: The parent recipe iterates over each returned row, passing the salesforce_contract_id to the child recipe.
  4. Processing (Child RF 4.1a):
    • Searches NetSuite for the sales order matching the SFDC Contract ID
    • Checks if a catchup invoice already exists (deduplication)
    • Creates a new invoice from the sales order header data
    • Retrieves the newly created invoice to get its internal ID
    • For each line item, calls the grandchild recipe to compute billing frequency
    • Updates the invoice with computed line items
  5. Output: A fully-formed invoice in NetSuite with correct line items, amounts, billing schedules, and revenue recognition dates. Processing status is logged to WORKATO_LOGS.

Level 2 -- Processing Detail

The child recipe (RF 4.1a) follows a strict sequence:

  1. Search for Sales Order by SFDC Contract ID
  2. If found, check if a catchup invoice already exists (prevents duplicates)
  3. If no existing invoice, create the invoice from the sales order
  4. Use custom Ruby code to extract and transform line item data
  5. For each line item, call RF 4.1b to compute billing frequency (monthly/quarterly/annual) based on the subscription term and billing schedule
  6. Accumulate all computed line items into a list
  7. Update the invoice with the complete line item list

All errors are caught by try/catch blocks and logged via Workato's mapper action for audit trail purposes.

7. Processing Stages

Stage 1: Ingestion (Master Recipe)

What Happens

The system runs a daily query against the data warehouse to find contracts that still need catchup invoices. It looks for contracts where a sales order was successfully created but no invoice exists yet.

How It Works

A CTE-based SQL query joins NETSUITE_EXPORT_SALES_ORDER (filtered to data_type='legacy_migration') with two instances of WORKATO_LOGS: one checking for completed sales orders (source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_SALES_ORDER') and one for catchup invoices (source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_CATCHUP_INVOICE'). The ROW_NUMBER() window function deduplicates logs by taking the most recent per external_id. The WHERE clause requires wl.netsuite_internal_id IS NOT NULL AND ci.netsuite_internal_id IS NULL.

Stage 2: Grouping & Iteration

What Happens

The results are grouped by contract, and the system processes each contract one at a time.

How It Works

The SQL returns DISTINCT(salesforce_contract_id), migration_date, pricebook_name. A For-Each loop iterates over the result set, passing each salesforce_contract_id to the child recipe via a synchronous Recipe Function call.

Stage 3: Sales Order Lookup (Child RF 4.1a)

What Happens

For each contract, the system looks up the corresponding sales order in NetSuite and verifies that no invoice already exists.

How It Works

NetSuite search_object_v2 finds the sales order by SFDC Contract ID. A second search checks for existing invoices to prevent duplicates. Custom Ruby code extracts and transforms the line item data from the sales order.

Stage 4: Invoice Creation & Line Computation

What Happens

The system creates a new invoice in NetSuite, then computes the correct billing frequency for each line item and updates the invoice with the full detail.

How It Works

add_object creates the invoice header. The newly created invoice is then retrieved via search_object_v2 to get its internal ID. For each line item, RF 4.1b is called to compute the billing frequency based on subscription term and billing schedule. The accumulated line items are written back via update_object.

Stage 5: Logging & Error Handling

What Happens

Every action is logged for audit purposes. Errors are caught and recorded without stopping the overall process.

How It Works

Try/catch blocks wrap each major operation. The catch handlers use Workato's mapper action to format error details and logger to record them. The master recipe has its own try/catch with 0 retries and a 2-second retry interval, ensuring failures in one contract do not block processing of others.

9. Key Business Logic

Gap Detection Logic

The core business logic identifies contracts that need catchup invoices. A contract qualifies if:

  1. It is a legacy migration record (data_type='legacy_migration')
  2. Its sales order has been successfully processed (the sales order log entry has a netsuite_internal_id)
  3. Its catchup invoice has NOT been processed (no catchup invoice log entry with a netsuite_internal_id)
View the SQL Query 
WITH stg_workato_log AS (
    SELECT external_id,
           job_run_completed_at AS last_successful_job_run_completed_at,
           netsuite_internal_id, source_table_name,
           error_type, error_message
    FROM SHARED_EXTERNAL_WORKATO.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_EXPORT_SALES_ORDER'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id
                               ORDER BY job_run_completed_at DESC) = 1
),
catchup_invoice AS (
    SELECT external_id,
           job_run_completed_at AS last_successful_job_run_completed_at,
           netsuite_internal_id, source_table_name,
           error_type, error_message
    FROM SHARED_EXTERNAL_WORKATO.WORKATO_LOGS
    WHERE external_id IS NOT NULL
      AND source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_CATCHUP_INVOICE'
      AND environment_type = 'prod'
    QUALIFY ROW_NUMBER() OVER (PARTITION BY external_id
                               ORDER BY job_run_completed_at DESC) = 1
)
SELECT DISTINCT(salesforce_contract_id), migration_date, pricebook_name
FROM SHARED_EXTERNAL_WORKATO.netsuite_export_sales_order so
LEFT JOIN stg_workato_log wl ON so.external_id = wl.external_id
LEFT JOIN catchup_invoice ci ON so.salesforce_contract_id = ci.external_id
WHERE data_type = 'legacy_migration'
  AND wl.netsuite_internal_id IS NOT NULL
  AND ci.netsuite_internal_id IS NULL
ORDER BY 2;

Billing Frequency Computation (RF 4.1b)

The grandchild recipe computes the billing frequency for each line item based on:

  • Subscription term (from the sales order line item)
  • Billing schedule (monthly, quarterly, annual)
  • Default subscription term

This recipe is shared across multiple integration flows (it remains Active even though the parent is Stopped), indicating it is a reusable utility used by other NetSuite integration recipes.

Reference spreadsheet: The recipe description links to a Google Sheets document containing the calculation rules.

Deduplication Strategy

The system prevents duplicate invoice creation at two levels:

  1. SQL-level dedup: The master recipe's query checks WORKATO_LOGS for existing catchup invoice entries. Only contracts WITHOUT a log entry are returned.
  2. NetSuite-level dedup: The child recipe searches NetSuite for existing invoices matching the sales order before creating a new one.

11. Edge Cases & Special Scenarios

Sales Order Not Found in NetSuite

If the child recipe searches NetSuite for a sales order using the SFDC Contract ID and finds no match, the recipe logs the missing sales order and skips invoice creation. This can happen if the sales order creation failed or was never initiated.

Catchup Invoice Already Exists

The child recipe performs a second NetSuite search to check if a catchup invoice already exists for the contract. If found, the recipe skips creation. This provides NetSuite-side deduplication in addition to the Snowflake log-based check in the master recipe.

Zero-Amount Line Items

The parent recipe's SQL query does not filter out zero-amount lines at the SQL level; however, the child recipe's custom Ruby code handles line item transformation and may exclude or handle zero-amount lines during the invoice update step.

Error in Invoice Update

If the invoice header is created successfully but the line item update fails, the try/catch block in RF 4.1a catches the error, logs it via the mapper and logger, and the process continues to the next contract. The invoice may exist in NetSuite with incomplete lines -- this would need manual remediation.

Multiple Line Items per Contract

A single SFDC Contract can have many line items (different products, quantities). The child recipe handles this via a For-Each loop over the sales order lines, calling RF 4.1b for each to compute billing frequency, then batching all lines into a single invoice update.

12. Operational Notes

Deduplication Mechanism

Deduplication happens at two layers:

  • Snowflake (pre-processing): The SQL query's LEFT JOIN on WORKATO_LOGS with source_table_name = 'SHARED_EXTERNAL_WORKATO.NETSUITE_CATCHUP_INVOICE' ensures only unprocessed contracts are returned. The ROW_NUMBER() window function takes the most recent log entry per external_id.
  • NetSuite (in-processing): RF 4.1a searches for existing invoices before creating a new one, preventing duplicates even if a log entry was missed.

Error Handling

Each recipe level has its own error handling:

  • Master (Rec 4.1): Try/catch with 0 retries, 2-second interval. Errors in one contract iteration do not stop others.
  • Child (RF 4.1a): Nested try/catch blocks around the invoice creation and update steps. Errors are formatted via mapper and logged.
  • Grandchild (RF 4.1b): Pure computation; errors in billing frequency calculation would propagate up to the child's catch block.
